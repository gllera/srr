## Verdict

The core bet — write-once immutable packs + one mutable pointer object, a single writer,
dumb static reads — is **sound and proven**: CDN-cached for free, offline-capable, and
still working when every server is down. The debt is not in the bet; it is in (1) the
number of coordinated side-structures the bet has accreted (idx summaries, meta shards,
bloom summaries, delta chain, GC low-water, ping/pong seen slots, gen counter — each
with its own sync function, fallback path, and GC), and (2) a few operational seams:
purge discipline, the `.locked` SIGKILL wedge, the single writer pinning fetch egress to
one IP, and two security gaps (SEC1, SEC2 — since planned in `FINDINGS-PLAN.md`).

## 1 · Security & privacy

<a id="sec3"></a>
### SEC3 — Admin origin ships no security headers
`P2 · S · BE-X2`
`cmd_serve.go:90-95,146-161,174-180` — `hostGuard` (Host/Origin allowlist) only; no CSP,
`X-Content-Type-Options: nosniff`, `X-Frame-Options`, or `Referrer-Policy` anywhere.
Defense-in-depth, not an open hole (Access-gated, loopback-bound, previews sandboxed in
`app.js:1036-1042`) — but the origin renders network-derived feed content/error strings
and the `el()` helper already supports `html:` (`app.js:78`). **Fix:** header
middleware — strict static CSP (`default-src 'self'` where feasible), nosniff, Referrer-Policy.

<a id="sec5"></a>
### SEC5 — Every secret goes to every external command
`P2 · M · BE-X1`
`mod/secrets.go:31-51` — `SubprocessEnv()` overlays the *entire* `secrets:` map onto the
environment of every shell pipe step of every feed and every external ingest
(`mod/main.go:272,325-329`). The blast radius of the weakest one-off script is the whole
credential set (Telegram keys, PATs, …). **Fix:** optional allowlist axis —
`Recipe.Secrets []string` + feed-level override (mirroring ingest/pipe);
`SubprocessEnv(names)` filters; an absent list keeps today's behavior.

## 3 · Broken today — frontend

<a id="feb1"></a>
### FEB1 — Footnotes dead on both sides of the contract
`P1 · M · FE-B1`
Backend never allowlists `id` — only `dir`/`lang` are global (`backend/mod/sanitize.go:81-82`),
so footnote targets are stripped at write time; the frontend independently rewrites bare
`#frag` hrefs to absolute CDN-directory URLs (`fmt.ts:171-175,95-108`) which open a new
tab via `<base target="_blank">` (`index.html:35`). Every longform/academic/newsletter
article with footnotes has broken round-trips. **Fix (two-sided):** allow a validated
`id` attribute through bluemonday; special-case bare-fragment anchors in `fmt.ts`
(in-page, override the base target). The backend half only helps articles fetched after it ships.

<a id="feb2"></a>
### FEB2 — Playing audio/video destroyed on prev/next; no position memory
`P1 · M · FE-B2`
`render()` calls `el.content.replaceChildren(...)` on every article change
(`app.ts:377`; `app.ts:451` is the empty-state swap, not an article render); no
`currentTime`/`playbackRate` handling exists anywhere
in `src/js` (grep empty). Backend `#enclosure` actively injects podcast
`<audio controls>` (`enclosure.go:119-122`), so the path is live: a swipe kills a
playing podcast dead with no resume. **Fix (minimum):** persist `currentTime` + rate per
chronIdx in-session and restore on re-render. The full feature is [RDR16](#rdr16).

<a id="feb3"></a>
### FEB3 — Broken-image collapse also hides the alt text
`P2 · S · FE-B3`
`collapseBrokenMedia` adds `.srr-broken` (`fmt.ts:231-240`) → `display:none`
(`styles.css:315-317`), hiding the element including its `alt` (which the backend
preserves, `sanitize.go:20`). **Fix:** on image failure render the alt as a small
placeholder caption; keep the collapse for alt-less media (it's deliberate).

<a id="feb4"></a>
### FEB4 — `<video>` without `controls` renders as a dead frame
`P2 · S · FE-B4`
`fmt.ts` forces `controls` on feed `<audio>` (`fmt.ts:196-204`) but the VIDEO branch
only routes `src`/`poster` (`fmt.ts:190-195`); a non-autoplay feed video lacking
`controls` renders inert. **Fix:** force `controls playsinline` on non-autoplay video —
the treatment audio already gets.

## 4 · Reliability, durability & ops

<a id="rel3"></a>
### REL3 — Lease + CAS locking (kill the SIGKILL wedge)
`P1 · M · AR-R3 + BE-A3`
`.locked` is a nil-payload marker with no owner, no TTL, no steal path (`db.go:19`,
`db.go:339-344`, `db.go:402-408`); the known v4.6.0 residual — SIGKILL leaves it behind
and wedges the loop until a human passes `--force` — and correctness is only advisory (a
racing writer does last-write-wins on db.gz). **Fix:** (a) lease: a lock object
`{owner, expires}` renewed each cycle, stealable when stale; (b) optional correctness
via conditional writes — S3/R2 support `If-Match`/`If-None-Match` on PUT; add `PutIf` to
`Backend` (local = `O_EXCL`+rename, SFTP = rename dance) and CAS the root pointer. The
lock becomes an optimization; races become clean retries.

<a id="rel11"></a>
### REL11 — `/metrics` endpoint on `srr serve`
`P2 · M · BE-O7 + IM-I21`
Health lives in db.gz fields + logs; the watchdog scrapes state, and tuning work
(FET1/FET2, [GRO5](#gro5)) has nothing to measure against —
regressions like the meta-tail write amplification were found by hand in the 07-10
audit. **Fix:** Prometheus text endpoint (cycle durations, bytes fetched/written,
per-feed status transitions, GC/consolidation counters), or a per-cycle structured
report object in the store.

## 5 · Fetch pipeline

<a id="fet3"></a>
### FET3 — Per-feed egress proxy — recovers the 16 nitter feeds
`P1 · M · IM-I03 + BE-F12`
Deployment fact: 16 of 71 feeds (~23%, X/Twitter via nitter) fail from dmz's datacenter
IP since the fetch loop moved there on 2026-07-09 (nitter.net resets HTTP/2,
poast/xcancel WAF-403). Highest immediate user-visible payoff — the feature *worked*
until 07-09. **Fix:** a per-feed (or per-recipe) `proxy:` setting (SOCKS5/HTTP) on the
fetch transport, routing only those feeds through bastion's residential IP (WireGuard,
or an ssh `-D` tunnel dmz→bastion); the `secrets:` section in `srr.yaml` already homes
credentials. ARC4 (planned, S25) is the architectural alternative that fixes the same
feeds without a proxy — one of the two suffices, so this stays open only as the
fallback if S25 stalls.

<a id="fet5"></a>
### FET5 — Store lock held across the entire network fan-out
`P1 · L · BE-F1`
`cmd_fetch.go:351-352` (verified) — `runFetch` wraps the whole cycle in
`withDBCtx(ctx, true, …)`; the errgroup fan-out (`cmd_fetch.go:438-457`) performs every
feed's HTTP GET, pipeline run, `#readability` fetches, and asset transcodes/uploads
inside the lock — held for minutes though nothing is written until after `g.Wait()`; a
running cycle 409s every GUI mutation. **Fix:** three phases — (a) brief locked snapshot
of feed state + recipes, (b) lock-free network fan-out producing per-feed results,
(c) short locked write phase re-loading db.gz and re-applying per-feed state by id
(discarding results for feeds whose URL changed underneath), then the existing
PutArticles→Commit chain. Lock hold drops to seconds. Composes with [REL3](#rel3);
largely subsumes REL7 (planned).

<a id="fet6"></a>
### FET6 — Network mods run serially per item
`P1 · M · BE-F2`
`feed.go:552-587` — the second pass loops candidates one at a time: `#readability` = one
GET per item (20 s default), `#selfhost` downloads each item's media sequentially
(120 s default). Only the post-pipe *upload* is concurrent (`feed.go:644-662`) — proving
item-level fan-out is already considered safe. A 50-item teaser feed with `#readability`
can hold one worker ~17 min. **Fix:** bounded errgroup over the candidate loop with an
order-preserving indexed result slice (published-sort happens later, `cmd_fetch.go:467`).

<a id="fet7"></a>
### FET7 — No in-cycle retry for transport blips; `Retry-After` ignored
`P2 · M · AR-F4 + BE-F7`
`ingest/feed.go:63-65` — every non-200/304 is an immediate hard error; one attempt per
feed per cycle (`feed.go:256-259`), so a connection reset or a `429 Retry-After: 2`
costs a full interval and dirties `ferr`/`fail_streak`. **Fix:** one short retry for
idempotent transport-level failures (reset/EOF pre-response); honor `Retry-After` when
≤ a few seconds. Distinct from FET1 (cross-cycle cadence).

<a id="fet8"></a>
### FET8 — Hard-coded 10 s whole-request fetch timeout
`P2 · S · BE-F8`
`cmd_fetch.go:342-345` — `http.Client{Timeout: 10s}` covers dial + TLS + TTFB + entire
body read; `MaxFeedSize` is 5000 KB, so a 5 MB feed must sustain ~4 Mbit/s or fail
permanently every cycle. Not configurable; inconsistent with `#readability`'s 20 s.
**Fix:** `--fetch-timeout`/`SRR_FETCH_TIMEOUT` (default ~30 s); optionally split budgets
(`ResponseHeaderTimeout` ~10 s + generous overall).

<a id="fet9"></a>
### FET9 — Typed/classified fetch errors
`P2 · M · BE-F11`
`ferr` is a flat string (`Feed.FetchError`); all failures look alike to backoff logic,
the admin GUI, and the operator. **Fix:** classify (dns, tls, http-4xx, http-5xx,
WAF-403, parse, timeout) → smarter backoff (a WAF block is permanent-ish, a 503 isn't),
GUI grouping, alerting. Feeds FET1 and REL6 (planned).

<a id="fet10"></a>
### FET10 — Batch JSONL protocol for external mods
`P2 · M–L · BE-F3`
`mod/main.go:280-352` — external mods run once per item per pipeline token: fresh
`/bin/sh -c` fork + full JSON encode of the item *including the whole parsed `Raw`
entry*, per call (external **ingest** is already batched). 100 new items × one Python
mod = 100 interpreter startups + 100 redundant serializations per step; stateful mods
(model loading, session reuse) are impossible — the real extensibility ceiling.
**Fix:** opt-in batch mode (one process per feed+step, items streamed as JSONL one line
in → one line out, order-preserving); today's per-item exec stays the compatible
default; the 64 MiB cap + `WaitDelay` hardening carries over per line.

<a id="fet11"></a>
### FET11 — One-parse DOM content session
`P2 · L · BE-F4`
Each HTML mod does its own parse/render round-trip (`dedupmedia.go:89`, `unlazy.go:71`,
`untrack.go:77`, `embed.go:65`, `enclosure.go:244`, `#selfhost`, the upload step, plus
`#sanitize`, `#minify`, and the always-on lang stamps) — ~8–11 full passes per article
on a rich pipe; pure CPU on a 4-core ARM box, and each round-trip risks serializer
perturbation (the mods' careful verbatim-on-no-op returns show the hazard is
recognized). **Fix:** optional `ProcessDOM(ctx, Params, *html.Node) (changed, err)`
interface + a per-item content session materializing strings only at boundaries; the
recommended pipe order already clusters the DOM mods contiguously.

<a id="fet12"></a>
### FET12 — WebSub push
`P2 · M · AR-F5 + IM-I14`
**Verified:** no WebSub/push anywhere; fetching is pure interval polling. dmz is already
publicly reachable through its tunnel — `srr serve` hosts a small HMAC-verified callback
route; subscribe for capable feeds (YouTube supports WebSub natively); a hub ping
triggers that one feed's fetch; the 5-min loop remains the fallback. New videos land in
seconds instead of poll-bound, and polling load drops for exactly those feeds. Pairs
with FET1.

<a id="fet13"></a>
### FET13 — Permanent redirects never repoint the feed URL
`P2 · M · AR-F3 + BE-F10`
The fetch client sets no `CheckRedirect` (`cmd_fetch.go:338-346`); net/http follows up
to 10 hops and `feedFetch` discards the final URL (`ingest/feed.go:53`). The only
repoint mechanism is HTML `<link rel=alternate>` discovery on parse failure. **Fix:**
track hops in `CheckRedirect` (pattern exists in `store/http.go`); when every hop was
301/308 for N consecutive cycles, set `Result.ResolvedURL` to the final URL — the
existing repoint path handles the rest **without** resetting the seen pool (unlike
`setFeedURL`, which deliberately clears validators). Leave 302/307 alone.

<a id="fet14"></a>
### FET14 — Pools/engine rebuilt per cycle; throwaway resolve clients
`P3 · S · BE-F13`
`cmd_fetch.go:358-386` — `bufPool` (workers × ~5 MB), `procPool` (`mod.New()` recompiles
bluemonday policies), and the ingest engine are rebuilt every cycle; only the HTTP
client is hoisted to `Run()`. Every resolve/import probe builds a throwaway transport +
5 MB buffer and never `CloseIdleConnections` (`cmd_feeds.go:30-31`,
`cmd_import.go:206-214`). **Fix:** hoist pools/engine to `Run()`; one shared
package-level resolve client with pooled buffers.

<a id="fet15"></a>
### FET15 — Cross-feed dedup
`P3 · S · IM-I17`
The seen pool dedups within a feed; the same story arriving via two feeds (or a
feed-migration overlap) isn't caught. **Fix:** an optional canonical-link-hash axis in
`seen.gz` — same columnar format, per-feed opt-in like `dt`.

## 6 · Growth & scaling

<a id="gro2"></a>
### GRO2 — Mixed retention windows re-scan the whole spread every cycle
`P2 · M · BE-G2`
`db_expire.go:56-80` — one global window `[minStart, maxCutoff)`: `minStart` = the
oldest `AddIdx` across expiring feeds, `maxCutoff` = the *shortest* retention's cutoff.
With `exp=365` alongside `exp=7`, ~358 days of packs are re-read every cycle (including
idle ones) to expire nothing. **Fix:** per-feed frontiers — `fetched_at` is
chron-monotone, so each feed's expirable region binary-searches (the
`cmd_art.go findWindow` precedent); walk only the union of non-empty regions — steady
state ≈ one cycle wide each.

<a id="gro3"></a>
### GRO3 — Idx/meta summaries rebuilt O(store) on every finalization
`P2 · S–M · BE-G3`
`db_pack.go:596-598` (SyncIdxSummary "always rebuilds from scratch");
`db_meta.go:387-393` re-reads every finalized shard's header — at 1M articles each
5k-article meta finalization issues ~200 GETs + a ~800 KB summary re-upload while the
just-computed bloom is discarded from memory (`db_meta.go:464-470`). O(store) work per
fixed-size increment = quadratic cumulative cost. **Fix:** extend incrementally — GET
the previous summary, append the in-memory bloom/header (gzip members concatenate
legally), publish; keep from-scratch as the post-bump/missing fallback (already the
self-heal story). Same shape for `idx/h<N>.gz`.

<a id="gro4"></a>
### GRO4 — No per-cycle pack read cache
`P2 · S · BE-G4`
`walkArticles` (`db_meta.go:141-158`) does `readGz` + parse per call, no memo; called by
`ExpireArticles`, once per managed out-feed, and by `SyncMeta` catch-up — only the delta
chain is memoized (`db.go:146-154`). **Fix:** cycle-scoped `map[key]→parsed pack` on
`DB` (reset where `deltaMemo` resets); finalized packs are immutable so even a
cross-cycle LRU is safe. Multiplies GRO1/[GRO2](#gro2)'s wins.

<a id="gro5"></a>
### GRO5 — Zopfli policy per series
`P2 · S · BE-G5`
`db_pack.go:330-358` — measured −4% on data packs vs −11% on idx; go-zopfli (pure Go)
runs on every finalized data pack, every 50k idx pack, and **every 5k meta shard**,
inline under the lock, on ARM hosts; `gzipBest` also gunzips input + round-trips output
(two extra inflations per finalization). **Fix:** benchmark once ([TST5](#tst5) first;
`finalGzip` is already a seam), then per-series policy: keep zopfli for idx (tiny
inputs, best ratio), stdlib/klauspost `BestCompression` for data/meta. Names are
write-once + cached forever, so the policy knob is the whole fix.

<a id="gro6"></a>
### GRO6 — Post-`gen --bump` meta rebuild is monolithic, not resumable
`P3 · S · BE-G6`
`db_meta.go:297,373-395` — after a bump, `SyncMeta` walks from 0 and advances
`MetaPacks` only after *every* shard save succeeds; a transient error at shard 190/200
re-pays the full-store walk + re-zopfli next cycle; a flaky store can make it never
complete. **Fix:** advance `MetaPacks` incrementally inside the loop — the crash
argument already holds (each shard is durable before coverage records it).

<a id="gro7"></a>
### GRO7 — Whole-batch multi-copy buffering — backfill import is the OOM path
`P3 · M · BE-G7`
`db_pack.go:621-636` (batch held twice), `:712-719` (full delta gzip in memory),
`:848-862` (consolidation copies entries + prior tail pack). Steady-state packs are
bounded, but the batch is not — a backfill import or a 64 MiB single article yields
~3-4× that in RSS, under the lock, on an 8 GB SBC. **Fix:** chunk oversized batches into
≤N-MB sub-batches through the existing materialization (each sub-batch is just "a cycle"
to the format; `shouldConsolidate`'s byte cap handles the rest). Zero format impact.

## 7 · Storage layer

<a id="sto1"></a>
### STO1 — Add `List` to the Backend interface — retire the compensation economy
`P2 · L · BE-S1`
`store/main.go:135-151` — the interface is exactly `Get/Stat/Put/AtomicPut/Rm/Close`.
The compensation economy for the missing `List`: the persisted `gcs` low-water mark +
`gcMaxSweep` (`db_pack.go:388-446`), an *accepted permanent leak* in `GCSummaries`
(`db_pack.go:448-453`), `syndicate rm`'s fragile files-before-Commit ordering
(`cmd_syndicate.go:347-366`), the whole `sitemap.txt` superset-manifest dance
(`cmd_frontend.go:241-284`), ad-hoc per-backend temp sweeps, and permanently unfindable
orphaned assets. Three of four backends list natively. **Fix:**
`List(ctx, prefix) ([]string, error)` with HTTP returning `ErrUnsupported` (callers keep
the computed-name fallback). Then GC becomes "list + delete below cutoff not in live
set", `inspect --validate` can find orphans, `frontend update` drops sitemap.txt.
Incrementally adoptable; makes the planned ARC2 manifest's GC trivial.

<a id="sto2"></a>
### STO2 — Unify missing-key semantics on `fs.ErrNotExist`; fix `Stat` conflation
`P2 · M · BE-S2`
Three ad-hoc contracts (Get's `ignoreMissing` flag; Stat's "(0, nil)" for missing,
`main.go:139-141`; Rm silent). `s3.go:139` discards the SDK error (no `%w`, no
`fs.ErrNotExist`); only S3 and HTTP fail to wrap (local and SFTP both preserve
`fs.ErrNotExist`). Consequences: `rmIfPresent`
(`cmd_syndicate.go:356-366`) treats Stat==0 as proof-of-absence, but HTTP HEAD without
Content-Length counts as 0 (`http.go:199-221`) → a present-but-undeletable out-file gets
its config dropped (the exact strand the comment forbids); every *new* asset pays HEAD +
full GET before its PUT (`assets.go:335-345`); `db_meta.go:290` treats a transient Stat
error as absence. **Fix:** all backends wrap missing in `fs.ErrNotExist`; change Stat to
`(size, exists, err)` or use `errors.Is` at call sites; retire `ignoreMissing`.

<a id="sto3"></a>
### STO3 — SFTP handshake per `store.Open`; HTTP `Close` flushes DefaultTransport
`P3 · M · BE-S3`
S3 memoizes its client (`s3.go:54-64`); SFTP dials TCP + SSH handshake + subsystem on
**every** `store.Open` (`sftp.go:58-125`) — i.e. per serve API request.
`http.go:247-250` `Close()` calls `CloseIdleConnections()` on the shared
`http.DefaultTransport`, flushing keep-alives for every DefaultTransport user in the
process. **Fix:** memoize SFTP sessions like `s3Clients` (keyed, liveness-checked, Close
= ref-release); per-config shared HTTP transport. (P3 because prod is S3/R2.)

<a id="sto4"></a>
### STO4 — Zero retry on transient errors; dead SFTP never redials
`P3 · M · BE-S4`
No retry/backoff anywhere in `store/` (grep). S3 gets the SDK's retryer implicitly; HTTP
ops are single `client.Do` shots; SFTP uses the one session dialed at open — after a
mid-cycle TCP reset every subsequent op fails. Expire is worst-hit: all-or-nothing
(`db_expire.go:131-141`), so one flaky Stat aborts the retention pass. **Fix:** small
`withRetry` (2-3 attempts, jittered, connection-class errors only) around
Get/Stat/Rm/AtomicPut/overwrite-Put — **excluding** exclusive-create Put (a retried lock
create would see its own `os.ErrExist` and self-deadlock); SFTP: detect
`ErrSSHFxConnectionLost`/EOF, redial once.

<a id="sto5"></a>
### STO5 — Assets buffered whole in memory; `octet-stream` without a peek command
`P2 · M · BE-S5`
`assets.go:218-222` `os.ReadFile` (up to ~24 MiB under `#selfhost`'s effective default
cap — `selfhostMaxBody` is 128 MiB but `--max-asset-size`, default ~24 MiB, clamps it),
payload buffered again at `assets.go:379`; peak ≈ `SRR_ASSET_WORKERS × (orig + payload)`
— video-heavy Telegram feeds spike hundreds of MB of transient heap for stream-shaped
work. With no `SRR_ASSET_PEEK`, `meta` stays zero (`assets.go:304-329`) → S3 objects
default to `application/octet-stream` on zero-config installs. **Fix:** hash via
`io.Copy(sha256, f)`, upload from the `*os.File` (seekable → S3 signing/retries keep
working), keep bytes in memory only below a small threshold; add
`http.DetectContentType(first 512 bytes)` as the ContentType fallback.

<a id="sto6"></a>
### STO6 — GC deletes are sequential single-key round-trips
`P3 · S–M · BE-S6`
`db_pack.go:388-395` — `gcMaxSweep=64` exists precisely because a backlog "would
otherwise issue thousands of sequential store deletes"; `GCLatest` can still do
64×4 = 256 serial WAN round-trips in a cycle, while `db_expire.go:145-167` already fans
Stat/Rm over an errgroup — the pattern just wasn't applied here. **Fix:** minimal —
reuse the errgroup in `gcSweep`/`GCLatest`; better — `RmAll(ctx, keys)` with an S3
`DeleteObjects` override (1000 keys/call), also useful to `frontend update` and expire.

<a id="sto7"></a>
### STO7 — `HTTP.AtomicPut` atomicity is asserted, not enforced
`P3 · M · BE-S7`
`http.go:146-150` — "the server makes the object visible atomically on completion" is
assumed, never arranged. True for S3-style PUT; a WebDAV/file-mapped server may write in
place, letting a reader observe a truncated db.gz — the exact failure `Local.AtomicPut`
spends fsync+rename to prevent, on the one mutable object every reader polls. **Fix:**
opt-in mode: PUT to `uniqueTempName(key)` + WebDAV `MOVE` (+`Overwrite: T`), falling
back to plain PUT; reuse the temp-name grammar so sweep logic (and [STO1](#sto1)'s List)
covers leftovers. At minimum, document the assumption.

## 9 · Data contract & format evolution

<a id="fmt1"></a>
### FMT1 — Stop reusing feed ids
`P2 · S · AR-D1`
Feed-id reuse after removal is a documented hazard class: the
`RemoveFeed → DrainDeltas` header-corruption guard, "entries below `add_idx` arise from
id reuse". Monotonic-forever allocation deletes the whole class. At ~71 feeds against a
65,536 ceiling, reuse buys nothing; dead slots cost 4 bytes per pack header each.

<a id="fmt2"></a>
### FMT2 — Expiration vs shared/saved assets: save-time pinning + refcount
`P1 · S–M (+M) · AR-D2 + IM-I11`
**Verified, deliberate:** `ExpireArticles` deletes every `assets/…` key referenced by an
expired article with no liveness check (`db_expire.go:35-38,142-175`) — a shared asset
dies with its first expiring referrer, and a ★-saved article keeps its text forever
(immutable packs) but silently loses images/media once the feed's `exp` window passes.
The backend can't exempt saved articles because the saved set is device-local. Two
complementary fixes: **(a) frontend save-time pinning (S–M, do first):** starring
pushes the article's pack-relative asset URLs into the service worker's pinned bucket
(the `pin.ts` + `srr-pinned-v1` machinery already exists for filter pins); un-starring
releases them; no contract change — makes the read-later queue actually durable.
**(b) backend refcount sidecar (M):** bump per ref at `PutArticles`, decrement at
expire, delete at zero — ping/pong like `seen.gz`; kills the collapse-then-heal loop and
makes `ab` exact.

<a id="fmt3"></a>
### FMT3 — `bg` is likely subsumed by the seen pool
`P3 · S · AR-D3`
Dedup has four overlapping mechanisms: `wm`, `bg` (BoundaryGUIDs), the seen pool, `dt`.
Since `bg` moved into the same sidecar as the pool (v4.4.0) with the same atomicity, the
pool likely covers its whole role. **Verify the dateless-guid window first**, then
delete `bg` and its 1024-cap machinery.

<a id="fmt4"></a>
### FMT4 — Comments-link field
`P2 · S · IM-I15`
RSS `<comments>` (Hacker News, Lobsters, Reddit) is currently discarded. Add a `cm`
short key to `ArticleData` and a "comments" affordance next to the reader permalink.
Small contract change; makes discussion-centric feeds first-class. (An `author` field is
the same shape if ever needed; comments is the higher-value half.)

<a id="fmt5"></a>
### FMT5 — Keyword watchlists (per-article axis)
`P2 · L · IM-I16`
A per-*article* classification axis — today tags are per-feed only. A watch rule (name +
pattern, `#filter`-style syntax) evaluated at ingest gives the reader a "⚡ watch" lane
per rule (price drops, a name you track, a CVE keyword). Cheap path avoiding any
idx-entry change: a chron-aligned **bitmap sidecar series** — one bit per article per
watch (~6 KB per 50k-article pack), write-once, delta-friendly, same GC/summary patterns
as every other series. Generalizes later: AI scoring (parked, [Appendix B](#app-b))
could write the same shape.

<a id="fmt6"></a>
### FMT6 — Meta-card enrichment: snippet / thumbnail / lang / date checkpoints
`P2 · XL · FE-R3 + IM-I05 + parts of IM-I04, IM-I08`
List rows show source · age · title because meta cards carry nothing else. Related wants
that all land in the same struct — decide the fields **once** (repeated meta format
bumps are the expensive part): a ~140-char folded **snippet** for a scannable dek; an
optional **lead-image asset key** for list thumbnails rendered through the deployed
imagor at `img.32b.io` (on-the-fly resize, URL-rewrite rule already in place — a
scannability win for YouTube/Telegram-heavy stores, at the cost of per-card bytes and a
design question vs the wire-service aesthetic); per-article **`g`** for language lanes;
per-shard **date→chron checkpoints** to unblock jump-to-date ([RDR15](#rdr15)).
Trade-off: meta shards grow ≈200 KB → 500 KB+ gz; `META_KEEP` (80) in the SW is the
tight bucket. **Proposal:** design doc weighing snippet-only vs snippet+thumb; prototype
snippet-only first. Note [ARC6](#arc6) touches the same series — decide together.

<a id="fmt7"></a>
### FMT7 — Full-text content search
`P2 · XL · IM-I01 + FE-R2 + AR-T2`
Search is titles-only by construction: meta cards are `{f,w,t}` and the blooms fold
title trigrams (`search.ts:1`); article body `c` is never indexed. In a store whose
retention is "forever", not finding a remembered phrase is the largest genuine
capability gap — radical in value, conservative in architecture, because the template
already exists (derived write-once bloom-pruned shards, warn-only sync, `gen` reset,
summary GC, SW grammar). Two shapes to evaluate: (a) widen the per-shard bloom to also
fold content trigrams (likely 16–32 KB/shard vs today's 4 KB; confirmation reads against
`data/` packs for candidate shards only), or (b) a parallel `fts/` series
(pagefind-style chunked inverted index built at consolidation also fits). Design doc first.

<a id="fmt8"></a>
### FMT8 — Generate the parsers, not just the constants
`P3 · S+L · AR-D5 + BE-A5`
`backend/idx_read.go` is a hand-maintained byte-for-byte mirror of
`frontend/src/js/idx.ts`; the binary idx layout is hand-implemented three times (writer
`db_pack.go:206-244`, Go reader `idx_read.go:195-250`, TS reader `idx.ts`) — only
constants are generated. `loadDeltaChain` (`db_pack.go:773-809`) and `loadDeltas`
(`idx_read.go:83-112`) are near-identical duplicates. **Fix:** (S) make `loadDeltas`
delegate to a shared lines-returning core now; (L) extend `srr gen-ts` with a
declarative layout table (field, width, endianness, count source) emitting the TS parser
scaffolding and a Go encode/decode pair — the generator + freshness gate
(`make generate-check`) already exist.

## 10 · Radical architecture

<a id="arc6"></a>
### ARC6 — Merge `idx/` and `meta/` into one derived series
`P2 · L · AR-S2`
`idx/` and `meta/` are both chron-ordered projections of `data/` at different strides
(50k vs 5k), each with its own header scheme, summary object (`h<N>` / `s<N>`), sync
function, db.gz fields (`hdrs` vs `mp`/`mt`), reader fallback path, and GC sweep. One
card series at 5k stride — per-shard header carrying cumulative per-feed counts,
data-pack boundary info, and the bloom, then the `{f,w,t}` cards — replaces all of it.
**Trade-off:** idx's 2-byte entries are bandwidth-optimal for navigation and cards are
fatter to scan within a shard — but cumulative counts at 5k grain (10× finer than
today) let filtered navigation locate the target shard **from the summary alone**, and
the shard it then fetches is one the list view needs anyway. Biggest pure
contract-shrink available; thins exactly the code that needs dedicated review agents
today, and shrinks the surface [FMT8](#fmt8) and [TST4](#tst4) must cover. Decide
together with [FMT6](#fmt6) (same series).

## 11 · Reader UX & features

<a id="rdr1"></a>
### RDR1 — Frontier jump-in silently consumes whole backlogs — add undo
`P1 · M · FE-M1`
Seen state is a per-feed **frontier** (single max-chron watermark), not a set
(`nav.ts:795-802,887-900`); `recordSeen` raises the frontier for the tapped feed **and
every other feed in the active filter** (`nav.ts:871-874`) — tap the newest headline on
`[ALL]` and the entire older backlog of all feeds flips to read
(`isRowUnread`, `nav.ts:1382-1385`). Deliberate, documented "everything before here is
caught up" model; the only recovery is the different-scope "Mark unread from here". The
model itself is right (tiny, sync-friendly) — its sharp edge is *silent and
irreversible*, which is what erodes trust in the unread numbers. **Fix:** keep the
model; add a transient **undo snackbar on any large frontier jump** (snapshot the
touched frontier keys before the raise; one tap restores); optionally an opt-in "raise
only on step-through, not on jump-in" mode.

<a id="rdr2"></a>
### RDR2 — "Mark all read" has no confirm and no undo
`P1 · S · FE-M2`
`app.ts:568-570` → `nav.markAllRead()` (`nav.ts:928-932`) raises every member frontier
to `total_art−1` immediately. The [RDR1](#rdr1) snapshot/undo snackbar covers this for free.

<a id="rdr3"></a>
### RDR3 — New-content arrivals are invisible — "N new" pill
`P2 · M · FE-M3`
On the list, `onStoreGrown` pages new rows in above the fold, scroll-pinned, fully
silent (`list.ts:866-911,1039-1063`); in the reader the pending count ticks up silently
(`app.ts:535-547,1049-1054`). Silence is the documented contract — but it makes fresh
arrivals undiscoverable without scrolling up on a hunch. **Fix:** a tappable "N new"
pill at the list top (scroll-to-top on tap) + a subtle pulse on the reader's pending
pill when it grows. Keeps the no-jank scroll-pinning; adds only a signal.

<a id="rdr4"></a>
### RDR4 — Consume the `g` language field
`P1 · S · AR-D4 + IM-I04 + FE-F1 + FE-D2`
**Verified:** `g` ships in every pack line (`format.gen.ts:66`) and **no frontend code
reads it** — articles render under hardcoded `<html lang="en">` (`index.html:2`). Buys
correct hyphenation (`hyphens:auto` needs a correct lang), screen-reader pronunciation
(the real a11y gap — non-English bodies announced in an English voice), and CJK font
selection. **Fix:** stamp `lang` on the reader content host at render + enable
`hyphens:auto` in prose CSS; add **`dir="auto"`** at the same time — explicit RTL markup
already survives sanitize (`dir`/`bdi`/`bdo` preserved, `sanitize.go:77,81,95`) but
undeclared RTL bodies render LTR. Note `g` is absent for all pre-2026-07-19 articles
("unknown" bucket). Language *lanes*/filter chip need `g` in meta cards →
[FMT6](#fmt6); a reader-side translate action keyed off the stored code is a later
radical extension.

<a id="rdr5"></a>
### RDR5 — Display settings: theme override, text size, measure
`P1 · M · IM-I07 + FE-F2`
Dark mode is `prefers-color-scheme`-only with no override (3 dark-scoped blocks,
`tokens.css:99,110` + `styles.css:2074`); no font-size control (fixed scale
`tokens.css:31-34`), fixed 680px
measure (`styles.css:76,1014`). The design harness already implements an Auto/Light/Dark
switch, so the token plumbing is proven. **Fix:** a small "Display" card off the
settings menu (theme override, text size, optionally measure), persisted in localStorage
(`keys.ts`) and carried in the profile blob as an opt-in pref (the `srr-img-proxy`
pattern; blob `v:2`→`v:3`). Small work, touched every day.

<a id="rdr6"></a>
### RDR6 — Share / copy-link affordance
`P2 · S · IM-I10 + FE-F3`
No `navigator.share`, no copy-URL affordance anywhere (grep empty); the masthead
permalink + `f` key is the only exit. **Fix:** one frontier-menu or toolbar entry —
`navigator.share({url: article.link})` where available, clipboard fallback.

<a id="rdr7"></a>
### RDR7 — Image lightbox / tap-to-zoom
`P2 · M · FE-F4`
Content images are inert — the only content listener is an `error` capture
(`app.ts:1120`); desktop has no enlargement path at all (mobile has page pinch-zoom).
**Fix:** minimal zero-dep overlay viewer for `.srr-content img` (fixed overlay +
transform); fits the modal patterns already in `dropdown.ts`.

<a id="rdr8"></a>
### RDR8 — Search upgrades: scope combos, highlighting, reachable from reader
`P2 · M · FE-F5`
`q:` is a standalone single-token mode — `filter.set` short-circuits, so it cannot
combine with a feed/tag scope (`nav.ts:446-461`); no term highlighting (rows render via
`textContent`); `/` is list-gated (`app.ts:1198-1201`). **Fix:** (a) allow `q:` + one
scope token ("search within tag"); (b) `<mark>` matched folded terms in result rows;
(c) let `/` work from the reader (open list in search mode).

<a id="rdr9"></a>
### RDR9 — Picker ergonomics at scale
`P2 · M · FE-F6`
Rows are alphabetical-only; the overlay reopens at `scrollTop = 0` every time
(`picker.ts:105`); no type-to-filter input; no favorites/pinned lanes. **In value
order:** type-to-filter row search → remembered scroll position across opens → optional
favorite lanes above the tag groups.

<a id="rdr10"></a>
### RDR10 — Keyboard-shortcuts help overlay
`P2 · S · FE-F7`
Two surfaces' worth of bindings (`KEY_ACTIONS` `app.ts:968-989`; list keys
`app.ts:1198-1214`) with only per-control `aria-keyshortcuts` hints; no `?` handler, no

`dropdown.ts`).

<a id="rdr11"></a>
### RDR11 — Pull-to-refresh

<a id="rdr15"></a>
### RDR15 — Jump-to-date — **corrected: infeasible client-only**
`P3 · blocked · IM-I08 + FE-F12`

proposed a client-only binary search over meta `w` timestamps; **FE's analysis overturns

backend-emitted date→chron checkpoint sidecar — folded into the [FMT6](#fmt6) format
decision (tappable sticky day dividers become the entry point once checkpoints exist).

<a id="rdr17"></a>

<a id="rdr18"></a>
### RDR18 — Sync: first-party endpoint + per-key saved-set LWW
`P3 · M · AR-U3`

### TST1 — Mechanize the stop-anytime crash audit
`P1 · M · BE-V1`
