# CLAUDE.md

## Project

SRR Frontend — single-page RSS reader. Zero runtime deps. Parcel + TypeScript + plain CSS.

## Commands

`npm run dev|build|test|lint|format|format-check|smoke` — dev server, production build (`../dist/srrf/`), vitest+jsdom, ESLint, Prettier write/check, boot smoke-test (`e2e/boot-smoke.mjs`). E2e entry points: `npm run test-contract|test-browser|test-stress` (each with its own `vitest.*.config.ts`; `vitest.shared.ts` holds common config), design harness `gen-design|shoot-design`, and `serve`/`servep` (the dev CORS pack server). Prefer the root `make` wrappers.

## Architecture

Entry: `src/index.html` → `src/styles.css` + `src/js/app.ts`. Bundler: Parcel 2. `SRR_CDN_URL` is replaced at build time: `parcel/resolve-cdn-url.js` resolves the value, `parcel/transformer-define.js` does the literal substitution in `base.ts`, and `parcel/optimizer-preload-db.js` injects a `<link rel="preload" … db.gz as="fetch">` into the HTML head. Resolution: env var → `cdn-url:` from `$SRR_CONFIG_INLINE` (raw YAML) → `cdn-url:` from `$SRR_CONFIG` (file path, default `$XDG_CONFIG_HOME/srr/srr.yaml`) → fallback `"."` (relative to `index.html`: with no cdn-url the store sits next to the bundle, so `PACK_BASE = new URL(".", location)` — `base.ts`; the dev server doesn't hit this fallback, `serve` sets `SRR_CDN_URL=http://localhost:3000` explicitly). The config read (the INLINE → file → default chain) is shared with the dev pack server via `parcel/srr-config.js` (`readConfigYaml`/`parseKey`).

Dependency chain: `app → nav → {data, search, fmt}`, `nav → data → {idx, cache, base}`, `app → fmt → base`, `app → list → {data, nav, fmt}`, `app → config → {data, nav, fmt}`, `app → dropdown → {fmt, profile}`, `app → pin`, `app → gestures → {dropdown}`, `{idx, data, sw, types.d} → format.gen` (generated, see below), `search → {data, cache, format.gen}`, `sw → sw-grammar → format.gen`, `{app, nav, fmt, pin, profile} → keys` (the localStorage-key constants hub). All in `src/js/`, strict mode. (`nav` owns the search **filter mode**, so it imports `search`; `dropdown` no longer imports `nav` or `data` — those moved to `config.ts`; `gestures` no longer imports `nav`.)

**Three surfaces.** `app.ts` tracks `view: "list" | "reader" | "config"` and switches via `showList` / `showReader` / `showConfig` (body classes `srr-view-list` / `srr-view-config`). The **list** (`list.ts`) is home: a scannable headline feed under the current filter, newest-first, **source-keyed** — each row carries a per-feed color rail + a colored mono "source · age" eyebrow over the title (`fmt.srcColorIndex` → `data-src`; read/unread shows as title weight/brightness + rail saturation, no separate dot). The **reader** (the `<article>`) is the drill-down — one article, prev/next: a focused reading surface with just back / save / prev / next. It's **source-keyed** like the list — `app.ts` stamps `data-src` (`fmt.srcColorIndex`) on `.srr-reader`, coloring the mono **masthead** (`.srr-source` in the source color) over an **absolute dateline** (`formatDate`, with the relative age on hover). Its **prose is serif** (`--font-serif`) while headings/masthead/code stay sans/mono (the "wire chrome wraps human prose" pairing); in-content links/blockquote stay neutral ink. **Edge feedback** spans list and reader: the list caps **both** ends with matched sign-offs over a dashed rule — "OLDEST" at the bottom (`.srr-wire-end`, `list.ts` `syncBottomTerminus`) and "LATEST" at the top (`.srr-wire-top`, `syncTopTerminus`), and in the reader a swipe/arrow toward a dead edge rings a **margin bell** (`app.ts` `bumpReaderEdge` — a lateral rubber-band on `.srr-reader` + a pulse on the greyed ◀/▶ — `srr-bell-left/right` + `srr-edge-pulse`), the horizontal cousin of the list's keyboard row bump; both honor `prefers-reduced-motion`. `app.ts` toggles them (`showList`/`showReader`; `srr-view-list` hides the reader-only prev/next + the save toggle + the back button). The toolbar (`.srr-toolbar`) holds the **list toolbar** (feed readout · prev/next disabled · settings gear) and the **reader toolbar** (back · ‹ prev · next › · save) — the prev/next pair centered in the thumb zone (`next` the emphasized accent pill), flanked by back/save at the edges. The **config surface** (`config.ts`, `.srr-config`, `srr-view-config` body class) is a third ephemeral surface that stacks over the list (`srr-view-list` stays set underneath), opened from the settings gear (`.srr-settings`) in the toolbar. It owns: the **quick-action icon bar** (search · unread · image-proxy · backup), the **contextual offline-pin row**, the **filter picker** (a static in-flow panel — `[ALL]`, "★ Saved", tag groups and feeds; the same source-color chips, health-tinted labels, and async unread badges as the old feed-menu, now rendered by `config.ts`), and last the **freshness/degradation status line**. The `onSelect` closure injected by `app.ts` receives the picked token and opens the **reader** at that filter's resume position (`guard(nav.switchFilter)` — the same primitive the filter-cycle uses), not the list. The **unread (catch-up) toggle** (`.srr-config-unread` inside config) is wired via `ConfigHooks.onUnreadToggle` to `app.ts`'s `toggleUnseenOnly` — one tap to show only unread in whatever's filtered; unseen-only spans every filter ([ALL]/feed/tag), persisted in `srr-unread-only`. **Title search** is a pinned `.srr-search-input` bar (toggled via the search quick-action in config / `/` key): `input` is debounced (200ms) into `applySearchQuery` which sets the `q:<query>` filter, `replaceState`s `#!q:<query>`, and `list.rerender()`s. The **feed/tag ⓘ info dialog** and **backup/restore dialog** live in `dropdown.ts` (see below) and are opened from the config surface's quick-action bar or the ⓘ button next to each filter row. Hash scheme: a **numeric position** (`#pos[!tokens]`) is the reader; **no position** (`#` or `#!tokens`) is the list at that filter — where a `q:<query>` token (URL-encoded, e.g. `#!q%3Aclimate`) is the **title-search filter mode** (see `nav.ts`). `app.route(hash)` picks the surface on boot and every `hashchange`; row-tap → `nav.goTo(chron)` (pushState reader hash) → reader; the back button (`goToList(true)`) / two-finger cycle / Escape from config → list. `srr-hash` persistence covers both surfaces, so a reload resumes where you were. Tapping a row commits into the reader; a filter change on the list re-filters and pushes a `#!filter` history entry without touching the reader's position, so browser-back walks back through those filters.

**Saved articles (`★`).** A per-article collection orthogonal to the feed/tag/seen axes: a device-local chronIdx set in `localStorage` `srr-saved` (`nav.toggleSaved`/`isSaved`/`savedCount`/`currentChron`). chronIdx is a permanent article address (finalized packs are immutable and never GC'd), so a saved id stays loadable forever — even after its feed is deleted (`feedTitle` then shows the `[DELETED]` tombstone). "★ Saved" is a **distinct nav mode** (`filter.saved`, reserved token `SAVED_TOKEN = "~saved"`): `filter.matches` becomes saved-set membership (feed-agnostic) and navigation walks the explicit set, not the idx packs. Both nav and the list route their neighbor walks through one seam — `nav.feedLeft`/`feedRight` (saved → a sync set scan, else `data.findLeft`/`findRight`); `step`/`last`/`goTo`/`showFeed` and `list.loadOlder` all go through it, so saved mode is decided in one place. The reader has a `.srr-save` star toggle (key `b`, `aria-pressed`); list rows carry a tappable `.srr-row-star` (filled on a saved row) that toggles without opening the reader and, in the Saved view, drops the row on un-save. `getFilterEntries` surfaces `~saved` in the keyboard/two-finger rotation only when something is saved; the feed menu shows a "★ Saved" row (count badge) under `[ALL]`, picked via the same `selectFilter`/`switchFilter` path as a feed.

| `idx.ts` | Binary idx pack parsing. Imports the format constants from `format.gen.ts`; re-exports `IDX_PACK_SIZE`. Each pack is `header ‖ entries ‖ footer`: each entry is 2 bytes (`feed_id:u16 LE`, low byte then high), and the data-pack boundaries live in a trailing u16 LE footer (`IDX_BOUNDARY_SIZE`) — the local indices where the packId advances, from which `parse()` rebuilds `bounds`. The header is variable-length — `IDX_HEADER_PREFIX` (2 state u32s `packId_base`/`packOff_base` + a `numSlots` u32), then `numSlots` cumulative `feedCounts` (`Uint32Array`), dense up to the high-water feed_id present when the pack was written. Reads of `feedCounts[id]`/`ownFeedCounts[id]` past `numSlots` are guarded to 0 (`countAt(arr, id)` helper) — a feed added after a finalized pack is simply absent from it. Exports `IdxPack`/`IdxHeader` (gains `numSlots`) interfaces, `makeIdxPack(buf, packIndex, packSize, slots)` (header decoded at construction, entries + footer lazily in `parse()`; `slots` = store high-water+1, threaded from data.ts, sizes `ownFeedCounts` and the hoisted `feedsToLookup`), `parseIdxHeaders()` (idx/h<N>.gz summary decode — a sequential variable-stride walk that must consume the buffer exactly), `lowerBound(n, isBelow)` (the reader's one binary-search primitive, also used by data.ts). |
| `data.ts` | CDN data layer: fetches `db.gz`, then the idx header summary (`idx/h<hdrs>.gz`) + latest idx pack only; finalized idx packs load lazily on first touch (`fetchIdxPack`, promise-deduped slots) with eager fetch-all as the fallback when `hdrs` lags or the summary 404s. Fetches JSONL data packs on demand (LRU-cached by pack ID; `loadArticle` resolves pack+offset via binary search on `IdxPack.bounds` and fetches inline — `loadDataPack` is inlined into it). Exports `db`, `init`, `loadArticle(chronIdx)`, `fetchPackBytes(path, isLatest)` (generic pack fetcher used by `search.ts` and the meta/ loaders; `isLatest=false` so a missing meta pack never triggers the guarded reload — that path is upstream-gated by `metaReady()`), async `getFeedId(chronIdx)`/`findLeft`/`findRight`/`countLeft`, sync `countAll` (latest pack + its cumulative header), `numFinalizedIdx()` (the idx finalized-pack formula), `numFinalizedMeta()` (meta finalized-pack formula, shared with `search.ts`), `metaReady()` (true when `mp`/`mt` fully cover `total_art`; gate for both list-from-meta and search), `loadMeta(chronIdx)` (returns `IMetaWire` card: reads meta/ when `metaReady()`, falls back to reading `data/` and projecting when not — so the home list never breaks while SyncMeta lags), `parseJsonl<T>(buf)` (shared JSONL decoder used by meta/ loaders), `feedTitle(feedId)` (display title with the `"[DELETED]"` tombstone fallback), `groupFeedsByTag(includeEmpty = false)` (no separate `activeFeeds` helper — the active-feeds logic is folded directly in; the default drops `total_art === 0` feeds, `includeEmpty` keeps them — the config picker passes `!isUnreadOnly()` so empty feeds show when read items are; cached per-flag). Re-exports `IDX_PACK_SIZE`, `META_PACK_SIZE`; all pack fetches resolve against base.ts's `PACK_BASE`, so pack addressing and content-URL resolution share one base. |
| `nav.ts` | Navigation state machine: hash routing (`#pos[!tokens]`), traversal, filtering. Returns `IShowFeed`. Uses `pushState`/`replaceState`. Tokens are feed IDs or tag names. Exports `filter`, `fromHash`, `left`, `right`, `first`, `last(replace = false)` (no `token` param — set the filter via `filter.set`/`switchFilter` first), `switchFilter`, `goTo`, `select` (set the cursor to an exact, already-known-matching chronIdx — the list's keyboard selection; no feed walk / idx fetch, no hash update or `recordSeen`, since moving the cursor isn't reading), `cycleFilter`, `cycleToken(dir)` (the next filter-entry token relative to `cycleOriginKey()`; `cycleFilter` is `switchFilter(cycleToken(dir))`; the list's `app.onCycle` uses it too), `currentFeedId()` (feed id of the article currently in the reader, -1 = none), `getFilterEntries`, `getCurrentFilterKey`, `filterLabel`, `pruneSeen`, `unreadCounts`, `tagUnreadFromCounts`, `isUnreadOnly`, `setUnreadOnly`, `feedLeft`/`feedRight` (the explicit-set-vs-feed neighbor seam), `isSaved`/`toggleSaved`/`savedCount`/`currentChron`/`anchorChron` (the reader's `pos` when it still matches the active filter, else -1)/`listAnchor` (the **list** anchor, and — since `render` selects it — the row the list highlights on a fresh filter: `anchorChron` when the reader's article still matches, else a feed/tag/`[ALL]`'s **oldest unread** article (the start of the unread backlog, to read forward from) computed by raising each member's bound past its seen high-water and taking the oldest match, falling back to -1 = newest when nothing is unread (`[ALL]` runs the same scan across every feed — `filter.clear` populates `filter.feeds` with all of them — so a fresh device lands at the oldest article overall, like a never-opened tag); saved/search (and an empty store) are always -1 = newest, and a query in particular always shows its newest hit regardless of seen state) and `SAVED_TOKEN` (saved articles), and `SEARCH_PREFIX`/`isSearchFilter`/`searchQuery`/`searchTruncated`/`searchAvailable`/`searchShort` (the search filter mode — below). `unreadCount(ch)` = the feed's articles strictly after its `srr-seen` position (on each view `recordSeen` advances the seen frontier of **every feed in the active filter** to `pos`: the shown article's own feed to that exact resume position, every OTHER feed in the navigation list by a one-way high-water — so opening an article marks all OLDER articles across the list seen, not just same-feed ones, and reading the newest in [ALL] clears every count. Search/`q:` mode is exempt — it returns before recording — and saved mode has no filter feeds, so each records only the shown article's own feed); both sides of the subtraction are idx-derived (`countAll` − `countLeft(seen+1)`) so db.gz `total_art` drift can't skew it; a feed never seen on this device counts as fully unread (its full backlog — `countAll`, a sync no-fetch count), so a fresh device badges every feed and the row badges sum to their tag header. A tag has no count of its own — the dropdown (`tagUnreadFromCounts`) sums its member feeds' `unreadCount`s onto the tag header. A **tag also has no stored seen position of its own**: `getSeen(tag)` derives it as the *oldest* read (min chronIdx) of the tag's member feeds, the anchor `switchFilter(tag)` resumes at, so opening a tag lands at its least-recently-read member and every unread article in the tag sits to the right (none skipped on the left). `pruneSeen` drops any legacy `tag:` key (tags stopped storing their own position). **Unseen-only mode** (`isUnreadOnly`/`setUnreadOnly`, persisted in `srr-unread-only`; device-local, not in the hash) applies to **every filter** ([ALL]/single feed/tag) — the one-tap catch-up view: when on, `filter.applyUnseen` (shared by `set` and `clear`, so it covers [ALL] too) raises EACH filtered feed's lower bound to `max(add_idx, seen+1)` (snapshotted at apply time, so the nav set is stable as you read this session), so `findLeft`/`findRight`/`matches`/search skip already-seen articles. The "member" wording below generalises from tags to whatever feeds the filter spans. The tag still opens at its **current position** — the resume anchor `getSeen` returns (its least-recently-read member, same as outside unseen-only), NOT its oldest unseen: `isValidSeen` validates that resume position against the true `add_idx` rather than the raised bound, so the raised bounds skip seen articles on Left/Right without bouncing the open forward. From the resume position, Right steps onto the first unseen. `has_left`/`has_right` are computed by neighbor probes — `feedLeft(pos-1)`/`feedRight(pos+1)` with `.catch(()=>-1)` so a cold-pack blip degrades to a disabled button rather than an error popup. The dropdown badge stays in lockstep: in unseen-only mode `feedUnread` (so `unreadCount`/`unreadCounts`/`tagUnreadFromCounts`) counts the **unread** article you're sitting on back into its feed — `recordSeen` marks it seen the instant you land, so without this the live-seen-derived badge would read one below the current article — and it ticks down as you step away. Scoped to the current article's feed, to unseen-only mode (`unseenActive()` — `unreadOnly && !filter.saved && !filter.search`), and to while that article actually matches the raised filter (`filter.matches(currentFeed, pos)`) — i.e. it's one of the unread being navigated, not the seen resume position you open on; OTHER feeds' badges are untouched. (This applies equally to `[ALL]`, single-feed, and tag filters — any filter where `applyUnseen` raises bounds.) **Search filter mode** is a third explicit-set mode beside feed-membership and ★ Saved: when the single token is `q:<query>` (`SEARCH_PREFIX`), `filter.search` is set and navigation walks an explicit set of matching chronIdxs — the title-search hits — exactly as ★ Saved walks the saved set (`feedLeft`/`feedRight`/`matches`/`showFeed` all branch on it, so **the list and the reader walk it for free** with no extra plumbing). The set is computed once per query by `search.loadHits(term, SEARCH_CAP=500)` (cached in `search.ts` via `cachedPromise` keyed by query string, so concurrent walks within one render dedupe); nav keeps only the snapshot (`searchSorted`/`searchSet`) and the supersession guard in `ensureSearchSet` — a late-resolving keystroke can't overwrite fresher hits. `searchTruncated()` flags the cap. The query rides in the shareable `#!q:<query>` hash like any token (reload/back/forward restore it; `tokensSuffix` escapes a literal `+` in the query to `%2B` so the `+`-separator split survives), and it behaves like a tag filter — picking another filter / two-finger cycle / arrow-cycle leaves it (search is **not** in `getFilterEntries`, so you can't cycle *into* it). `switchFilter`/`last` open it at the newest hit; `fromHash` loads the set before validating a `#pos!q:…` deep-link so the position is honored. The UI (toolbar magnifier + pinned search bar) lives in `app.ts`. |

| `fmt.ts` | Pure utilities (no DOM imports): `sanitizeHtml`, `timeAgo`, `formatDate`, `URL_DENY`, `imgProxy(url, prefix)`, `getImgProxy()`, `setImgProxy(value)`, `isValidProxy(value)` (empty or `https?://`-prefixed; the image-proxy dialog validates before storing — `setImgProxy` stays a dumb setter), `extractImageUrls(html)` (regex-scrapes `<img src>` — both quoted and unquoted values, the latter to catch `#minify`'s quote-stripped CDN URLs — and returns only http(s) URLs; used by nav.ts prefetch), `collapseBrokenMedia(e)` (delegated error handler: hides — `.srr-broken` — the `<img>`/`<video>` whose resource failed, targeting the `<video>` host for `<source>` failures; old articles outlive their external images. | `search.ts` | Title search over the `meta/` pack series (the derived projection shared with the home list). `available()` gate delegates to `data.metaReady()` (true when `mp`/`mt` fully cover `total_art` — same gate the list uses). `fold(s)` (mirrors `foldSearchText` byte-for-byte). `bloomBits(gram)` (FNV-1a-64 double-hash, 7 probes, mask `& (2^15-1)` for `SEARCH_BLOOM_BYTES`=4096). `shortQuery(q)` (true when no word has ≥3 runes — can't prune shards). Async generator `search(q, limit?)` yields newest-first `ISearchHit[]` batches: latest meta tail first, then bloom-pruned finalized meta shards nf-1..0 (shards are 5,000 entries each, aligned with `META_PACK_SIZE`); AND of folded substring tests per word; stops once `limit` hits are collected. `loadHits(query, cap) → Promise<HitSet>` (`{chrons, truncated}`) is the caching wrapper consumed by `nav.ts`'s `ensureSearchSet`: cached via `cachedPromise` on a `string`-keyed `LRU(8)`, so concurrent neighbor walks within one render dedupe and a rejection drops the slot to retry. On the first query the summary fetch overlaps the latest-tail fetch. Lazy: nothing fetched until first query. Shard LRU(8). Missing latest tail degrades to finalized-only (console.warn); missing summary rejects. |
| `list.ts` | The **list surface** (home): a scannable headline feed that is a **bidirectional infinite window anchored at the filter's reading position** (`nav.listAnchor()`) — opening it drops you back at the article the reader last sat on (when it still matches), else a feed/tag/`[ALL]`'s **oldest unread** article (the start of the unread backlog, to read forward from, with the already-read older rows below it — `[ALL]` runs the same scan across every feed), else the newest (saved/search, an empty store, or a feed/tag/`[ALL]` with nothing unread). Newer ("next") articles page in **above** the anchor and older ones **below**. **Read amplification win:** the list reads thin `meta/` cards (`data.loadMeta`, `IMetaWire {f,w,t}`) for each row instead of full `data/` records — O(meta-pack) network cost for the home list, not O(data-pack). When `metaReady()` is false (SyncMeta lagged), `loadMeta` falls back to `data/` transparently, so the list never breaks. The full `data/` record is only fetched by `loadArticle` when a row is opened in the reader. **Anchor positioning:** returning FROM THE READER (the `center` arg, set in app.ts when `view` was `"reader"`) **centers** the anchor in the viewport and the reader's current article carries `.srr-row-current` (a source-tinted wash; the colored rail already marks the left edge) so you land back on what you were reading; centering is gated to the live reader article (`seed === nav.anchorChron()`), so an oldest-unread anchor (a fresh feed/tag/`[ALL]` filter, date scrub) is **top-aligned** instead — the start-of-the-backlog framing. `scrollChronToView(chron, center)` does both via `window.scrollTo` (clamped, so a near-edge anchor lands as close to centered as the content allows). **Land-once for a fresh anchor (`landOnceMode = anchoredMid && !center`):** a reader-return (`center`) anchors immediately (its pack is warm), but a fresh anchor (boot/filter change) would otherwise scroll onto still-skeleton, pre-font rows and then **bump** as `content-visibility` paints and web fonts reflow them taller. So it stays at the top during load (and **holds the `IntersectionObserver`** so the top sentinel can't runaway-page newer rows), then lands ONCE: after `document.fonts.ready` it CONVERGES the measurement without moving — each frame re-pins every row's true height and recomputes `chronScrollTarget(seed)` — and only when that target stops changing (layout settled, bounded ~20 frames) does it `scrollChronToView` a single time and start the observer. The per-fill `reassertAnchor` likewise runs only for the centered reader-return. The highlight is set per row in `rowEl` and re-derived in `refresh()`, keyed on `nav.currentChron()`. ## Data Structures

See root `CLAUDE.md` Data Contract for db.gz, IFeed, IArticle, pack format, CDN layout, and chronIdx.

Frontend-specific additions:
- `feeds` in `IDB` is `Record<number, IFeed>` (JSON object keyed by feed ID); defaults to `{}` if absent. `feed.id` is populated from object keys at init.
- **IArticle**: `{ f, a, p, t, l, c }` — feed_id, fetched_at, published, title, link, content. Loaded from JSONL data packs.
- **IShowFeed**: `{ has_left, has_right, article, feed, placeholder? }` — `has_left`/`has_right` are computed by neighbor probes (`feedLeft(pos-1)`/`feedRight(pos+1)` with `.catch(()=>-1)`) so the prev/next buttons enable precisely when a step would move. `placeholder` is true only for the synthetic "(no matching articles)" article; `app.ts` renders it as the **shared directed empty state** (`list.emptyStateEl`, `.srr-list-empty` — same search/caught-up/saved/filtered wording as the list), hiding the article chrome via `.srr-reader-empty`, so the reader's no-match state and the list's speak one voice.
- Dev (`npm run dev` / `make dev-fe`): `concurrently` runs Parcel HMR (port 1234) + a CORS pack server (`serve`, port 3000). The pack server serves the **configured local store**, not a fixed path: `parcel/resolve-store.js` resolves the directory from `$SRR_STORE` → `store:` in the active config → `../packs` fallback (same precedence as the backend; an `s3://`/`sftp://` store, missing config, or absent dir falls back with a stderr note). ## Key Behaviors

**guard() mutex** (app.ts): all async UI actions go through it. Drops concurrent calls. On error → popup with retry.

**Eager fetch**: `data.ts` starts `fetch("db.gz")` at module load (before `init()` call).

**Init**: `data.init()` loads db.gz, the idx header summary `idx/h<hdrs>.gz` (when `db.hdrs === numFinalizedIdx()`), and the latest idx pack — O(1) requests at any store size. Finalized idx packs are fetched lazily by chronIdx addressing when navigation touches them; `idxHeaders` (one `IdxHeader` per pack, latest last) powers sync counting (`countAll`), pack-skip deltas (`packHasCandidate`), and the pack-level timestamp search without any fetch. When `hdrs` lags or the summary fetch fails, init falls back to the old eager fetch-all path and peels each pack's own header (`hdrs` stays optional on the wire type — an absent value simply never equals `numFinalizedIdx()`). Latest packs are generation-named `L<seq>.gz` (`db.seq`, normalized `??= 0` for the omitempty empty-store case); every pack name is write-once, so all pack fetches use `cache: "force-cache"`. A non-OK response on a latest pack means this tab's db.gz predates the backend GC grace window — `assertPackOk` triggers one `location.reload()` (sessionStorage-guarded against loops, guard cleared after a successful `init()`) and always throws so callers never touch the body.

**Article loading**: `loadArticle(chronIdx)` resolves pack+offset via binary search on `IdxPack.bounds`, fetches and parses the JSONL data pack (LRU-cached by pack ID, max 20 packs).

**Neighbor prefetch** (nav.ts): after each `left()`/`right()`, the neighbor lookup (`findLeft`/`findRight` — may lazily fetch an idx pack) runs un-awaited off the critical path: its promise lands in `nextLeft`/`nextRight` (consumed by the next keypress; a rejection clears the slot instead of failing the committed navigation) and on resolution `schedulePrefetch(target)` warms the neighbor's images on `requestIdleCallback` (`{timeout:500}`; WebKit has no rIC, so a `setTimeout(…, 200)` fallback keeps iOS/Safari prefetching instead of stalling at every data-pack boundary) — it `loadArticle()`s the neighbor, runs `fmt.extractImageUrls()` over its content, and for each URL creates an `Image` with `fetchPriority="low"`/`decoding="async"` and `img.src = imgProxy(...)`. `abortPrefetch()` cancels in-flight loads by setting `img.src=""` (WHATWG image-update steps) and dropping the refs. The `currentPrefetch` array (also the freshness token: a pending idle callback bails when `my !== currentPrefetch`) bounds memory to one neighbor at a time. Edits to nav must preserve this abort/freshness discipline.

**Content fade-in**: double `requestAnimationFrame` for opacity transition.

**Service worker** (`src/sw.ts`, registered best-effort from `app.ts` with `{type: "module"}` so it can import `LATEST_KEEP` from the generated contract — module-SW support predates the SW's `DecompressionStream` requirement in every engine, so the module type never limits compatibility): offline-capable cache. Lives at `src/` root (not `src/js/`) so Parcel emits it at the deployment root, giving it a default scope over the whole env (`/srr/`, `/srr.tmp/`, …). Buckets keyed by mutability: `srr-assets-v1` (content-hash `assets/…` — immutable, cache-first), `srr-packs-v3` (write-once pack names — finalized `idx|data|meta/<n>.gz`, latest generations `idx|data|meta/L<seq>.gz`, idx header summaries `idx/h<N>.gz`, and meta bloom summaries `meta/s<N>.gz` — cache-first with revalidate-on-miss; only `db.gz` is network-first → cache fallback; offline consistency is structural since a cached db.gz of seq N can only pair with the write-once `L<N>`), `srr-shell-v1` (navigation network-first + content-hashed JS/CSS cache-first; runtime-cached, no precache manifest — keeps the SW hand-written and zero-dep), `srr-meta-v1` (two synthetic entries: the last-seen db.gz `gen` and `seq`). **Manifest tracking** (`checkManifest`): on each successful online `db.gz` fetch whose `ETag`/`Last-Modified` differs from the cached copy's (an unchanged validator means unchanged bytes, so the common no-change load skips the parse and the redundant `cache.put` — a cheap header compare keeps the awaited check off the boot path's budget) the SW gunzips a clone (raw gzip bytes, no `Content-Encoding` — same manual `DecompressionStream` as `data.ts`) and reads `gen` + `seq` + `hdrs` + `mp` (absent == 0). One `parsePackName` helper (in `src/js/sw-grammar.ts`, extracted for unit-testability) owns the pack-name grammar (`series`/`kind`/`n`) for the fetch route, the cache bound, and the manifest prunes — built from the generated `PACK_SERIES_KINDS` table (backend `store.PackSeries`) and strict per-series like the store's `packKeyRe`: a kind letter another series owns (`data/h3.gz`) is not a pack name. A `gen` change (any change, not just increments) purges the whole packs bucket (in-place store rebuild — same names, new bytes); a `seq` change (gen unchanged) prunes cached `L<g>` latest generations with `g < seq - LATEST_KEEP` (imported from `format.gen.ts` = the backend's `latestKeep` GC grace window), idx summaries `h<g>` with `g < hdrs - LATEST_KEEP`, and meta bloom summaries `meta/s<g>` with `g < mp - LATEST_KEEP` (all ride the `seq` prune signal rather than separate meta keys — each can advance without a seq bump on a zero-article retry cycle, but that strands at most one stale name until the next article-producing fetch). Both awaited *before* the db.gz response resolves, so the purge is race-free against the page's subsequent idx/data requests. Best-effort: a parse failure never blocks serving db.gz. The operator bumps `srr gen --bump` after an in-place store rebuild; the `srr-packs-vN` constant now only marks cache-format changes. **Cache bound** (`enforceCacheBounds`): after each successful online `db.gz` fetch (via `event.waitUntil`, off the critical path — and never offline, so a device can't lose a pack it can't refetch) each finalized pack series is capped per `SERIES_KEEP` — `idx`/`data` at `PACK_KEEP` (100), `meta` at `META_KEEP` (80, tighter because shards are ~200 KB gz each) — lowest-numbered evicted first (the numeric names encode chron age, so no access-time bookkeeping), and the assets bucket at `ASSET_KEEP` (500), oldest-cached first (Cache key insertion order; `cacheFirst` never re-puts on a hit). `db.gz` and the `L<seq>`/`h<N>`/`s<N>` names are exempt — offline consistency depends on them and checkManifest owns their pruning. Evicting a pack still being read just costs one CDN refetch on the next miss. Needs a secure context (HTTPS or localhost): production CF HTTPS qualifies, an `http://` LAN-IP deploy does not — registration just no-ops there and the app runs off the network. Coverage: `e2e/browser/reader.e2e.test.ts` ("serves the reader offline after one online visit", "purges stale finalized packs when db.gz gen changes", "bounds the pack and asset caches, evicting oldest entries first"). **Offline pin** (`srr-pinned-v1`): a fifth bucket, eviction-exempt and gen-purged, populated via the `"pin"` SW message from the page. The `⋯` overflow menu shows "Download for offline" / "Remove offline copy" for the current filter scope ([ALL] / feed / tag / unread — saved and search scopes are deferred to v2). On pin: `app.ts` calls `data.packNamesForFilter(filter.feeds)` to enumerate the exact write-once pack names (relative paths like `meta/0.gz`, `data/1.gz`, `idx/L1.gz` — the same form the fetch handler sees after the `/packs/` prefix), stores them in the `srr-pins` localStorage registry (`pin.ts`), and posts `{ type: "pin", names }` to the SW controller via a `MessageChannel`. The SW validates every name through `parsePackName` (no arbitrary cache-key injection), fetches each with `cache: "no-cache"` (ensures fresh bytes at pin time), and puts them into `srr-pinned-v1`. Progress is reported back as `{ type: "pin-progress", done, total }` messages and shown transiently in the status bar. The pack-fetch branch (`packCacheFirst`) checks `srr-pinned-v1` before `srr-packs-v3`, so a pinned pack serves offline even after the rolling-window PACKS eviction drops it. `enforceCacheBounds` never touches `srr-pinned-v1`. `checkManifest` purges `srr-pinned-v1` alongside `srr-packs-v3` on a gen change (in-place rebuild invalidates all cached pack bytes) but leaves it untouched on seq-only changes (latest packs are write-once generation-named — a cached `L<g>` in PINNED stays valid for its seq). On unpin: `app.ts` calls `unpinFilter(key)` (clears the registry entry) and posts `{ type: "unpin", names }` so the SW deletes the exact cache entries. The unread-scope note: a pin is a snapshot — new unread arriving after the pin is not auto-pinned. Neutral pin/unpin copy. SW message protocol summary: `{ type: "pin", names: string[] }` (+ optional `MessagePort` in `ports[0]` for progress); `{ type: "unpin", names: string[] }` (remove specific entries); `{ type: "unpin-all" }` (clear all of PINNED). Coverage: `e2e/browser/reader.e2e.test.ts` ("caches named packs into srr-pinned-v1 on a 'pin' message", "pinned packs survive enforceCacheBounds (eviction-exempt)", "gen change purges srr-pinned-v1 alongside srr-packs-v3" — all three require SW controller, same sandbox limit as the pre-existing SW-controller tests).

**Installable (PWA)** — `src/manifest.webmanifest` (linked from `index.html`) + the SW's fetch handler satisfy the browser install criteria. `start_url`/`scope` are `"."` (deployment-relative) so the same bundle installs under `/srr/` or `/srr.tmp/`. Icons: `src/icon.svg` is the single source of truth — the same RSS glyph as the inline favicon, recolored white on the brand-orange tile and inset into the maskable safe zone. The raster fallbacks are generated from it with `rsvg-convert -w N -h N src/icon.svg -o src/icon-N.png` (192, 512, and a 180 `apple-touch-icon.png`); the manifest also references `icon.svg` directly as a scalable `any` icon. Regenerate the PNGs whenever `icon.svg` changes. Coverage: `e2e/browser/reader.e2e.test.ts` ("exposes a valid, installable web app manifest").

## State Machines (nav.ts)

State: `pos` (chronIdx), `filter` (object with `active`, `feeds`, `tokens`, `matches()`, `clear()`, `set(tokens)`).

**`filter`** — active when `tokens` non-empty:
- `filter.clear()`: empties tokens, repopulates `feeds` from all feeds with `total_art > 0`
- `filter.set(tokens)`: resolves tokens (numeric feed IDs or tag names) into `feeds` map (`feed_id → add_idx`); falls back to `clear()` if no token resolves
- `last(replace = false)`: jumps to the last matching article under the current filter (set the filter first via `filter.set`/`switchFilter`)
- `goTo(idx)`: navigate to chronIdx; if filter active and target doesn't match, snap forward to next match; falls back to `last()` if none
- `switchFilter(token)`: sets filter to token (or clears if `""`); resumes at last seen position for that feed/tag if valid (in unseen-only mode too — `isValidSeen` validates against the true `add_idx`, not the raised bound, so you open at your current position, not the oldest unseen), otherwise jumps to `first()`. `""` ([ALL]) opens at the **oldest unseen** article across all feeds (the same `oldestUnread` scan `listAnchor` uses), falling back to `last()` only when fully caught up — landing on the newest would let `recordSeen`'s cross-feed frontier raise mark the whole store read
- `cycleToken(dir)`: returns the next filter-entry token by `dir` (+1/-1) relative to `cycleOriginKey()`
- `cycleFilter(dir)`: `switchFilter(cycleToken(dir))`
- `getFilterEntries()`: returns `["", "tagName", ..., "feedId", ...]` built via `data.groupFeedsByTag()`
- `getCurrentFilterKey()`: returns `""`, the single token, or `""` for multi-token filters (URL-only edge case)
- `filterLabel(key)`: resolves a filter key to its human label (`""`→"All", `SAVED_TOKEN`→"★ Saved", numeric feed id→`data.feedTitle`, tag name→itself). The single source for the toolbar feed label, the document title, and the list's all-caught-up line, so none of them ever shows a raw feed id
- Navigation uses `findLeft`/`findRight` — async (may lazily fetch an idx pack); finalized packs without a filter match are skipped via summary header deltas, no fetch. `filter.clear()`/`set()` stay synchronous via `data.countAll`
- Hash: `#pos[!tokens]` — `!` segment, `+`-separated tokens, each `encodeURIComponent`-wrapped to survive special chars in tag names.

## Deployment

GitHub Actions `release.yml` (version tags `v*.*.*` or manual dispatch), three jobs: `verify` (gates the other two), `release` (builds the bundle with **no** cdn-url — pack base falls back to `"."` for same-origin self-hosting — tars it as `srrf.tar.gz` and attaches it plus the `dist/srr-*` binaries to the GitHub Release; consumed by `srr frontend update`), and `cf-pages` (deploys the hosted reader to **Cloudflare Pages** — project `srr-frontend`, branch `main`, via `cloudflare/wrangler-action@v4` — building with the `SRR_CDN_URL` secret from the `ci` environment; also needs `CLOUDFLARE_API_TOKEN`/`CLOUDFLARE_ACCOUNT_ID`). `SRR_CONFIG_INLINE` is no longer read anywhere in `release.yml` (only `cron.yml`'s fetch job still uses it).
