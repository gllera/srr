# CLAUDE.md

## Project

SRR Frontend — single-page RSS reader. Zero runtime deps. Parcel + TypeScript + plain CSS.

## Commands

`npm run dev|build|test|lint|format` — dev server, production build (`../dist/srrf/`), vitest+jsdom, ESLint, Prettier.

## Architecture

Entry: `src/index.html` → `src/styles.css` + `src/js/app.ts`. Bundler: Parcel 2. `SRR_CDN_URL` is replaced at build time via `parcel/resolve-cdn-url.js`. Resolution: env var → `cdn-url:` from `$SRR_CONFIG_INLINE` (raw YAML) → `cdn-url:` from `$SRR_CONFIG` (file path, default `$XDG_CONFIG_HOME/srr/srr.yaml`) → fallback `http://localhost:3000`. The config read (the INLINE → file → default chain) is shared with the dev pack server via `parcel/srr-config.js` (`readConfigYaml`/`parseKey`).

Dependency chain: `app → nav → data → {idx, cache, base}`, `app → fmt → base`, `app → dropdown → {data, nav, search, format.gen}`, `app → gestures → {nav, dropdown}`, `{idx, data, sw, types.d} → format.gen` (generated, see below), `search → {data, cache, format.gen}`. All in `src/js/`, strict mode.

Re-exports `IDX_PACK_SIZE`; all pack fetches resolve against base.ts's `PACK_BASE`, so pack addressing and content-URL resolution share one base. |
| `nav.ts` | Navigation state machine: hash routing (`#pos[!tokens]`), traversal, filtering. Returns `IShowFeed`. Uses `pushState`/`replaceState`. Tokens are channel IDs or tag names. Exports `filter`, `fromHash`, `left`, `right`, `first`, `last`, `switchFilter`, `goTo`, `cycleFilter`, `getFilterEntries`, `getCurrentFilterKey`, `isSingleFilter`, `pruneSeen`, `unreadCount`, `unreadCounts`, `tagUnreadFromCounts`, `isUnreadOnly`, `setUnreadOnly`, `peek`. `unreadCount(ch)` = the channel's articles strictly after its `srr-seen` position (`recordSeen` bumps the shown article's own channel key on every view, filtered or not, so reading via [ALL] clears counts); both sides of the subtraction are idx-derived (`countAll` − `countLeft(seen+1)`) so db.gz `total_art` drift can't skew it; a channel never seen on this device counts as fully unread (its full backlog — `countAll`, a sync no-fetch count), so a fresh device badges every channel and the row badges sum to their tag header. A tag has no count of its own — the dropdown (`tagUnreadFromCounts`) sums its member channels' `unreadCount`s onto the tag header. A **tag also has no stored seen position of its own**: `getSeen(tag)` derives it as the *oldest* read (min chronIdx) of the tag's member channels, the anchor `switchFilter(tag)` resumes at, so opening a tag lands at its least-recently-read member and every unread article in the tag sits to the right (none skipped on the left). `pruneSeen` drops any legacy `tag:` key (tags stopped storing their own position). **Unseen-only mode** (`isUnreadOnly`/`setUnreadOnly`, persisted in `srr-unread-only`; device-local, not in the hash) applies to a **single-tag filter only**: when on, `filter.set` raises each tag member's lower bound to `max(add_idx, seen+1)` (snapshotted at apply time, so the nav set is stable as you read this session), so `findLeft`/`findRight`/`matches`/peek/search skip already-seen articles. The tag still opens at its **current position** — the resume anchor `getSeen` returns (its least-recently-read member, same as outside unseen-only), NOT its oldest unseen: `isValidSeen` validates that resume position against the true `add_idx` rather than the raised bound, so the raised bounds skip seen articles on Left/Right without bouncing the open forward. From the resume position, Right steps onto the first unseen. The toolbar counter shows unread-remaining (`showFeed` branches to `unreadTally` — per-member `countAll − countLeft`, summed — because countLeft's cumulative-header shortcut is only exact for true `add_idx` bounds), counting the article you're on **when it is itself unread** (Option A's `+matchesPos`): at the resume position you're on a seen article, so the counter is just the unread to your right — equal to the tag's dropdown badge whenever no unseen sits to your left (a never-seen member with older articles can leave some there, reachable with Left); stepping onto an unseen article then counts it too, so the counter still equals the badge and counts down to 1 on the last unseen — mirroring a channel, which resumes on an already-read article and shows its full unread count to the right. `has_right` still tracks the strictly-after count, so next disables on the last unseen. The dropdown badge stays in lockstep: in unseen-only tag mode `chanUnread` (so `unreadCount`/`unreadCounts`/`tagUnreadFromCounts`) counts the **unread** article you're sitting on back into its channel — `recordSeen` marks it seen the instant you land, so without this the live-seen-derived badge would read one *below* the snapshot-based counter as you read — and it ticks down with the counter. Scoped to the current article's channel (`currentChan`), to unseen-only tag mode (`filter.unreadMembers !== null`), and to while that article actually matches the raised filter (`filter.matches(currentChan, pos)`) — i.e. it's one of the unread being navigated, not the seen resume position you open on (there `matchesPos` is 0 and the counter doesn't count it either) — exactly mirroring `showFeed`'s `+matchesPos`; channel/`[ALL]` badges are untouched. Single-channel and `[ALL]` filters ignore the mode (navigate everything). `peek(span=10)` = the headlines around `pos` under the current filter (`IPeekItem[]`, chron order: up to `span` matches each side plus the shown article, raw wire title + chan_id — the "(untitled)" placeholder and "[DELETED]" tombstone are applied by dropdown's `headlineRow` renderer — `when` falling back to fetched_at); the two directional walks run concurrently over `findLeft`/`findRight` and `loadArticle`, so the common case (neighbors in the resident idx pack + LRU'd data pack) costs zero fetches. |
| `cache.ts` | `makeLRU<T>(maxSize)` (returns the `LRU<T>` interface): LRU cache via Map insertion order. `cachedPromise(lru, key, make)`: joins in-flight-or-resolved work per key, dropping a rejected promise from its slot (identity-guarded) so the next call retries. `lazySlot(make)`: the single-slot variant (a 1-entry LRU). Used by `data.ts` (idx slots — capacity covers every pack so nothing evicts — and the data-pack LRU) and `search.ts` (shards, summary, latest tail). |
| `fmt.ts` | Pure utilities (no DOM imports): `sanitizeHtml`, `timeAgo`, `formatDate`, `URL_DENY`, `imgProxy(url, prefix)`, `getImgProxy()`, `setImgProxy(value)`, `isValidProxy(value)` (empty or `https?://`-prefixed; the dropdown editor validates before storing — `setImgProxy` stays a dumb setter), `extractImageUrls(html)` (regex-scrapes `<img src>` — both quoted and unquoted values, the latter to catch `#minify`'s quote-stripped CDN URLs — and returns only http(s) URLs; used by nav.ts prefetch), `collapseBrokenMedia(e)` (delegated error handler: hides — `.srr-broken` — the `<img>`/`<video>` whose resource failed, targeting the `<video>` host for `<source>` failures; old articles outlive their external images. The error event doesn't bubble, so app.ts registers it on `.srr-content` with `capture: true`). `sanitizeHtml` strips dangerous elements/attributes for defense-in-depth and resolves the proxy prefix once per call; on every `<img>` it forces `loading=lazy` + `decoding=async` (a long article's images must not compete with first render or the neighbor prefetch) and `referrerpolicy=no-referrer` (links already carry `rel=noreferrer`; without it each image fetch tells the feed host where the reader lives — the leak the optional proxy exists to prevent; `<video>` takes no referrerpolicy attribute). Relative URLs: `<img src>`/`<video src>`/`<video poster>`/`<a href>` values that are *relative references* (no URL scheme — the self-hosted `assets/` keys plus any other relative URL the feed carried, including protocol-relative `//host`) resolve against the pack base (base.ts's `PACK_BASE`) and bypass the image proxy, dropped (attribute removed) when the resolved URL escapes that base (bounds-checked — a `//host` or sub-path `../` traversal off-origin is an info-leak vector); external http(s) image/poster URLs keep the proxy path, external video URLs and anchor hrefs pass through (a link is navigation, not an auto-loaded resource). `imgProxy(url, prefix)` requires the caller to pass the resolved prefix — production callers hoist `getImgProxy()` out of per-image loops. Effective prefix comes from localStorage `srr-img-proxy` (set via channel dropdown image-proxy icon); empty/absent = passthrough (no proxy by default). |

## Data Structures

See root `CLAUDE.md` Data Contract for db.gz, IChannel, IArticle, pack format, CDN layout, and chronIdx.

Frontend-specific additions:
- `channels` in `IDB` is `Record<number, IChannel>` (JSON object keyed by channel ID); defaults to `{}` if absent. `channel.id` is populated from object keys at init.
- **IArticle**: `{ s, a, p, t, l, c }` — chan_id, fetched_at, published, title, link, content. Loaded from JSONL data packs.
- **IShowFeed**: `{ article, has_left, has_right, filtered, channel, countRight }` — `countRight`: always `number` (never null); count of filtered articles strictly after `pos`, except in unseen-only tag mode where it also counts the article at `pos` **when that article is itself unread** (you're navigating the unread set, not sitting on the seen resume position you open the tag on) so it equals the tag's unread badge whenever no unseen sits to your left. `has_right` always tracks the strictly-after count.
- Dev (`npm run dev` / `make dev-fe`): `concurrently` runs Parcel HMR (port 1234) + a CORS pack server (`serve`, port 3000). The pack server serves the **configured local store**, not a fixed path: `parcel/resolve-store.js` resolves the directory from `$SRR_STORE` → `store:` in the active config → `../packs` fallback (same precedence as the backend; an `s3://`/`sftp://` store, missing config, or absent dir falls back with a stderr note). Select a store with `SRR_CONFIG=~/.config/srr/srr.tmp.yaml make dev-fe`. The `serve` script forces `SRR_CDN_URL=http://localhost:3000` (the config's `cdn-url:` is a production same-origin path the local app can't use) unless `SRR_CDN_URL` is already set. `serve`'s directory listing lets you browse the pack files at `http://localhost:3000`.

## Key Behaviors

**guard() mutex** (app.ts): all async UI actions go through it. Drops concurrent calls. On error → popup with retry.

**Eager fetch**: `data.ts` starts `fetch("db.gz")` at module load (before `init()` call).

**Init**: `data.init()` loads db.gz, the idx header summary `idx/h<hdrs>.gz` (when `db.hdrs === numFinalizedIdx()`), and the latest idx pack — O(1) requests at any store size. Finalized idx packs are fetched lazily by chronIdx addressing when navigation touches them; `idxHeaders` (one `IdxHeader` per pack, latest last) powers sync counting (`countAll`), pack-skip deltas (`packHasCandidate`), and the pack-level timestamp search without any fetch. When `hdrs` lags or the summary fetch fails, init falls back to the old eager fetch-all path and peels each pack's own header (`hdrs` stays optional on the wire type — an absent value simply never equals `numFinalizedIdx()`). Latest packs are generation-named `L<seq>.gz` (`db.seq`, normalized `??= 0` for the omitempty empty-store case); every pack name is write-once, so all pack fetches use `cache: "force-cache"`. A non-OK response on a latest pack means this tab's db.gz predates the backend GC grace window — `assertPackOk` triggers one `location.reload()` (sessionStorage-guarded against loops, guard cleared after a successful `init()`) and always throws so callers never touch the body.

**Article loading**: `loadArticle(chronIdx)` resolves pack+offset via binary search on `IdxPack.bounds`, fetches and parses the JSONL data pack (LRU-cached by pack ID, max 20 packs).

**Neighbor prefetch** (nav.ts): after each `left()`/`right()`, the neighbor lookup (`findLeft`/`findRight` — may lazily fetch an idx pack) runs un-awaited off the critical path: its promise lands in `nextLeft`/`nextRight` (consumed by the next keypress; a rejection clears the slot instead of failing the committed navigation) and on resolution `schedulePrefetch(target)` warms the neighbor's images on `requestIdleCallback` (`{timeout:500}`; WebKit has no rIC, so a `setTimeout(…, 200)` fallback keeps iOS/Safari prefetching instead of stalling at every data-pack boundary) — it `loadArticle()`s the neighbor, runs `fmt.extractImageUrls()` over its content, and for each URL creates an `Image` with `fetchPriority="low"`/`decoding="async"` and `img.src = imgProxy(...)`. `abortPrefetch()` cancels in-flight loads by setting `img.src=""` (WHATWG image-update steps) and dropping the refs. The `currentPrefetch` array (also the freshness token: a pending idle callback bails when `my !== currentPrefetch`) bounds memory to one neighbor at a time. Edits to nav must preserve this abort/freshness discipline.

**Content fade-in**: double `requestAnimationFrame` for opacity transition.

**Service worker** (`src/sw.ts`, registered best-effort from `app.ts` with `{type: "module"}` so it can import `LATEST_KEEP` from the generated contract — module-SW support predates the SW's `DecompressionStream` requirement in every engine, so the module type never limits compatibility): offline-capable cache. Lives at `src/` root (not `src/js/`) so Parcel emits it at the deployment root, giving it a default scope over the whole env (`/srr/`, `/srr.tmp/`, …). Buckets keyed by mutability: `srr-assets-v1` (content-hash `assets/…` — immutable, cache-first), `srr-packs-v3` (write-once pack names — finalized `idx|data|search/<n>.gz`, latest generations `idx|data|search/L<seq>.gz`, idx header summaries `idx/h<N>.gz`, and search bloom summaries `search/s<N>.gz` — cache-first with revalidate-on-miss; only `db.gz` is network-first → cache fallback; offline consistency is structural since a cached db.gz of seq N can only pair with the write-once `L<N>`), `srr-shell-v1` (navigation network-first + content-hashed JS/CSS cache-first; runtime-cached, no precache manifest — keeps the SW hand-written and zero-dep), `srr-meta-v1` (two synthetic entries: the last-seen db.gz `gen` and `seq`). **Manifest tracking** (`checkManifest`): on each successful online `db.gz` fetch whose `ETag`/`Last-Modified` differs from the cached copy's (an unchanged validator means unchanged bytes, so the common no-change load skips the parse and the redundant `cache.put` — a cheap header compare keeps the awaited check off the boot path's budget) the SW gunzips a clone (raw gzip bytes, no `Content-Encoding` — same manual `DecompressionStream` as `data.ts`) and reads `gen` + `seq` + `hdrs` + `srch` (absent == 0). One `parsePackName` helper owns the pack-name grammar (`series`/`kind`/`n`) for the fetch route, the cache bound, and the manifest prunes — built from the generated `PACK_SERIES_KINDS` table (backend `store.PackSeries`) and strict per-series like the store's `packKeyRe`: a kind letter another series owns (`data/h3.gz`) is not a pack name. A `gen` change (any change, not just increments) purges the whole packs bucket (in-place store rebuild — same names, new bytes); a `seq` change (gen unchanged) prunes cached `L<g>` latest generations with `g < seq - LATEST_KEEP` (imported from `format.gen.ts` = the backend's `latestKeep` GC grace window), idx summaries `h<g>` with `g < hdrs - LATEST_KEEP`, and search bloom summaries `search/s<g>` with `g < srch - LATEST_KEEP` (all ride the `seq` prune signal rather than separate meta keys — each can advance without a seq bump on a zero-article retry cycle, but that strands at most one stale name until the next article-producing fetch). Both awaited *before* the db.gz response resolves, so the purge is race-free against the page's subsequent idx/data requests. Best-effort: a parse failure never blocks serving db.gz. The operator bumps `srr gen --bump` after an in-place store rebuild; the `srr-packs-vN` constant now only marks cache-format changes. **Cache bound** (`enforceCacheBounds`): after each successful online `db.gz` fetch (via `event.waitUntil`, off the critical path — and never offline, so a device can't lose a pack it can't refetch) each finalized pack series is capped per `SERIES_KEEP` — `idx`/`data` at `PACK_KEEP` (100), `search` at `SEARCH_KEEP` (30, tighter because shards are ~1 MB each) — lowest-numbered evicted first (the numeric names encode chron age, so no access-time bookkeeping), and the assets bucket at `ASSET_KEEP` (500), oldest-cached first (Cache key insertion order; `cacheFirst` never re-puts on a hit). `db.gz` and the `L<seq>`/`h<N>`/`s<N>` names are exempt — offline consistency depends on them and checkManifest owns their pruning. Evicting a pack still being read just costs one CDN refetch on the next miss. Needs a secure context (HTTPS or localhost): production CF HTTPS qualifies, an `http://` LAN-IP deploy does not — registration just no-ops there and the app runs off the network. Coverage: `e2e/browser/reader.e2e.test.ts` ("serves the reader offline after one online visit", "purges stale finalized packs when db.gz gen changes", "bounds the pack and asset caches, evicting oldest entries first").

**Installable (PWA)** — `src/manifest.webmanifest` (linked from `index.html`) + the SW's fetch handler satisfy the browser install criteria. `start_url`/`scope` are `"."` (deployment-relative) so the same bundle installs under `/srr/` or `/srr.tmp/`. Icons: `src/icon.svg` is the single source of truth — the same RSS glyph as the inline favicon, recolored white on the brand-orange tile and inset into the maskable safe zone. The raster fallbacks are generated from it with `rsvg-convert -w N -h N src/icon.svg -o src/icon-N.png` (192, 512, and a 180 `apple-touch-icon.png`); the manifest also references `icon.svg` directly as a scalable `any` icon. Regenerate the PNGs whenever `icon.svg` changes. Coverage: `e2e/browser/reader.e2e.test.ts` ("exposes a valid, installable web app manifest").

## State Machines (nav.ts)

State: `pos` (chronIdx), `filter` (object with `active`, `channels`, `chanTotal`, `tokens`, `matches()`, `clear()`, `set(tokens)`).

**`filter`** — active when `tokens` non-empty:
- `filter.clear()`: empties tokens, repopulates `channels` from all channels with `total_art > 0`
- `filter.set(tokens)`: resolves tokens (numeric channel IDs or tag names) into `channels` map (`chan_id → add_idx`); falls back to `clear()` if no token resolves
- `last(token?)`: no arg = use current filter; `""` = clear filter; otherwise `filter.set([token])`; always jumps to last matching article
- `goTo(idx)`: navigate to chronIdx; if filter active and target doesn't match, snap forward to next match; falls back to `last()` if none
- `switchFilter(token)`: sets filter to token (or clears if `""`); resumes at last seen position for that channel/tag if valid (in unseen-only mode too — `isValidSeen` validates against the true `add_idx`, not the raised bound, so you open at your current position, not the oldest unseen), otherwise jumps to `first()`
- `cycleFilter(dir)`: steps through `getFilterEntries()` by `dir` (+1/-1), calls `switchFilter()`
- `getFilterEntries()`: returns `["", "tagName", ..., "channelId", ...]` built via `data.groupChannelsByTag()`
- `getCurrentFilterKey()`: returns `""`, the single token, or `""` for multi-token filters (URL-only edge case)
- Navigation uses `findLeft`/`findRight` — async (may lazily fetch an idx pack); finalized packs without a filter match are skipped via summary header deltas, no fetch. `filter.clear()`/`set()` stay synchronous via `data.countAll`
- Hash: `#pos[!tokens]` — `!` segment, `+`-separated tokens, each `encodeURIComponent`-wrapped to survive special chars in tag names.

**Time-range jumps** (dropdown.ts channel menu): "8h"/"16h"/"1d"/"7d" chips compute `Math.floor(Date.now()/1000) - seconds` (unix seconds; the chips carry `t:<seconds>` data-values — `t:28800`/`t:57600`/`t:86400`/`t:604800`), look up `data.findChronForTimestamp(ts)`, and call `nav.goTo(chron)` so the user lands at the article from that point and can navigate right. The calendar chip opens the date editor for arbitrary archive dates — the same `findChronForTimestamp` → `goTo` path with `ts` = local midnight of the picked day (O(1) requests at any depth, courtesy of the resident summary headers).

## Test Patterns

`src/js/nav.test.ts` — large nav suite. `src/js/data.test.ts` — pure-function cases only. `src/js/idx.test.ts` — idx binary-parsing unit tests. `src/js/fmt.test.ts` — sanitizeHtml / collapseBrokenMedia / timeAgo / formatDate / isValidProxy tests. `src/js/cache.test.ts` — LRU cache tests. `src/js/dropdown.test.ts` — image-proxy inline-editor, date-jump, feed-error-badge, unread-badge, menu-keyboard-navigation, headlines-peek, and search-menu tests (mocks `./data`/`./nav`/`./search`; seeds the dropdown DOM skeleton, then `vi.resetModules()` + dynamic import because the module captures DOM refs at load — each import adds another document-level keydown listener, which is why the handler only acts on its own attached menus and the keyboard/peek/search describes call `closeAllDropdowns()` in `afterEach`). `src/js/search.test.ts` — `fold`, `bloomBits` (cross-language parity pins against `db_search_test.go` vectors), `available` gate, `shortQuery`, and the `search()` async generator (mocks `./data` + `fetchPackBytes`; the module holds lazy slots so each test gets a fresh instance via `vi.resetModules()` + dynamic import).

**nav.test.ts**:
- **Mock**: `vi.hoisted()` + `vi.mock("./data", ...)` with same shape as data.ts exports. Mocks `getChannelId`, `loadArticle`, `groupChannelsByTag`, `findLeft`, `findRight`, `countLeft`, `findChronForTimestamp`.
- **Reset**: `beforeEach` resets data state, calls `nav.filter.clear()`, and re-spies `history.pushState`/`replaceState`.
- **Helpers**: `makeArticle(overrides)`, `makeChannel(overrides)` — factory with defaults. `setupIndex(entries)` — populates `db.channels` and wires `getChannelId`/`loadArticle` mocks.
- **Hash checks**: spy on `history.pushState`/`replaceState` (note the spy accumulates across tests in the same describe).

**data.test.ts**: mocks `./data` with inline reimplementations of `findChronForTimestamp` and `groupChannelsByTag` driven by writable `db`/`fetchedAts` state — data.ts's module-load `fetch` would otherwise fire under jsdom.

**E2e (`e2e/`)** — writer↔reader contract: the unit tests above mock `./data`; the e2e suite runs the REAL `srrb` binary to write packs from canned feeds (`e2e/harness.ts` `srr()` + in-process `feedServer()`, `e2e/fixtures.ts`), then reads them back with the REAL frontend code. Two layers + their own configs (excluded from `npm test` via `vitest.config.ts` `test.exclude`):
- **contract** (`e2e/contract/`, `vitest.contract.config.ts`, jsdom, in `make verify`): `mountReader()` installs a `fetch` shim mapping CDN URLs → store files (raw `.gz` bytes, no `Content-Encoding` — data.ts decompresses via `DecompressionStream`), `vi.resetModules()` + dynamic-imports the real `data.ts`/`nav.ts` (its module-load `db.gz` fetch must hit the shim, so stub-before-import), then asserts every chronIdx round-trips, pack splits, dedup/seq-generation advance, latest-pack GC grace window, and nav filtering. Cross-checks `srr inspect --validate`. `summary.e2e.test.ts` is the one synthetic-store exception: it hand-builds a 100k-article store (two finalized idx packs + `idx/h2.gz`) so the real reader proves the summary fast path (O(1) boot, lazy pack fetch, header-delta skip) and the eager fallback within the verify time budget — real ≥50k writer coverage lives in the backend Go tests. `search.e2e.test.ts` tests the search series end-to-end: real `srrb` writes a store with Unicode titles, the real `search.ts` reads it back and verifies `available()`, `search(q)` hits/misses, and seq-advance (a second fetch adds a new article and `search()` finds it). Bloom and 50k-shard coverage stays in Go tests (`db_search_test.go`) per the `summary.e2e.test.ts` precedent.
- **browser** (`e2e/browser/`, `vitest.browser.config.ts`, Puppeteer, opt-in via `make test-browser`/`test-e2e`): `serve.ts` global-setup builds the real bundle with relative `SRR_CDN_URL=/packs/` and serves it + a per-run pack dir from one origin; scenarios drive headless Chrome (render, keyboard nav, deep-link, tag filter, headlines peek). Reuses the Chromium under `~/.cache/puppeteer/` (`puppeteer` pinned to 25.0.2). Gotcha: set `Connection: close` + `server.closeAllConnections()` or `server.close()` stalls on Chrome keep-alive sockets.
- Content that must force data-pack splits has to be incompressible (`fixtures.ts` seeded alphanumeric) — packs roll on COMPRESSED size (`db_pack.go` `data.Len() >= PackSize<<10`).
