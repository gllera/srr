# CLAUDE.md

## Project

SRR Frontend — single-page RSS reader. Zero runtime deps. Parcel + TypeScript + plain CSS.

## Commands

`npm run dev|build|test|lint|format` — dev server, production build (`../dist/srrf/`), vitest+jsdom, ESLint, Prettier.

## Architecture

Entry: `src/index.html` → `src/styles.css` + `src/js/app.ts`. Bundler: Parcel 2. `SRR_CDN_URL` is replaced at build time via `parcel/resolve-cdn-url.js`. Resolution: env var → `cdn-url:` from `$SRR_CONFIG_INLINE` (raw YAML) → `cdn-url:` from `$SRR_CONFIG` (file path, default `$XDG_CONFIG_HOME/srr/srr.yaml`) → fallback `http://localhost:3000`. The config read (the INLINE → file → default chain) is shared with the dev pack server via `parcel/srr-config.js` (`readConfigYaml`/`parseKey`).

Dependency chain: `app → nav → {data, search, fmt}`, `nav → data → {idx, cache, base}`, `app → fmt → base`, `app → list → {data, nav, fmt}`, `app → dropdown → {data, nav}`, `app → gestures → {nav, dropdown}`, `{idx, data, sw, types.d} → format.gen` (generated, see below), `search → {data, cache, format.gen}`. All in `src/js/`, strict mode. (`nav` owns the search **filter mode**, so it imports `search`; `dropdown` no longer does.)

**Two surfaces, one toolbar.** The **list** (`list.ts`) is home: a scannable headline feed under the current filter, newest-first, **source-keyed** — each row carries a per-feed color rail + a colored mono "source · age" eyebrow over the title (`fmt.srcColorIndex` → `data-src`; read/unread shows as title weight/brightness + rail saturation, no separate dot). Above the feed sits the **front-page masthead** (`.srr-masthead`, `app.ts` `refreshMasthead`): a wire nameplate (`SRR · YOUR WIRE`, mono caps over a dashed teleprinter rule) + a live status line — the unread total across the **whole** wire (async, summed from `nav.unreadCounts`, the same machinery as the menu badges) and the last sync (`fmt.timeAgo(db.fetched_at)`). It's plain-flow (not sticky): `render()` lands at scrollTop 0 only when anchored at the newest edge, so the masthead greets you at the front of the wire and scrolls away into the archive; CSS hides it on the reader and during search. The **reader** (the `<article>`) is the drill-down — one article, prev/next: a focused reading surface with just back / save / prev / next. It's **source-keyed** like the list — `app.ts` stamps `data-src` (`fmt.srcColorIndex`) on `.srr-reader`, giving a full-height colored **spine** + a mono **masthead** (`.srr-source` in the source color) over an **absolute dateline** (`formatDate`, with the relative age on hover) followed by the **dispatch length** (`.srr-readlen`, `fmt.readMinutes` → "N MIN READ"; omitted with its separator when empty). The spine doubles as a **read-through gauge**: the `.srr-reader` border is a pale wash of the source color (the unread track) and an absolute `.srr-read-spine` bar inks the full color down it as you scroll (`--read` 0→1, driven by `gestures.ts`'s existing scroll listener; `app.ts` calls `gestures.syncReadProgress()` after each render so a short article that needs no scroll still reads as fully inked). Its **prose is serif** (`--font-serif`) while headings/masthead/code stay sans/mono (the "wire chrome wraps human prose" pairing); in-content links/blockquote stay neutral ink. At the article's end sits **read on** (`.srr-readon`, `app.ts` `renderReadon`): a "NEXT" divider (the masthead's perforated rule again — the wire keeps feeding) over a list-row-style preview of the next dispatch (source-colored eyebrow + headline in sans), tapping it = the toolbar's next; at the newest match it's a terminal "you're at the latest" line. The preview comes from `nav.peek("right")` (the neighbor the prefetch already warmed), token-guarded so a fast next/prev never paints a stale one. `app.ts` toggles them (`showList`/`showReader`, body class `srr-view-list` hides the reader-only prev/next + the save toggle + the back button). The toolbar is **one row of four equal slots** (`.srr-toolbar`, `display: grid; grid-template-columns: repeat(4, minmax(0, 1fr))`, no left/right groups): the DOM interleaves the two surfaces' controls (`back · filter ▾ · prev · search · next · unread · save · ⋯`) so each surface's *visible four* — the other surface's are `display:none`, so they don't take a grid cell — auto-place into slots 1–4 in the right order, nothing cornered and no dead center. The fixed slots are deliberate: a control never shifts when a neighbour changes width, so the **variable-width filter readout** ("ALL" vs a long feed/tag) ellipsizes inside its own slot (`min-width:0`, left-aligned) instead of pushing the search/unread/⋯ icons around. Edge controls hug the edges (`justify-self`: filter/back start, save/⋯ end); the rest center in their slot. The toolbar carries `z-index: 6` so its upward-opening menus clear the list's sticky day dividers (`z-index: 1`) and the sticky search bar (`z-index: 5`) — staying below the error popup (`z-index: 1000`). So the **list toolbar** is filter ▾ · search · unread · ⋯, and the **reader toolbar** is back · ‹ prev · next › · save — the prev/next pair centered in the thumb zone (`next` the emphasized accent pill; `prev` is now shown on mobile too, where it used to be hidden), flanked by back/save at the edges. Hidden per surface via `.srr-view-list .srr-prev/.srr-next/.srr-save/.srr-back` (reader-only) and `body:not(.srr-view-list) .srr-dropdown, .srr-search, .srr-unread` (list-only, incl. the pinned search bar). The filter/⋯ menus open upward, each anchored to the screen edge its trigger sits at — `#srr-feed-menu` left, `#srr-overflow-menu` right — so the fixed-width panel always opens inward on a narrow phone. The **⋯ overflow** menu (`showOverflowMenu`) holds **Jump to a date** (relocated off the toolbar — its row pops the native date picker on a hidden `<input type=date>` that now rides *inside* the ⋯ dropdown so the calendar anchors near ⋯) and **Image proxy…** (opens a centered settings dialog — `dropdown.showImgProxyDialog` — for the optional image-proxy prefix, no longer an inline editor in the menu). The **unread (catch-up) toggle** (`.srr-unread`, `app.ts` `toggleUnseenOnly`/`refreshUnreadButton`) is a list-only toolbar button — one tap to show only unread in whatever's filtered. Unseen-only is **global** now ([ALL]/feed/tag, not just a single tag): `nav.filter.applyUnseen` raises every member's bound past its seen high-water, so it's the catch-up button for the whole wire. It's an outline dot that fills accent when on (`aria-pressed`). The search button toggles the `q:<query>` list filter and reveals the pinned search bar; `nav.last()` ("open the latest in the current filter") survives only as the reader's `e` key, not a toolbar button. Hash scheme: a **numeric position** (`#pos[!tokens]`) is the reader; **no position** (`#` or `#!tokens`) is the list at that filter — where a `q:<query>` token (URL-encoded, e.g. `#!q%3Aclimate`) is the **title-search filter mode** (see `nav.ts`). `app.route(hash)` picks the surface on boot and every `hashchange`; row-tap → `nav.goTo(chron)` (pushState reader hash) → reader; the toolbar back button (`goToList(true)`) / two-finger cycle / picking a feed-or-tag from the menu → the list (the dropdown's list-vs-reader behavior is the optional `FeedMenuHost` it's given). `srr-hash` persistence covers both surfaces, so a reload resumes where you were. Tapping a row is the commit into the reader; a filter change on the list re-filters the list and pushes a `#!filter` history entry without touching the reader's position, so browser-back walks back through those filters.

**Saved articles (`★`).** A per-article collection orthogonal to the feed/tag/seen axes: a device-local chronIdx set in `localStorage` `srr-saved` (`nav.toggleSaved`/`isSaved`/`savedCount`/`currentChron`). chronIdx is a permanent article address (finalized packs are immutable and never GC'd), so a saved id stays loadable forever — even after its feed is deleted (`feedTitle` then shows the `[DELETED]` tombstone). "★ Saved" is a **distinct nav mode** (`filter.saved`, reserved token `SAVED_TOKEN = "~saved"`): `filter.matches` becomes saved-set membership (feed-agnostic) and navigation walks the explicit set, not the idx packs. Both nav and the list route their neighbor walks through one seam — `nav.feedLeft`/`feedRight` (saved → a sync set scan, else `data.findLeft`/`findRight`); `step`/`last`/`goTo`/`showFeed` and `list.loadOlder` all go through it, so saved mode is decided in one place. The reader has a `.srr-save` star toggle (key `b`, `aria-pressed`); list rows carry a tappable `.srr-row-star` (filled on a saved row) that toggles without opening the reader and, in the Saved view, drops the row on un-save. `getFilterEntries` surfaces `~saved` in the keyboard/two-finger rotation only when something is saved; the feed menu shows a "★ Saved" row (count badge) under `[ALL]`, picked via the same `selectFilter`/`switchFilter` path as a feed.

| `idx.ts` | Binary idx pack parsing. Imports the format constants from `format.gen.ts`; re-exports `IDX_PACK_SIZE`. Each entry is 3 bytes (`feed_id:u16 LE` assembled manually + `packed:u8`); the header is variable-length — `IDX_HEADER_PREFIX` (3 state u32s + a `numSlots` u32), then `numSlots` cumulative `feedCounts` (`Uint32Array`), dense up to the high-water feed_id present when the pack was written. | `data.ts` | CDN data layer: fetches `db.gz`, then the idx header summary (`idx/h<hdrs>.gz`) + latest idx pack only; finalized idx packs load lazily on first touch (`fetchIdxPack`, promise-deduped slots) with eager fetch-all as the fallback when `hdrs` lags or the summary 404s. | `nav.ts` | Navigation state machine: hash routing (`#pos[!tokens]`), traversal, filtering. Returns `IShowFeed`. `peek(dir)` is a non-navigating neighbor lookup (`feedLeft`/`feedRight` + the deduped `loadArticle` cache) → `{chron, article}` or null at the edge, respecting every filter mode — it feeds the reader's "read on" preview. Uses `pushState`/`replaceState`. Tokens are feed IDs or tag names. Exports `filter`, `fromHash`, `left`, `right`, `first`, `last`, `switchFilter`, `goTo`, `seek` (position the list cursor at a chronIdx — snap-forward like `goTo` but WITHOUT loading the article or opening the reader; the date jump uses it to reposition the list), `select` (set the cursor to an exact, already-known-matching chronIdx — the list's keyboard selection; no feed walk / idx fetch, no hash update or `recordSeen`, since moving the cursor isn't reading), `cycleFilter`, `getFilterEntries`, `getCurrentFilterKey`, `isSingleFilter`, `pruneSeen`, `unreadCount`, `unreadCounts`, `tagUnreadFromCounts`, `isUnreadOnly`, `setUnreadOnly`, `feedLeft`/`feedRight` (the explicit-set-vs-feed neighbor seam), `isSaved`/`toggleSaved`/`savedCount`/`currentChron`/`anchorChron` (the reader's `pos` when it still matches the active filter, else -1)/`listAnchor` (the **list** anchor — `anchorChron` when the reader's article still matches, else a tag/feed's `getSeen` resume position when that position is still a row of the active filter, else that tag/feed's **oldest** matching article so a never-opened filter starts at the beginning of its backlog with the unread rows above it, else -1 = newest for `[ALL]`/saved/search; mirrors `switchFilter`'s reader resume) and `SAVED_TOKEN` (saved articles), and `SEARCH_PREFIX`/`isSearchFilter`/`searchQuery`/`searchTruncated`/`searchAvailable`/`searchShort` (the search filter mode — below). `unreadCount(ch)` = the feed's articles strictly after its `srr-seen` position (on each view `recordSeen` advances the seen frontier of **every feed in the active filter** to `pos`: the shown article's own feed to that exact resume position, every OTHER feed in the navigation list by a one-way high-water — so opening an article marks all OLDER articles across the list seen, not just same-feed ones, and reading the newest in [ALL] clears every count. Search/`q:` mode is exempt — it returns before recording — and saved mode has no filter feeds, so each records only the shown article's own feed); both sides of the subtraction are idx-derived (`countAll` − `countLeft(seen+1)`) so db.gz `total_art` drift can't skew it; a feed never seen on this device counts as fully unread (its full backlog — `countAll`, a sync no-fetch count), so a fresh device badges every feed and the row badges sum to their tag header. A tag has no count of its own — the dropdown (`tagUnreadFromCounts`) sums its member feeds' `unreadCount`s onto the tag header. A **tag also has no stored seen position of its own**: `getSeen(tag)` derives it as the *oldest* read (min chronIdx) of the tag's member feeds, the anchor `switchFilter(tag)` resumes at, so opening a tag lands at its least-recently-read member and every unread article in the tag sits to the right (none skipped on the left). `pruneSeen` drops any legacy `tag:` key (tags stopped storing their own position). **Unseen-only mode** (`isUnreadOnly`/`setUnreadOnly`, persisted in `srr-unread-only`; device-local, not in the hash) applies to **every filter** ([ALL]/single feed/tag) — the one-tap catch-up view: when on, `filter.applyUnseen` (shared by `set` and `clear`, so it covers [ALL] too) raises EACH filtered feed's lower bound to `max(add_idx, seen+1)` (snapshotted at apply time, so the nav set is stable as you read this session), so `findLeft`/`findRight`/`matches`/search skip already-seen articles. The "member" wording below generalises from tags to whatever feeds the filter spans. The tag still opens at its **current position** — the resume anchor `getSeen` returns (its least-recently-read member, same as outside unseen-only), NOT its oldest unseen: `isValidSeen` validates that resume position against the true `add_idx` rather than the raised bound, so the raised bounds skip seen articles on Left/Right without bouncing the open forward. From the resume position, Right steps onto the first unseen. `showFeed`'s `countRight` reports unread-remaining (it branches to `unreadTally` — per-member `countAll − countLeft`, summed — because countLeft's cumulative-header shortcut is only exact for true `add_idx` bounds), counting the article you're on **when it is itself unread** (Option A's `+matchesPos`): at the resume position you're on a seen article, so `countRight` is just the unread to your right — equal to the tag's dropdown badge whenever no unseen sits to your left (a never-seen member with older articles can leave some there, reachable with Left); stepping onto an unseen article then counts it too, so `countRight` still equals the badge and counts down to 1 on the last unseen — mirroring a feed, which resumes on an already-read article and shows its full unread count to the right. `has_right` still tracks the strictly-after count, so next disables on the last unseen. The dropdown badge stays in lockstep: in unseen-only mode `feedUnread` (so `unreadCount`/`unreadCounts`/`tagUnreadFromCounts`) counts the **unread** article you're sitting on back into its feed — `recordSeen` marks it seen the instant you land, so without this the live-seen-derived badge would read one *below* the snapshot-based `countRight` as you read — and it ticks down with `countRight`. Scoped to the current article's feed (`currentFeed`), to unseen-only mode (`filter.unreadMembers !== null`), and to while that article actually matches the raised filter (`filter.matches(currentFeed, pos)`) — i.e. it's one of the unread being navigated, not the seen resume position you open on (there `matchesPos` is 0 and `countRight` doesn't count it either) — exactly mirroring `showFeed`'s `+matchesPos`; OTHER feeds' badges are untouched. (The `unreadMembers` machinery is generic over whatever feeds the filter holds, so this all applies equally to `[ALL]` and single-feed filters now, not just tags.) **Search filter mode** is a third explicit-set mode beside feed-membership and ★ Saved: when the single token is `q:<query>` (`SEARCH_PREFIX`), `filter.search` is set and navigation walks an explicit set of matching chronIdxs — the title-search hits — exactly as ★ Saved walks the saved set (`feedLeft`/`feedRight`/`matches`/`showFeed` all branch on it, so **the list and the reader walk it for free** with no extra plumbing). The set is computed once per query from the `meta/` shards (`ensureSearchSet` → `search.search(term, SEARCH_CAP=500)`, newest-first, capped; `searchTruncated()` flags the cap), cached, and recomputed when the query changes; it commits only if the term is still active when it resolves (a superseded keystroke can't overwrite fresher hits). The query rides in the shareable `#!q:<query>` hash like any token (reload/back/forward restore it; `tokensSuffix` escapes a literal `+` in the query to `%2B` so the `+`-separator split survives), and it behaves like a tag filter — picking another filter / two-finger cycle / arrow-cycle leaves it (search is **not** in `getFilterEntries`, so you can't cycle *into* it). `switchFilter`/`last` open it at the newest hit; `fromHash` loads the set before validating a `#pos!q:…` deep-link so the position is honored. The UI (toolbar magnifier + pinned search bar) lives in `app.ts`. |

## Data Structures

See root `CLAUDE.md` Data Contract for db.gz, IFeed, IArticle, pack format, CDN layout, and chronIdx.

Frontend-specific additions:
- `feeds` in `IDB` is `Record<number, IFeed>` (JSON object keyed by feed ID); defaults to `{}` if absent. `feed.id` is populated from object keys at init.
- **IArticle**: `{ f, a, p, t, l, c }` — feed_id, fetched_at, published, title, link, content. Loaded from JSONL data packs.
- **IShowFeed**: `{ article, has_left, has_right, filtered, feed, countRight }` — `countRight`: always `number` (never null); count of filtered articles strictly after `pos`, except in unseen-only mode where it also counts the article at `pos` **when that article is itself unread** (you're navigating the unread set, not sitting on the seen resume position you open the tag on) so it equals the tag's unread badge whenever no unseen sits to your left. `has_right` always tracks the strictly-after count.
- Dev (`npm run dev` / `make dev-fe`): `concurrently` runs Parcel HMR (port 1234) + a CORS pack server (`serve`, port 3000). The pack server serves the **configured local store**, not a fixed path: `parcel/resolve-store.js` resolves the directory from `$SRR_STORE` → `store:` in the active config → `../packs` fallback (same precedence as the backend; an `s3://`/`sftp://` store, missing config, or absent dir falls back with a stderr note). Select a store with `SRR_CONFIG=~/.config/srr/srr.tmp.yaml make dev-fe`. The `serve` script forces `SRR_CDN_URL=http://localhost:3000` (the config's `cdn-url:` is a production same-origin path the local app can't use) unless `SRR_CDN_URL` is already set. `serve`'s directory listing lets you browse the pack files at `http://localhost:3000`.

## Key Behaviors

**guard() mutex** (app.ts): all async UI actions go through it. Drops concurrent calls. On error → popup with retry.

**Eager fetch**: `data.ts` starts `fetch("db.gz")` at module load (before `init()` call).

**Init**: `data.init()` loads db.gz, the idx header summary `idx/h<hdrs>.gz` (when `db.hdrs === numFinalizedIdx()`), and the latest idx pack — O(1) requests at any store size. Finalized idx packs are fetched lazily by chronIdx addressing when navigation touches them; `idxHeaders` (one `IdxHeader` per pack, latest last) powers sync counting (`countAll`), pack-skip deltas (`packHasCandidate`), and the pack-level timestamp search without any fetch. When `hdrs` lags or the summary fetch fails, init falls back to the old eager fetch-all path and peels each pack's own header (`hdrs` stays optional on the wire type — an absent value simply never equals `numFinalizedIdx()`). Latest packs are generation-named `L<seq>.gz` (`db.seq`, normalized `??= 0` for the omitempty empty-store case); every pack name is write-once, so all pack fetches use `cache: "force-cache"`. A non-OK response on a latest pack means this tab's db.gz predates the backend GC grace window — `assertPackOk` triggers one `location.reload()` (sessionStorage-guarded against loops, guard cleared after a successful `init()`) and always throws so callers never touch the body.

**Article loading**: `loadArticle(chronIdx)` resolves pack+offset via binary search on `IdxPack.bounds`, fetches and parses the JSONL data pack (LRU-cached by pack ID, max 20 packs).

**Neighbor prefetch** (nav.ts): after each `left()`/`right()`, the neighbor lookup (`findLeft`/`findRight` — may lazily fetch an idx pack) runs un-awaited off the critical path: its promise lands in `nextLeft`/`nextRight` (consumed by the next keypress; a rejection clears the slot instead of failing the committed navigation) and on resolution `schedulePrefetch(target)` warms the neighbor's images on `requestIdleCallback` (`{timeout:500}`; WebKit has no rIC, so a `setTimeout(…, 200)` fallback keeps iOS/Safari prefetching instead of stalling at every data-pack boundary) — it `loadArticle()`s the neighbor, runs `fmt.extractImageUrls()` over its content, and for each URL creates an `Image` with `fetchPriority="low"`/`decoding="async"` and `img.src = imgProxy(...)`. `abortPrefetch()` cancels in-flight loads by setting `img.src=""` (WHATWG image-update steps) and dropping the refs. The `currentPrefetch` array (also the freshness token: a pending idle callback bails when `my !== currentPrefetch`) bounds memory to one neighbor at a time. Edits to nav must preserve this abort/freshness discipline.

**Content fade-in**: double `requestAnimationFrame` for opacity transition.

**Service worker** (`src/sw.ts`, registered best-effort from `app.ts` with `{type: "module"}` so it can import `LATEST_KEEP` from the generated contract — module-SW support predates the SW's `DecompressionStream` requirement in every engine, so the module type never limits compatibility): offline-capable cache. Lives at `src/` root (not `src/js/`) so Parcel emits it at the deployment root, giving it a default scope over the whole env (`/srr/`, `/srr.tmp/`, …). Buckets keyed by mutability: `srr-assets-v1` (content-hash `assets/…` — immutable, cache-first), `srr-packs-v3` (write-once pack names — finalized `idx|data|meta/<n>.gz`, latest generations `idx|data|meta/L<seq>.gz`, idx header summaries `idx/h<N>.gz`, and meta bloom summaries `meta/s<N>.gz` — cache-first with revalidate-on-miss; only `db.gz` is network-first → cache fallback; offline consistency is structural since a cached db.gz of seq N can only pair with the write-once `L<N>`), `srr-shell-v1` (navigation network-first + content-hashed JS/CSS cache-first; runtime-cached, no precache manifest — keeps the SW hand-written and zero-dep), `srr-meta-v1` (two synthetic entries: the last-seen db.gz `gen` and `seq`). **Manifest tracking** (`checkManifest`): on each successful online `db.gz` fetch whose `ETag`/`Last-Modified` differs from the cached copy's (an unchanged validator means unchanged bytes, so the common no-change load skips the parse and the redundant `cache.put` — a cheap header compare keeps the awaited check off the boot path's budget) the SW gunzips a clone (raw gzip bytes, no `Content-Encoding` — same manual `DecompressionStream` as `data.ts`) and reads `gen` + `seq` + `hdrs` + `mp` (absent == 0). One `parsePackName` helper owns the pack-name grammar (`series`/`kind`/`n`) for the fetch route, the cache bound, and the manifest prunes — built from the generated `PACK_SERIES_KINDS` table (backend `store.PackSeries`) and strict per-series like the store's `packKeyRe`: a kind letter another series owns (`data/h3.gz`) is not a pack name. A `gen` change (any change, not just increments) purges the whole packs bucket (in-place store rebuild — same names, new bytes); a `seq` change (gen unchanged) prunes cached `L<g>` latest generations with `g < seq - LATEST_KEEP` (imported from `format.gen.ts` = the backend's `latestKeep` GC grace window), idx summaries `h<g>` with `g < hdrs - LATEST_KEEP`, and meta bloom summaries `meta/s<g>` with `g < mp - LATEST_KEEP` (all ride the `seq` prune signal rather than separate meta keys — each can advance without a seq bump on a zero-article retry cycle, but that strands at most one stale name until the next article-producing fetch). Both awaited *before* the db.gz response resolves, so the purge is race-free against the page's subsequent idx/data requests. Best-effort: a parse failure never blocks serving db.gz. The operator bumps `srr gen --bump` after an in-place store rebuild; the `srr-packs-vN` constant now only marks cache-format changes. **Cache bound** (`enforceCacheBounds`): after each successful online `db.gz` fetch (via `event.waitUntil`, off the critical path — and never offline, so a device can't lose a pack it can't refetch) each finalized pack series is capped per `SERIES_KEEP` — `idx`/`data` at `PACK_KEEP` (100), `meta` at `META_KEEP` (30, tighter because shards are ~100 KB gz each) — lowest-numbered evicted first (the numeric names encode chron age, so no access-time bookkeeping), and the assets bucket at `ASSET_KEEP` (500), oldest-cached first (Cache key insertion order; `cacheFirst` never re-puts on a hit). `db.gz` and the `L<seq>`/`h<N>`/`s<N>` names are exempt — offline consistency depends on them and checkManifest owns their pruning. Evicting a pack still being read just costs one CDN refetch on the next miss. Needs a secure context (HTTPS or localhost): production CF HTTPS qualifies, an `http://` LAN-IP deploy does not — registration just no-ops there and the app runs off the network. Coverage: `e2e/browser/reader.e2e.test.ts` ("serves the reader offline after one online visit", "purges stale finalized packs when db.gz gen changes", "bounds the pack and asset caches, evicting oldest entries first").

**Installable (PWA)** — `src/manifest.webmanifest` (linked from `index.html`) + the SW's fetch handler satisfy the browser install criteria. `start_url`/`scope` are `"."` (deployment-relative) so the same bundle installs under `/srr/` or `/srr.tmp/`. Icons: `src/icon.svg` is the single source of truth — the same RSS glyph as the inline favicon, recolored white on the brand-orange tile and inset into the maskable safe zone. The raster fallbacks are generated from it with `rsvg-convert -w N -h N src/icon.svg -o src/icon-N.png` (192, 512, and a 180 `apple-touch-icon.png`); the manifest also references `icon.svg` directly as a scalable `any` icon. Regenerate the PNGs whenever `icon.svg` changes. Coverage: `e2e/browser/reader.e2e.test.ts` ("exposes a valid, installable web app manifest").

## State Machines (nav.ts)

State: `pos` (chronIdx), `filter` (object with `active`, `feeds`, `feedTotal`, `tokens`, `matches()`, `clear()`, `set(tokens)`).

**`filter`** — active when `tokens` non-empty:
- `filter.clear()`: empties tokens, repopulates `feeds` from all feeds with `total_art > 0`
- `filter.set(tokens)`: resolves tokens (numeric feed IDs or tag names) into `feeds` map (`feed_id → add_idx`); falls back to `clear()` if no token resolves
- `last(token?)`: no arg = use current filter; `""` = clear filter; otherwise `filter.set([token])`; always jumps to last matching article
- `goTo(idx)`: navigate to chronIdx; if filter active and target doesn't match, snap forward to next match; falls back to `last()` if none
- `switchFilter(token)`: sets filter to token (or clears if `""`); resumes at last seen position for that feed/tag if valid (in unseen-only mode too — `isValidSeen` validates against the true `add_idx`, not the raised bound, so you open at your current position, not the oldest unseen), otherwise jumps to `first()`
- `cycleFilter(dir)`: steps through `getFilterEntries()` by `dir` (+1/-1), calls `switchFilter()`
- `getFilterEntries()`: returns `["", "tagName", ..., "feedId", ...]` built via `data.groupFeedsByTag()`
- `getCurrentFilterKey()`: returns `""`, the single token, or `""` for multi-token filters (URL-only edge case)
- Navigation uses `findLeft`/`findRight` — async (may lazily fetch an idx pack); finalized packs without a filter match are skipped via summary header deltas, no fetch. `filter.clear()`/`set()` stay synchronous via `data.countAll`
- Hash: `#pos[!tokens]` — `!` segment, `+`-separated tokens, each `encodeURIComponent`-wrapped to survive special chars in tag names.

**Date jump** (dropdown.ts `openDatePicker`/`dateJump`, app.ts `.srr-jump` button + `jumpToDate`): the button pops the browser's native date picker directly (no menu, time presets, or text-entry step). Picking a day computes `ts` = local midnight of that day; `dateJump` hands it to `jumpToDate`, which looks up `data.findChronForTimestamp(ts)`, `nav.seek`s the cursor to the first article at-or-after that point (snapping forward through the active filter, **without** loading the article or opening the reader), then re-renders the **list** anchored there (the seeked article at the top, newer above, older below). So the date jump scrubs the *list* timeline — it does **not** open the reader (the list is home; tapping a row is the commit into the reader). O(1) requests at any archive depth, courtesy of the resident summary headers. The picker is clamped to `[first_fetched day, today]`. (The old "8 hours ago"/"16 hours ago"/1d/7d presets and the "Pick a date…" text-entry step were dropped; "latest" lives on the reader's `e` key = `nav.last()`.)

**E2e (`e2e/`)** — writer↔reader contract: the unit tests above mock `./data`; the e2e suite runs the REAL `srrb` binary to write packs from canned feeds (`e2e/harness.ts` `srr()` + in-process `feedServer()`, `e2e/fixtures.ts`), then reads them back with the REAL frontend code. Two layers + their own configs (excluded from `npm test` via `vitest.config.ts` `test.exclude`):
- **contract** (`e2e/contract/`, `vitest.contract.config.ts`, jsdom, in `make verify`): `mountReader()` installs a `fetch` shim mapping CDN URLs → store files (raw `.gz` bytes, no `Content-Encoding` — data.ts decompresses via `DecompressionStream`; the shim serves `meta/*.gz` generically along with `db.gz`/`idx/*.gz`/`data/*.gz`), `vi.resetModules()` + dynamic-imports the real `data.ts`/`nav.ts` (its module-load `db.gz` fetch must hit the shim, so stub-before-import), then asserts every chronIdx round-trips, pack splits, dedup/seq-generation advance, latest-pack GC grace window, and nav filtering. Cross-checks `srr inspect --validate`. `summary.e2e.test.ts` is the one synthetic-store exception: it hand-builds a 100k-article store (two finalized idx packs + `idx/h2.gz`) so the real reader proves the summary fast path (O(1) boot, lazy pack fetch, header-delta skip) and the eager fallback within the verify time budget — real ≥50k writer coverage lives in the backend Go tests. `search.e2e.test.ts` tests the meta/ series end-to-end (search now reads the meta/ projection): real `srrb` writes a store with Unicode titles, the real `search.ts` reads it back and verifies `available()`, `search(q)` hits/misses, seq-advance (a second fetch adds a new article and `search()` finds it), and `srr inspect --validate` coverage. `metafallback.e2e.test.ts` tests the data/ fallback: by setting `db.mp=0/mt=0` (forcing `metaReady()=false`) after a real fetch, it proves `loadMeta(chron)` still returns the correct `{f,w,t}` card by reading `data/`. Bloom and 5k-shard coverage stays in Go tests (`db_meta_test.go`) per the `summary.e2e.test.ts` precedent.
- **browser** (`e2e/browser/`, `vitest.browser.config.ts`, Puppeteer, opt-in via `make test-browser`/`test-e2e`): `serve.ts` global-setup builds the real bundle with relative `SRR_CDN_URL=/packs/` and serves it + a per-run pack dir from one origin; scenarios drive headless Chrome (render, keyboard nav, deep-link, tag filter, search, saved articles). Reuses the Chromium under `~/.cache/puppeteer/` (`puppeteer` pinned to 25.0.2). Gotcha: set `Connection: close` + `server.closeAllConnections()` or `server.close()` stalls on Chrome keep-alive sockets.
- Content that must force data-pack splits has to be incompressible (`fixtures.ts` seeded alphanumeric) — packs roll on COMPRESSED size (`db_pack.go` `data.Len() >= PackSize<<10`).
