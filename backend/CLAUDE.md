# CLAUDE.md

## Project

**SRR Backend** — Static RSS Reader Backend. Go CLI that fetches RSS/Atom/RDF feeds into gzip-compressed pack series (idx/, data/). Backends: local filesystem, S3, SFTP, HTTP.

## Commands

The root `Makefile` is the unified entry point for both halves of the monorepo; prefer it over raw `go` commands:

- `make build-be` — build the binary to `dist/srr`
- `make test-be` — `go test ./...`
- `make vet-be` — `go vet ./...`
- `make format-be` / `make format-check-be` — `gofmt -w .` / gofmt gate (the latter runs inside `make verify-be`)
- `make lint-be` — `golangci-lint run ./...` (gate-clean; runs inside `verify-be` — config in `.golangci.yml`: std-error-handling preset, errcheck + SA5011 off in tests)
- `make generate` / `make generate-check` — regenerate / freshness-check `frontend/src/js/format.gen.ts` (`srr gen-ts`)
- `make verify-be` — vet + gofmt check + lint + build + test + generate-check
- `make release VERSION=…` — cross-compile all platforms (`CGO_ENABLED=0 -ldflags "-s -w"`)

Direct equivalents when working inside `backend/`: `go build -o srr .` (the Makefile artifact is `dist/srr`), `go test ./...`, `go test -run TestName .`, `go test -v ./store/`.

## Architecture

- **`main.go`** — CLI via `alecthomas/kong` + YAML config. Source: `$SRR_CONFIG_INLINE` (raw YAML content, takes precedence) → `$SRR_CONFIG` (file path) → `$XDG_CONFIG_HOME/srr/srr.yaml`. (Remote URL config sources and a `srr config <key> <value>` file updater were built then deliberately reverted 2026-07-04 — `SRR_CONFIG` is a local path and `srr config` is read-only.) `Globals` struct for flags. Command groups: `feed` (alias `f`): `add`, `upd`, `rm`, `ls`, `show`, `edit`, `apply`, `import`, `export`; `art` (alias `a`): `fetch`, `ls`; `asset`: `heal` (`cmd_asset.go` — overwrite an existing `assets/` object's bytes in place, the operator repair path for a published-broken asset: key grammar enforced via `assetKeyRe` (renamed from `healKeyRe` — now also the validation gate for expiration's asset harvesting, see `db_expire.go`), must already exist unless `--create` (re-create a referenced-but-deleted object), Content-Type from `--content-type` or the key extension; a deliberate hash≠bytes break since immutable articles reference the key forever); `recipe`; `syndicate`; `gen`; `preview` (alias `p`); `config` (alias `c`); `inspect` (alias `i`); `frontend` (alias `fe`): `update`; `version`; `serve`; hidden `gen-ts`. The optional top-level `secrets:` YAML section (a flat env-var name→value map) is parsed by `parseSecrets` (`secrets.go`) and pushed via `mod.SetSecrets` into `mod.SubprocessEnv`, which merges it into every external ingest/mod command's environment (secrets win over the ambient env; NOT applied to asset-process/asset-peek). An empty name or a name containing `=` is a hard error; values stay plaintext in `srr.yaml`.
- **`cmd_inspect.go` / `cmd_inspect_check.go` / `cmd_inspect_report.go`** — `InspectCmd` (`srr inspect`): pack consistency checker. Split across three files by responsibility: command entry + fetchers (`cmd_inspect.go`: `openFetcher`/`httpFetcher`/`loadCore`/`loadDataPack`), validation checks invoked by `--validate` (`cmd_inspect_check.go`), single-mode reports for `--chron`/`--filter`/`--list-tags`/`--from-hash` (`cmd_inspect_report.go`). The idx parse + addressing mirror itself lives in `idx_read.go` (shared with `srr art ls`). Modes: `--chron N` (single-entry inspection: feed_id + resolved data packId/offset + the stored fetched_at), `--validate` (full sweep: bounds-vs-data, db-meta cross-check, feedCounts continuity across pack boundaries, unknown feed_ids, latest-pack file presence, idx header-summary coverage/content vs `hdrs`, meta coverage/consistency via `checkMeta` — overclaim, latest tail count vs `mt`, shard sizes, every title's grams probe positive in its shard bloom, summary == concat of shard headers; db-meta expiration checks: `0 ≤ add_idx ≤ total_articles` and `0 ≤ xp ≤ total_art` range checks plus a live-count cross-check — live idx entries at `chron ≥ add_idx` must equal `total_art − xp` (`feedIDStats`; reuse-proof, since a reused id's legacy entries sit below `add_idx`) — replacing the pre-expiration "first idx occurrence < add_idx" issue, which is now expected behavior; the all-time `total_art == idx count` check stays, a known id-reuse limitation, commented in place), `--filter <tag|feed_id>` (filter resolution + count, mirrors frontend filter math — sums live counts, `TotalArt − Expired`), `--from-hash '<hash>'` (replays `nav.fromHash` end-to-end), `--list-tags` (mirrors `groupFeedsByTag`; per-tag article totals are live counts too). `--url` for HTTP CDN; otherwise uses `--store`. Diagnoses the `(reading 'f')` crash class in frontend `nav.ts` (`showFeed` accesses `data.db.feeds[article.f]`).
- **`cmd_fetch.go`** — `signal.NotifyContext` for graceful shutdown. `errgroup` (`golang.org/x/sync/errgroup`) with `SetLimit(globals.Workers)` and a `sync.Pool` for feed buffers. Articles sorted by published time (ascending) before storage. Order: `PutArticles` → `SyncIdxSummary` → `SyncMeta` → `ExpireArticles` → `SyncOutFeeds` → `Commit` → GC sweeps — expiration runs before the out-feed sync so the same cycle's syndication already excludes what just expired; warn-only like the sync steps, and it runs on every cycle including serve's `--interval` loop and the GUI single-feed fetch (`only`-mode) — global maintenance, like `SyncIdxSummary`/`SyncMeta` (production `now` = `db.core.FetchedAt`) (all sync/GC steps warn-only; each GC sweep runs only when its counter — `Seq`/`HdrPacks`/`MetaPacks` — advanced this run, since most cycles fetch nothing and would just re-delete an already-empty window; a crash-leaked name is still swept by the next advancing run) → the ingest-cache age sweep (`sweepAssetCache`, `--cache-max-age`/`SRR_CACHE_MAX_AGE` default 72h, 0 disables; post-Commit only, deletes cache files whose mtime aged out — see "External file self-hosting" for the mtime-as-relevance contract). `--interval` / `SRR_FETCH_INTERVAL` duration flag runs fetch in a loop; the loop body is `FetchCmd.fetchLoop(ctx, client)` (single cycle when interval ≤ 0, else cycle→sleep→repeat until ctx cancel, returning nil on clean shutdown and logging — not propagating — a failed cycle), shared with `srr serve --interval`. **Feed-subset filter** (`feedFilter`, embedded in both `FetchCmd` and `ServeCmd` so kong flattens the flags into each): `--tag`/`-g`, `--feed`/`-i`, `--exclude-tag`, `--exclude-feed` — all `sep:","` (comma-separated **or** repeated), env `SRR_FETCH_TAG`/`_FEED`/`_EXCLUDE_TAG`/`_EXCLUDE_FEED` (an empty env value is a no-op, kong yields an empty slice), so the same restriction reaches the persistent `srr serve --interval` loop (serve copies its `feedFilter` into the constructed `FetchCmd`). `FetchCmd.selectFeeds(db)` resolves the cycle's feeds: the GUI single-feed `only` path is unchanged (exact ids, hard error on unknown); otherwise `feedFilter.apply` builds the include set = union of tag matches (`matchTag`: **hierarchical** — exact, or under the `sel/` subtree, so `news` matches `news/tech` but not the sibling `news2`) and feed-id matches — or **every feed when no include selector is given** — then subtracts the exclude matches (returned sorted by id) — and finally, on the unattended loop path only, the **dormancy backoff** thins the result (`backoffActive`/`targetInterval`/`filterDue`): each feed's poll interval drifts up as `(now − last_new)/8` from the loop `--interval` base (first skip after ~8× base of quiet), capped at `--fetch-backoff-max`/`SRR_FETCH_BACKOFF_MAX` (default 1h; 0 = kill switch, poll everything every cycle), keyed on `LastOK` as the poll clock (`now` = `db.core.FetchedAt`). A feed that produces snaps back to base next cycle (its `LastNew` moves to ~now); failing feeds freeze `LastOK` and are never skipped (backoff must not mask an outage); one-shot runs, the GUI `only` path, and explicit include selectors (`--tag`/`--feed`) always poll at full rate. Warn-only, never abort (safe for a shared per-box config): each selector matching no feed in the store, and an empty final set, log one `slog.Warn`; a zero-feed cycle still runs global maintenance (expiration/sync/GC), like the `only` path. Error summary (fetched/failed counts) logged at end. The end-of-pipeline asset-upload step is parallel too: `Feed.fetchURL` defers self-hosting to `fetchRun.uploadAssets`, which fans out marker-bearing items over a per-feed `errgroup`. The **run-global** asset worker pool lives on the shared `assetFetcher` (`a.sem`, sized by `SRR_ASSET_WORKERS`/`--asset-workers`, default nproc, independent of `--workers`) and is held by the singleflight **leader** job only — the goroutine that actually peeks/transcodes/uploads a unique asset — so followers coalescing on the same bytes never pin a slot, and total concurrent peek/transcode/upload jobs across all feeds is capped regardless of how assets distribute. The singleflight body runs under the **run/shutdown ctx** (`a.baseCtx`), not the per-feed errgroup ctx, so one feed's cancellation can't poison a follower feed sharing an asset (only run shutdown aborts it; a follower whose own feed cancels bails via the `DoChan` caller-select). Marker-less items (`mod.HasAssetMarkers` is false) skip the fan-out; `FetchCmd.fetch` sizes the pool (cap ≥ 1). **Live fetch stats** (`progress.go`): when stderr is a terminal, `runFetch` renders an in-place status line — `fetching · feeds done/total · new articles N · failed N · assets N (N active)`, verb flipping to `saving` for the pack-write half of the cycle — via `startFetchProgress` (200 ms ticker; feed workers bump counters where `onFeed` fires, asset job counts live on `assetFetcher.active/done`, bumped by the singleflight leader). The process-wide `status` line coordinates with logging: `main` routes the default slog handler through it (`log.SetOutput(status)`), so a log record erases the line, prints, and redraws it beneath — plain `\r` + space padding, no ANSI. On a non-terminal stderr (services, pipes, tests) everything is a no-op/passthrough, so logs stay byte-clean.
- **`cmd_feeds.go`** — One feed = one URL. `AddCmd` (`srr feed add`, strict-create; flags `-t/--title`, `-u/--url` (single, required), `-g/--tag`, `-r/--recipe`, `-i/--ingest` + repeatable `-p/--pipe` (feed-level {ingest, pipe} overrides on top of the recipe — see Recipes), `-e/--expire-days N` (retention window; 0 = keep forever)), `UpdCmd` (`srr feed upd ID`, feed-level fields — incl. `-e/--expire-days`, `-i` (`""` clears) and `-p` (a lone `-p ""` clears) — plus `-u/--url` to repoint the source), `RmCmd` (`srr feed rm`), `LsCmd` (`srr feed ls`, filter by `-g/--tag`, yaml/json output), `ShowCmd` (`srr feed show ID`), `ApplyCmd` (`srr feed apply` reads feed JSON from `--file PATH` or stdin; accepts a single object or an array; whole-input atomic; create when `id` absent, full-replace update when `id` present, error if `id` references a missing feed), `EditCmd` (`srr feed edit ID` opens `$VISUAL`/`$EDITOR`/`vi` on the feed JSON; no-change is a no-op; id changes refused; tempfile preserved on parse or apply error). The per-feed fetch state (`ETag`/`LastModified`/`Watermark`/`BoundaryGUIDs`/`FetchError`, plus the `LastOK`/`FailStreak`/`LastNew` vitals) is preserved when the URL is unchanged across `feed upd -u …`, `feed apply` updates, and `feed edit` (`setFeedURL` no-ops on an unchanged URL); changing the URL resets it (a new source shares no dedup/cache history). `feedView` is the flat JSON/YAML shape `{id?, title, url, error?, tag?, recipe?, ingest?, pipe?, no_title?, expire_days?, expired?}` — `error` and `expired` are read-only (server-owned: the last `FetchError` and the cumulative expired-entry counter), reported by ls/show/edit but never applied back. Empty-string convention on `-t`/`-g`/`-r`/`-i` clears the field where allowed (title and url cannot be cleared). `normalizeFeed` validates the recipe name exists (eager error on unknown recipe), runs the feed-level pipe through `filterPipe` + `validatePipe(…, allowDefault=true)` (same token rules as a recipe pipe; `#default` allowed), and bounds `expire_days` to `0..36500` (100 years). **Subscribe-time discovery (automatic, networked):** `feed add` and `feed upd -u` (only when the URL actually changes) resolve the URL through `ingest.Resolve` (the `resolveFeedURL` package var — a test seam) when the feed's effective ingest strategy is the built-in `#feed` (`resolvesFeed` gates on `ingest.Select(feedIngest, recipe.Ingest, default.Ingest)` == `ingest.Builtin` — the feed-level ingest override participates; external-ingest feeds are stored as-is). A homepage URL is repointed to its discovered feed; an unresolvable URL is a **hard error** that rejects the add/repoint. `feed apply` / `feed edit` deliberately stay offline. So `add`/`upd -u` are no longer purely offline — they make one network probe.
- **`cmd_art.go`** — `ArtCmd` (`srr art ls`): lists stored articles newest-first with cursor pagination (`-b/--before`), filter by feed ID (`-i`) or tag (`-g`), with `-l/--limit` (default 50). Outputs JSON. `readAllIdx` walks the packs parsed by the shared `idx_read.go` reader (`loadIdxPacks` + `getPackRef`), filtering by `db.Feeds()`/`AddIdx`; `loadContent` parses JSONL data packs to fill in the per-article `ArticleData` (title, link, published, content) from the data pack.
- **`cmd_import.go`** — OPML import with hierarchical ID selection (`-a` all, `-i` specific). Each OPML `xmlUrl` becomes one single-URL feed. OPML group hierarchy auto-resolves to hierarchical tags; `-g/--tag` overrides. Cross-folder URL dedup: a `seen` set on `importWalker` emits a URL once even when it is cross-listed in several folders (the first folder visited — case-insensitive name order — wins its tag). `-r/--recipe` stamps every imported feed. `-n/--dry-run` lists resulting feeds without importing. `applyImportDefaults` centralises post-walk Tag/Recipe stamping. (No merge mode — that was removed with the Feed concept.) **Subscribe-time discovery (automatic, networked):** the shared `resolveImportBatch` pipeline (`applyImportDefaults` → `importRecipes` → up-front `validateRecipeRef`, so a typo'd `--recipe` fails before any URL is probed → `resolveImportFeeds`) is called by both `ImportCmd.Run` and serve's `handleImport`; `resolveImportFeeds` probes every `#feed` feed concurrently (bounded by `globals.Workers`; external-ingest feeds pass through untouched) via the `resolveFeedURL` seam, repointing homepage URLs to their discovered feed. Import is **partial-success**, not atomic on resolution: resolvable feeds are imported, unresolvable ones are skipped and reported (`reportImportFailures` → the test-overridable `stdout`). `--dry-run` probes too, so its preview shows resolved URLs and the skip report. `importRecipes` reads the db.gz recipes map (read-only) to gate the `#feed` decision per each feed's resolved recipe.
- **`opml.go`** — OPML XML parsing. `ParseOPMLTree` builds `OPMLNode` tree from file; each `OPMLNode.Feed` is a single-URL `Feed`. `normalizeGroupName` converts group names to tag-safe identifiers.
- **`cmd_export.go`** — `ExportCmd` (`srr feed export`): writes the current feeds as OPML 2.0, the inverse of import — hierarchical tags become nested outline groups, one leaf outline per feed URL. `buildOPML`/`outlinesOf` are reused by serve's `handleExport` (`GET /api/export`).
- **`cmd_gen.go`** — `GenCmd` (`srr gen`): prints the db.gz `gen` counter; `-b/--bump` increments and commits it via `BumpGen` (the SW cache-invalidation key after an in-place store rebuild; also resets `HdrPacks`/`MetaPacks`/`MetaTail` — see db.go).
- **`utils.go`** — reflection/print helpers for `srr config`: `toKebab`, `fieldName`, `maskSecret`, `printFields`.
- **`cmd_preview.go`** — `preview` subcommand: loads a recipe (`-r/--recipe`, default `default`) from db.gz, applies optional ad-hoc overrides (`-p` pipe steps, `-i` ingest strategy) with **feed-level override semantics** — `renderPreview` resolves `resolvePipe(resolvePipe(default.Pipe, recipe.Pipe), pipeOverride)` and `previewFetch` resolves `ingest.Select(ingestOverride, recipe.Ingest, default.Ingest)`, so `-p`/`-i` simulate exactly what a feed carrying those overrides would run (`#default` in `-p` expands to the recipe's effective pipe) — dispatches through the ingest engine, runs the module pipeline, and serves rendered articles via local HTTP server (`-a/--addr`). Opens the DB read-only.
- **`cmd_serve.go` / `serve_overview.go` / `serve_feeds.go` / `serve_recipes.go` / `serve_syndicate.go` / `serve_fetch.go` / `serve_tools.go`** — `ServeCmd` (`srr serve`): a localhost web admin GUI. Serves an embedded `webui/` bundle (`//go:embed`) at `/` and a JSON API under `/api/*`. Each handler opens its own `withDB` scope per request — read-only (no lock) for GET, `withDB(true)` (acquire→commit→release) for mutations; no persistent lock or DB handle. A Host/Origin loopback allowlist (`hostGuard`) blocks cross-origin access to the mutating API — with one carve-out: a non-loopback `Origin` is allowed when the browser-set `Sec-Fetch-Site: same-origin` fetch metadata asserts the request came from the GUI's own page, which is how a Host-rewriting proxy deployment works (cloudflared tunnel + dashboard `httpHostHeader: localhost:8088`, e.g. `srr.llera.eu` — without the carve-out every browser mutation 403'd while GETs sailed through, since browsers only attach `Origin` to non-GET requests). The Host check stays unconditional (DNS-rebinding pages are same-origin to the browser but can't present a loopback Host). Binds `localhost:8088` by default (`--addr`/`SRR_SERVE_ADDR`). Optional `--interval`/`SRR_SERVE_INTERVAL` (default 0 = off) runs the same all-feeds fetch loop in-process — a background goroutine driving `FetchCmd.fetchLoop` under the server's signal context, so one Ctrl-C/SIGTERM stops both; a running cycle holds the store lock for its duration, so a concurrent GUI mutation gets the same 409 contract as a separate fetch process. Reuses CLI logic via shared helpers: `saveFeed` (feed upsert + subscribe-time discovery), `setRecipe`/`removeRecipe` (also back `srr recipe`), `setOutFeed`/`removeOutFeed` (also back `srr syndicate`), `renderPreview` (also backs `srr preview`), and `FetchCmd.runFetch(ctx, client, onFeed func(feedProgress))` (also backs `srr art fetch`; the unexported `FetchCmd.only []int` field restricts a cycle to those feed ids — set by the SSE `/api/fetch?id=N` handler for the GUI's per-row fetch, empty = every feed, unknown id fails the cycle in-band; the SSE stream carries per-feed progress). Lock contention → 409 (`os.ErrExist`); a triggered fetch holds the lock for its duration like the cron loop. `/api/inspect` captures the existing `srr inspect` report. **`GET /api/overview`** (`serve_overview.go`) is the single-read store snapshot every read-tab renders from: feeds projected to the UI list shape (map-key id + friendly `error`, plus writable `expire_days`/`dedup_days`/`dedup_title` and read-only server-owned `expired`, still omitting the per-feed cache/dedup internals bg/etag/wm), the derived tag buckets (live article counts: `TotalArt − Expired` — the overview is a display projection), `db.core.Recipes` directly (no defensive copy — the map is consumed read-only for JSON marshaling), the syndication `out` list, `gen`/`fetched_at`/`total_art`, the effective store-wide `dedup_days` default (the Tools tab renders/edits it), `cdn_url` (from `globals.CdnURL`, omitted when unset — lets the syndicate tab link the live `out/<name>` files), and `version` (the running binary's version — the webui header labels itself with it, `#ver` next to the wordmark) — one db.gz read per request. (All projection logic is inlined into `getOverview`; the granular `GET /api/feeds`/`/api/tags`/`/api/recipes`/`/api/syndicate`/`/api/gen` read endpoints were removed — the webui only ever called `/api/overview`.) Full API surface (`registerAPI`): `GET /api/overview`, `POST /api/feeds`, `POST /api/feeds/apply`, `PUT|DELETE /api/feeds/{id}`, `PUT|DELETE /api/recipes/{name}`, `PUT|DELETE /api/syndicate/{name}`, `GET /api/preview`, `GET /api/resolve` (read-only add-feed probe: ingest-fetch the URL through a recipe → `{url, title, items}` — the discovery-resolved URL, the feed's own channel-level title, item count; optional `&ingest=` carries the feed-level ingest override; advisory, save re-resolves), `POST /api/gen/bump`, `PUT /api/dedup` (store-wide default seen.gz dedup horizon — the Tools tab's Dedup pool panel, mirroring `srr dedup --days N`; days in [0, 36500], 0 resets to the built-in default, echoes the effective value), `GET /api/export`, `POST /api/import` (import + export live in the Tools tab's OPML panel; the webui only ever calls the `dry_run=1` half: every OPML feed lands in a review sheet — per-row include checkbox, editable title/tag/recipe — with already-subscribed URLs pre-unchecked and dry-run resolution failures kept as pre-unchecked rows too, error inline, rather than dropped: the probe ran the default recipe's ingest, so an "unresolvable" URL may just need a recipe with an external ingest. The kept rows commit via `POST /api/feeds/apply`, atomic and offline — resolved URLs were folded by the dry run, unresolved ones store as-is. The non-dry import-all commit path stays for direct API users), `GET /api/inspect`, `/api/fetch` (SSE; optional `?id=N` restricts the cycle to one feed — malformed id → 400 pre-stream, unknown id → in-band error event). Webui (`webui/app.js`): the Feeds tab is a triage-first health console — a fifth `stale` grade (fetching fine but no new article in 30 days, `STALE_AFTER`) joins ok/warn/err/idle, board stats are grade-filter toggles, a store-pulse strip alarms when the store-wide `fetched_at` ages past 6h/24h (`storePulse`; carries an inline fetch-all), failing rows show their fetch error inline under the title (`.rowerr`, plus the dot tooltip), title/last-new/articles headers sort (`FEED_SORTS`, numeric columns first-click descending), rows stack to two lines under 720px (CSS-only), per-row actions fetch one feed / preview it through its recipe in an in-place dialog / edit; the add-feed dialog is URL-first — paste/Enter/blur probes `GET /api/resolve` (a homepage folds to its discovered feed URL, the feed's own title fills an empty title box, a wire-voice status line reports items · title or the verbatim error) and tags/recipes pick via one-click toggle chips; everything beyond that primary flow — the feed-level "Ingest"/"Pipe" overrides (shared `stepsEditor` with the recipe modal), the "Expire after" days field (`min 0` / `max 36500`, empty = keep forever), the per-feed dedup-pool overrides (dedup-days `min -1`: empty/0 inherits the store default, -1 disables the pool for this feed, N sets its horizon; plus an "Also dedup by title" checkbox) and the hide-titles flag — folds behind an "Advanced" `<details>` that auto-opens when any of it is non-default and, while closed, wears one summary chip per value set (`ingest` / `pipe` / `expire Nd` / `dedup Nd` or `dedup off` / `dedup title` / `no titles`), so saved config never hides silently (the table's recipe cell and the preview dialog mark override-carrying feeds with an "override" chip, and the row preview passes `pipe`/`ingest` to `/api/preview` so it matches a real fetch) — feed writes are full-replace like `feed apply`, so the modal always sends the no_title, expire_days, dedup_days, dedup_title, ingest, and pipe values (contract pinned by `TestServeFeedSaveOmittedExpireDaysZeroes`, `TestServeFeedSaveRoundTripsIngestPipe`, and `TestServeFeedSaveRoundTripsDedup`), and every SSE `feed` event ticks the board+table live via `applyFeedEvent` (optimistic vitals mirror keyed on the event's `id`, reconciled by the post-stream refresh; Tools' handler only mutates the snapshot); the snapshot re-pulls after every fetch stream and on window focus when >30s stale (Tools is never redrawn in place — that would wipe its streamed logs); the current tab rides `location.hash` (boot + `hashchange`); the syndicate modal picks tags/feeds via snapshot-fed checkbox lists; a running global fetch is cancellable (AbortController → request-context cancellation).
- **`cmd_recipe.go`** — `RecipeGroup` (`srr recipe`): sub-commands `ls` (print all recipes as JSON/YAML), `show <name>` (print one; errors if missing), `set <name> [-i ingest] [-p step]…` (add or replace — full-replace semantics; each axis falls back to `default` at resolution time, not at write time; `#default` is validated and forbidden in `'default'` itself via `validatePipe(allowDefault = name != defaultRecipeName)`), `rm <name>` (refuses `'default'` and any name still referenced by a feed). Also hosts `filterPipe` (drops empty strings; returns nil for all-empty input as the CLI "clear" sentinel) and `validatePipe(steps, allowDefault bool)` (rejects `#default` when `allowDefault` is false; validates built-in names; unknown `#`-tokens error).
- **`cmd_syndicate.go`** — `SyndicateGroup` (`srr syndicate`): manages the db.gz `out[]` list. Sub-commands: `ls` (print current Out as JSON), `set <name> -f rss|json [-t title] [-g tags] [-i feed-ids] [-l limit]` (upsert by name; validates format, requires ≥1 selector, checks feed ids exist, defaults limit to `outDefaultLimit`=50), `rm <name>` (remove entry + best-effort delete `out/<name>.<ext>` from store). Also hosts `outFileKey`, `outTitle`, `joinURL` helpers used by `db_out.go`.
- **`db_out.go`** — `SyncOutFeeds(ctx)`: writes `out/<name>.rss` (RSS 2.0) or `out/<name>.json` (JSON Feed 1.1) for each `OutFeed` entry. No-op when `Out` is empty or `SRR_CDN_URL` is unset (warn-only skip). For each output: builds feed-id set from Tags∪FeedIDs union; walks a tail window (limit×10, widened to full store if needed) via `walkArticles`, skipping expired articles (`chron < the feed's AddIdx`); rewrites relative asset refs to absolute CDN URLs via `rewriteAssetURLs` (a dedicated HTML walk over `outAssetAttrs` — `RewriteAttrs` in `mod/` only fires on `#`-prefixed markers, not arbitrary relative refs, so a separate rewriter is used here); serialises via `marshalRSS`/`marshalJSONFeed`; writes via `Backend.Put(…, true)` (overwrite). Warn-logs each per-feed failure, then returns a non-nil error (`"%d of %d syndication output(s) failed"`) when any output failed — `cmd_fetch` relies on that to leave `lastOutSig` unadvanced and retry next cycle; a clean run returns nil. Still warn-only at the call site — the durable article batch is never at risk. The `IOutFeedWire` TS type is generated by `cmd_gents.go` and appears in `IDBWire.out?`, but **the frontend/service-worker ignores the `out` field** — it is backend-only config and output. `outFeedsSig` is the idle-cycle rewrite gate's signature (out config + per-feed tag AND `AddIdx`): an expiration advance alone changes it, so an otherwise-idle cycle still rewrites `out/*` without the expired items. Also hosts the shared HTML-walk helpers `parseBodyFragment`/`visitAssetAttrs` (factored from `rewriteAssetURLs`; `db_expire.go`'s `collectAssetRefs` reuses them so the harvested attr set can't drift from the rewriter's).
- **`db_expire.go`** — `ExpireArticles(ctx, now)`: applies each feed's `ExpireDays` retention policy (see root `CLAUDE.md` → Feeds `exp`/`xp`). Expiration clock is `fetched_at` — globally chron-monotone (each batch stamped with its cycle timestamp), so the expired set per feed is always a contiguous prefix, exactly what `AddIdx` models. One `walkArticles(min(AddIdx) → TotalArticles)` pass early-stopped by the `errExpireDone` sentinel at the first article with `fetched_at ≥ max(cutoff)`; an article expires iff `fetched_at < now − ExpireDays·86400` (strictly older). For each expired article the `assets/…` keys its content references are harvested by `collectAssetRefs` — `parseBodyFragment` + `visitAssetAttrs` (shared with `db_out.go`'s `rewriteAssetURLs`, so the attr set can't drift) with every candidate validated against the strict `assetKeyRe` grammar, not a bare prefix: the keys feed `Rm`, which path-joins on local/SFTP, so adversarial content like `assets/../victim` is never harvested (leak-safe, never delete-unsafe); unparseable HTML contributes nothing (it already published as-is — an error here would wedge retention forever). All-or-nothing: any walk, `Stat`, or `Rm` failure returns before ANY `AddIdx`/`Expired`/`AssetBytes` change is applied, so the next cycle recomputes the same window and retries idempotently (`Rm` is silent on missing). **Asset-bytes accounting**: every harvested key is `Stat`-measured BEFORE the delete loop (a stat failure aborts with nothing deleted — clean retry; each phase fans its independent per-key store round-trips out over a bounded errgroup — `max(1, globals.Workers)` — with the Wait between them preserving the no-Rm-before-every-Stat boundary), and each deleted object's size reduces the AssetBytes of the expiring feed that first referenced it (`assetOwner`, chron-order deterministic), clamped at 0 (a cross-feed shared asset may have been charged to the uploader but expired via another feed's article; a mid-delete Rm failure loses the decrement for keys it did delete — the retry stats them as 0 — accepted skew, same class as the no-liveness-check trade-off). On success: `AddIdx` bumps past the expired prefix (logical deletion — packs/idx headers are immutable), `Feed.Expired` accumulates the count (the reader's header-count correction, see the data contract), and one INFO summary logs articles/assets/asset_bytes/feeds plus an `advanced` counter for the dormant-feed frontier advance: the walk records `stopChron` (the early-stop chron, else `TotalArticles`) and per-feed `sawLive`, and an expiring feed that saw no live own entry AND expired nothing this cycle jumps `AddIdx` straight to `stopChron` (the advanced-over region provably holds zero own entries — `xp` unchanged, the live-count cross-check holds; a feed that expired its tail waits one cycle before jumping), keeping the steady-state walk bounded by ingestion since the last cycle instead of growing with a dormant feed's dead time. Deliberately NO liveness check (accepted trade-off): an asset shared with a live article dies too — the reader collapses the broken media; `srr asset heal --create` (or a content-hash re-upload) is the repair path. The gated fixture generator `gen_expire_test.go` (`TestGenExpireStore`, `SRR_GENEXPIRE_OUT=<dir>`; the `genbig_test.go` precedent — production write path with injectable timestamps) builds the aged store consumed by `frontend/e2e/contract/expire.e2e.test.ts`, the writer↔reader expiration contract suite.

monotonic — expiration is logical deletion) and AssetBytes `ab` (live assets/ store footprint: bumped by the stored payload size at the actual upload — `uploadAssets` accumulates the sizes `UploadCacheRef` reports for the Puts it led, so dedup hits and singleflight followers add nothing — and reduced, clamped at 0, by expiration's asset deletes), TotalArt, AddIdx (the old per-URL `Feed` type was folded in — there is no longer a `Feeds []*Feed`). `Feed.Fetch` resolves the recipe and `default` recipe from `run.recipes` via `recipeFor`, resolves effective pipeline via `resolvePipe(resolvePipe(default.Pipe, recipe.Pipe), feed.Pipe)` and effective ingest via `ingest.Select(feed.Ingest, recipe.Ingest, default.Ingest)` (feed-level overrides win per axis), validates the pipeline, then calls `Feed.fetchURL` for the single URL, setting/clearing `ch.FetchError`. `fetchURL` dispatches I/O+parse through the shared `*ingest.Fetcher` engine, then applies the dedup/watermark/pipeline path on `ingest.Result.Items`. Dedup model: `Watermark` is the max published unix-second ever seen and `BoundaryGUIDs` remembers as much of the current fetch's GUID window as the cap allows — so a feed that re-dates its whole window to ~now on every rebuild (fresh pubDates, stable GUIDs — e.g. blog.ofertitas.es) still dedups: the watermark can't skip re-dated items (they always sit above it), so the GUID memory has to cover them. Repopulated each non-empty fetch from the current response (no carry-over) and hard-capped at `maxBoundaryGUIDs` (1024) so one misbehaving feed can't bloat the `seen.gz` sidecar (the cap's rationale before the ping/pong relocation was db.gz; `bg` now lives in `seen.gz` — see the `db.go` bullet below and `backend/SEEN-POOL-PLAN.md`). Two classes: **must** (dateless, or `pub >= maxPub` — the watermark will NOT skip these next fetch, so each must live in `bg` or be skipped from ingestion) and **opt** (`pub < maxPub` — watermark-protected in the stable case, remembered most-recent-first only to survive re-dating, and never skipped from ingestion). The `must` class is capped first, retaining already-stored (priorBoundary) GUIDs before the smallest-hash rest, so over-cap `must` items are skipped entirely rather than ingested-but-forgotten (which would duplicate them on every subsequent fetch); `opt` GUIDs fill any remaining cap slots. A `NotModified` result or a non-modified response with zero items preserves prior `Watermark`/`BoundaryGUIDs` so the dedup state survives a transient empty feed. An item is new iff its GUID isn't in the prior fetch's `BoundaryGUIDs` AND (`pub == 0` OR `pub >= Watermark`). Item `pub` is clamped to `fetchedAt` so a publisher CMS bug that ships a far-future date can't poison `Watermark`. Within-fetch dedup uses a per-GUID set checked first. **Stale-response guard**: a 200 whose newest dated item sits strictly below `Watermark` and that carries no dateless items is a stale cache copy (flappy CDN serving an older feed generation — observed on YouTube's feed endpoint); it is ignored wholesale like `NotModified` (WARN log; `Watermark`/`BoundaryGUIDs`/ETag/LastModified all preserved) — every dated item in it would be watermark-skipped anyway, but rebuilding `bg` from it evicts the watermark GUIDs and the fresh copy one cycle later would re-ingest them as duplicates. Remaining trade-off: a response that keeps one watermark sibling while dropping another (guard can't fire — its max pub equals `Watermark`), or dateless items that disappear for one fetch and reappear, are still re-ingested as duplicates (snapshot semantics over carry-over). Also hosts the `resolvePipe` helper, the `#default` token constant (`pipeDefault`), the `maxBoundaryGUIDs` const, the `fetchRun` struct (run-scoped deps including `recipes map[string]Recipe`), and `uint32Set`. (History: this file was `channel.go` until the Channel→Feed rename; the *old* per-URL `feed.go` was deleted when its `Feed` type was folded in — everything lives here now.)

### Ingest (`ingest/`)

An "ingest strategy" is the I/O+parse layer that produces `mod.RawItem`s from a URL. The default zero-config behaviour is the historical RSS/Atom/RDF path (`#feed`); alternative strategies — shipped out-of-repo as external shell commands — cover URLs that aren't feeds. Selection per feed: `ingest.Select(feed.Ingest, recipe.Ingest, default.Ingest)` (variadic, first non-empty wins) — the feed-level override, falling back to the feed's resolved recipe's ingest, falling back to the `default` recipe's ingest, falling back to `#feed`. Empty strings fall through; persistence uses `omitempty`.

Built-in names start with `#` and resolve through the package's `Register` registry (init-time). Anything not starting with `#` is treated as a shell command per the external-ingest protocol. Only `#feed` ships built-in; source-specific strategies live out-of-repo as external commands.

Mirrors `mod/`'s architecture: a `Fetcher` struct (built by `New()`) holds the registered built-in `FetchFunc`s in a name→func map; `Fetcher.Fetch(ctx, args, ...)` dispatches by name — built-ins for matching keys, shell-command fallthrough for anything else.

- **`main.go`** — `Request` / `Result` structs, `FetchFunc` type (`func(ctx, *http.Client, buf, Request) (Result, error)`), direct registration `Register(name, FetchFunc)` (built-ins are stateless, so `New()` shares the registry rather than instantiating one per call), `Select(feed, global) string` precedence helper (final `#feed` fallback, exported as the `Builtin` const — the token callers compare against to gate subscribe-time discovery), the `Fetcher` struct + `New() *Fetcher`, and `Fetcher.Fetch(ctx, args, client, buf, req)` which routes `args` to a built-in if registered or executes it as a shell command otherwise. External wire schema is the JSON encoding of `Request` (stdin) and `Result` (stdout) — see struct tags; stderr is passthrough. A non-zero exit code or empty stdout is a hard error (fails just that feed's fetch); stdout is capped at 64 MiB. Items on the wire are `mod.RawItem` records, so external strategies emit `guid` as the FNV-32a hash (uint32) of any stable per-item string, matching the dedup contract used by built-ins. `published` is RFC3339 (null/absent for dateless). The author-facing protocol spec + a reference implementation live in `backend/README.md` (Ingest Strategies); the protocol tests are in `ingest/external_test.go`.

**External file self-hosting.** The ingest engine owns no asset machinery: it just sets the external command's working directory to `Request.AssetDir` (`cmd.Dir`) so the command stashes/references files with relative paths. The caller (`main`) owns the rest. `FetchCmd.fetch` creates **one cache dir shared by every feed this run** = `globals.CacheDir` (`MkdirAll`, kept across runs; **always set** — the kong `${cacheDir}` default covers unset, main()'s post-parse floor covers an explicitly empty `cache-dir:`, both resolving `defaultCacheDir()` = `os.UserCacheDir()/srr`), stores it on `fetchRun.cacheDir` (creation is **mandatory** — never empty, since a command handed an empty working dir would run in SRR's cwd), and `Feed.fetchURL` hands it to **every** feed as both `Request.AssetDir` and the upload-step `cacheDir` — no strategy gate, so a built-in fetcher may download into it too. Whichever fetcher ran owns the file layout inside the shared dir. The upload itself is the **automatic end-of-pipeline step** `fetchRun.uploadAssets` (run from `Feed.fetchURL` after `processItem`, which stays a pure, store-free transform; it fans marker-bearing items out concurrently, the peek/transcode/upload work bounded by the `assetFetcher`'s run-global `SRR_ASSET_WORKERS` pool (held by the singleflight leader job)): for each item it calls `mod.RewriteAttrs(content, fn)` (the generic walk over `assetAttrs` — img/video/audio `src`, video `poster`, + `a href`), and `fn` scans for **`#`-markers** — a value starting with `#` whose remainder names an existing regular file in `cacheDir` (e.g. `#/photo.jpg`) — uploading each through `assetFetcher.UploadCacheRef` (content-hash key, upload-if-absent) and rewriting the marker to the returned key. `UploadCacheRef` classifies its guard failures with the `errNotAsset` sentinel — references naming no regular file (an ordinary `#fragment` anchor) or resolving outside the cache dir (attacker-influenced content) are **declined** by `fn` (the marker stays untouched, one containment implementation lives in `UploadCacheRef` alone); any other upload failure (store error) returns the error from `fn`, which `RewriteAttrs` surfaces so `Feed.fetchURL` **hard-fails the whole feed** — feed state (watermark/dedup/etag) is left untouched, so a transient failure self-heals next fetch. (Oversize is no longer one of these: the `--max-asset-size` cap is enforced **at download** by `#selfhost`/ingest, so an over-cap asset is dropped there — keeping its remote URL — and never reaches the upload step.) `<video poster>` is **not** a usable target *for ingest-injected markers*: `#sanitize`'s poster regex (`^(https?://|assets/)`) strips a `#`-poster before the upload step runs (an ingest strategy's markers ride the pipeline from the start, so they pass through `#sanitize`). The built-in `#selfhost` mod, by contrast, runs *after* `#sanitize` (placed after `#default`) and **does** self-host posters — its markers are created post-sanitize and never re-sanitized. Download dedup is the command's (cache check); upload dedup is SRR's (content hash + store existence check). The cache is **self-maintaining**: after each cycle's Commit, `sweepAssetCache` (cmd_fetch.go) deletes files unused for longer than `--cache-max-age`/`SRR_CACHE_MAX_AGE` (default 72h; 0 disables). Relevance = **mtime**: `UploadCacheRef` touches every source file it consumes (before the memo/store-hit/upload branching, so all paths count), and the cache consumers also touch on reuse (`srr-telegram`'s size-validated cache check, `#selfhost`'s URL cache), so any file a live feed still references stays fresh — including a big backlog warming up across failing cycles — while orphans (dropped items' media, interrupted-run debris, consumed downloads nothing re-references) age out. A `FetchFunc` owns only I/O and parsing — dedup, watermarking, pipeline modules, and storage all stay in `Feed.fetchURL`/`processItem` and operate uniformly on `Result.Items`. Built-in `FetchFunc`s and the shell-command path must be concurrent-safe (one `*Fetcher` is shared across workers; unlike `*mod.Module` it isn't pooled).
- **`feed.go`** — Built-in `#feed` strategy (registered inline in `init`): the zero-config default. HTTP GET with `If-None-Match` / `If-Modified-Since` hints into the shared per-worker buffer, then `ParseFeed`. Returns `NotModified` on 304; populates `ETag` / `LastModified` for the next request. **Feed auto-discovery**: when `ParseFeed` reports `errNotFeed` (the document is not a recognized feed — an HTML page, unknown XML root, or non-XML — *as opposed to* a genuine fault parsing a real feed) and the response looks like HTML (`text/html` Content-Type or body starting with `<!doctype html`/`<html`), `discoverFeedLink` (`feed_discover.go`) scans for `<link rel="alternate" type="application/rss+xml|atom+xml|…">` in the HTML head, fetches the discovered URL, and sets `Result.ResolvedURL`; `feed.go` persists `ch.URL = ResolvedURL` without resetting dedup/etag/vitals. One-hop guard: a discovered page that is also HTML-without-feed returns the original error and does not recurse. **`errNotFeed`** is the unexported sentinel that classifies "not a feed" so the discovery branch (here, fetch-time) never fires on a broken-but-real feed. **`Resolve(ctx, client, rawURL, maxSize)`** is the subscribe-time wrapper: it runs the same `feedFetch` discovery once and returns the canonical feed URL (rawURL if it already serves a feed, the discovered target if it's an advertising HTML page, or an error if neither) — used by `feed add`/`upd -u`/`import` (see `cmd_feeds.go`/`cmd_import.go`). Also houses the streaming RSS/Atom/RDF parser: `ParseFeed(data, fn) (title, err)` (returns the feed's own channel/feed-level title — parent-checked so an `<item>`/`<image>` title never masquerades as it; surfaced as `Result.Title`, consumed by serve's `GET /api/resolve`; a non-feed document is reported wrapped in `errNotFeed`; a fault parsing a recognized feed is a plain error) and the package-private `hash(string)` used for GUID stability (the FNV-32a contract external fetchers replicate).
- **`feed_discover.go`** — `discoverFeedLink(htmlData, baseURL)`: scans HTML for the first `<link rel="alternate">` with a recognised feed MIME type using `golang.org/x/net/html.NewTokenizer`; returns the href resolved against baseURL. Case-insensitive rel (including multi-valued), tolerant of malformed/partial HTML.

### Store (`store/`)

Low-level storage interface: `Get`/`Put`/`AtomicPut`/`Rm`/`Close`. Registry selects by URL scheme; local = empty scheme `""`. Config registry: backends call `RegisterConfig` in `init()` with a config struct pointer; `LoadConfigs` reads YAML sections into them. `main.go` also owns the write-once pack-name grammar: the `PackSeries` series→kinds table (the contract's single source — emitted to the frontend as `PACK_SERIES_KINDS` by `srr gen-ts`) builds `packKeyRe`, which `cacheControlForKey` uses to stamp immutable Cache-Control on pack keys (strict per-series: `data/h3.gz` is not a pack name). `cacheControlForKey` also classifies the self-hosted frontend shell at the store root (`srr frontend update`): `index.html`/`manifest.webmanifest`/`sitemap.txt` revalidate, content-hashed root assets (`feHashedRe` = `<name>.<8+ hex>.<ext>`, slash-free so it never matches a pack key) are immutable. Its sibling `contentTypeForKey` stamps the meta-unset Content-Type the same grammar-driven way: db.gz + `packKeyRe` matches → `application/gzip` (honest type for SRR's own gzip objects; deliberately NO `Content-Encoding: gzip` — the reader gunzips manually via `DecompressionStream`, so a transparently-decompressing CDN/browser would hand it pre-inflated bytes and break it), everything else `application/octet-stream`. S3 + HTTP backends only (local/SFTP ignore Cache-Control/Content-Type, deferring to the static server's mime-by-extension).

| Method | Signature | Behavior |
|---|---|---|
| `Get(ignoreMissing)` | returns `io.ReadCloser` | Controls error-on-missing; streaming read |
| `Stat` | returns `int64` | Object size without reading the body (os/sftp stat, S3 HeadObject — its missing-key code is the bodyless `NotFound`, not `NoSuchKey` — HTTP HEAD); missing key = `(0, nil)`, silent like Rm — expiration's AssetBytes accounting treats absent as zero (a retried expire cycle re-stats keys an aborted predecessor already deleted), and the upload dedup probe (`resolveAndUpload`) trusts only `size > 0` as a hit, falling back to the `Get` probe on 0 since a HEAD can't tell absence from a zero-byte object or a missing Content-Length |
| `Put(ignoreExisting)` | accepts `io.Reader` | Controls overwrite vs exclusive create; streaming write |
| `AtomicPut` | accepts `io.Reader` | temp-then-rename (local/SFTP); overwrite (S3); streaming write |
| `Rm` | — | Silent on missing (local/SFTP warn + return nil; S3 unconditional delete) |

- **`local.go`** — Auto-creates subdirs via `os.MkdirAll`.
- **`s3.go`** — `IfNoneMatch` precondition headers + CRC32 checksums. `S3Config`: region, endpoint, profile, static credentials.
- **`sftp.go`** — Auto-creates subdirs via `client.MkdirAll`. Auth chain: URL password → config password → config/default private key → `~/.ssh/` keys → SSH agent → error. Host key verification via `~/.ssh/known_hosts` by default (`SFTPConfig.Insecure` to skip).
- **`http.go`** — Plain HTTP object store (WebDAV-style / S3-compatible endpoint), registered for both `http` and `https` schemes (one shared `http` config section): `Get`=GET (404/410 → missing), `Put`/`AtomicPut`=PUT (AtomicPut is a plain overwrite like S3; metadata rides request headers — Content-Type from meta, then `contentTypeForKey` (db.gz + pack names → `application/gzip`), then the `application/octet-stream` default; Content-Encoding only when meta sets it; Cache-Control from `cacheControlForKey` — servers that don't store metadata serve their own headers like local/SFTP), `Rm`=DELETE (404/410 → nil), `Stat`=HEAD (size from Content-Length, -1/absent counts as 0; 404/410 → `(0, nil)`). Exclusive create maps to `If-None-Match: *` with 412 wrapped in `os.ErrExist` (the lock/409 contract); a server that ignores conditionals makes `.locked` best-effort. Auth: URL userinfo (net/http basic auth), `HTTPConfig.Token` (bearer), or `HTTPConfig.Headers` (extra headers on every request, e.g. Cloudflare Access service tokens; an explicit `Authorization` entry wins over Token; both secret-tagged — the map masks whole via `maskSecret`'s map case. `SRR_HTTP_HEADERS` env-overrides it through `loadEnv`'s `map[string]string` case (`parseEnvMap`: comma-separated `Name: value` entries, split on the first colon, trimmed; replaces the YAML map whole)); `Insecure` skips TLS verification. **Redirect guard**: `CheckRedirect` follows redirects only when the original request was a GET or HEAD (`Stat`) — net/http otherwise silently replays a 301/302/303-redirected PUT/DELETE as a bodiless GET and reports the GET's status, turning a write that never happened into a false success (found by conformance audit; pinned by `TestHTTPWriteRedirectFailsLoudly`). No client timeout by design (uploads may run long; every op carries the caller's ctx); URLs log via `Redacted()` so userinfo passwords never leak. One shared `http` config section for both schemes (no `SRR_HTTPS_*` names, deliberate).

### Pack Storage (`db.go` + `db_pack.go` + `db_meta.go`)

See root `CLAUDE.md` Data Contract for pack format spec (idx/, data/, meta/ series), db.gz schema, CDN layout, and file-based locking.

Split across four files by concern:
- **`db.go`** — `DB`, `DBCore`, `Recipe`, `OutFeed`, lifecycle (`NewDB`/`Close`/`Commit`), feed CRUD (`AddFeed`/`RemoveFeed`/`FeedByID`/`Feeds`), `withDB`/`withDBCtx` boilerplate wrappers, and shared utilities (`gunzip`, `readGz`, `jsonEncode`). `Commit` serializes `DBCore` via `AtomicPut`. `BumpGen` owns the bump-implies-reset invariant (Gen++ zeroes `HdrPacks`/`MetaPacks`/`MetaTail`; `srr gen --bump` is a thin caller). `DBCore.Recipes map[string]Recipe` is the named-recipe map (backend-only config; `omitempty`); `DBCore.Out []OutFeed` is the syndication config (default nil → no-op; `omitempty`). `DBCore.SeenFlag` (`sf`) names the active `seen.gz` ping/pong slot (`seen.0.gz`/`seen.1.gz`, `seenSlotKey`): `SyncSeen` flips it in memory after writing the inactive slot, and `Commit` publishes the flip — the same write-once "db.gz names the current generation" contract as `Seq`/`HdrPacks`/`MetaPacks`. `defaultRecipeName = "default"` is the reserved fallback; `recipeFor(recipes, name) Recipe` resolves an empty or unknown name to `default`. `NewDB` seeds the `default` recipe (`["#sanitize","#minify"]`) when absent, then loads the seen sidecar (`db.seen = db.loadSeen(ctx)`) and hydrates each feed's `ETag`/`LastModified`/`BoundaryGUIDs` from it (`db.seen.hydrateFeeds`) — no db.gz migration: a pre-relocation store's inline `bg` is simply ignored (`json:"-"` skips it) and the sidecar refills it on the next fetch. Also holds the format atom constants including `searchGram`/`searchBloomBytes`/`searchBloomK` (exported to TS as `SEARCH_GRAM`/`SEARCH_BLOOM_BYTES`/`SEARCH_BLOOM_K` by `cmd_gents.go`) and `metaPackSize` (exported as `META_PACK_SIZE`).
- **`db_pack.go`** — Binary idx + JSONL data pack writer. Contains `ArticleData` and `Item` types (plus `Item.articleData`, the one Item→ArticleData mapping), the `pack` struct (`newPack`/`writeIdx` — a 2-byte entry `feed_id:u16 LE` — /`writeIdxFooter`+`parseIdxFooter` — the data-pack boundary footer, u16 LE local indices where the packId advances; `parseIdxFooter` recovers it on the append path — /`writeIdxHeader` — the variable-length header: 2 state u32s (`packId_base`, `packOff_base`) + a `numSlots` u32 prefix, then `numSlots` cumulative-count u32s sized to the high-water feed_id present /`writeArticle`), `loadPack` (returns the decompressed bytes; `packFromBytes` wraps them into an appendable pack, guarding the empty case so a fresh store keeps `Len()==0`), `savePack`/`savePackFinal`/`checkLatestIdx` (`checkLatestIdx` verifies the latest idx pack's entry count against db.gz and returns the entries-end offset the append path splits the footer at; `savePackFinal` publishes finalized — immutable, downloaded-forever — names recompressed by `gzipBest`: zopfli-grade deflate via `foobaz/go-zopfli`, still plain gzip to readers, measured −4% data / −11% idx; the candidate must round-trip byte-for-byte through the stdlib gzip reader or the save hard-fails — no silent fallback, a finalized name is cached forever so corrupt bytes must never publish. Latest packs, summaries, and db.gz keep fast stdlib gzip. `finalGzip` is the test seam — `setupTestDB` stubs it to identity so 50k-boundary tests skip the CPU), and the `PutArticles` driver (returns the `[]ArticleData` it persisted, in pack order — `SyncMeta` consumes that slice so derived meta cards can't drift from the packs). Also `dataKeyFor` and `parseDataPack` (used by `cmd_art.go` and `cmd_inspect.go` for read-side access), and the idx header-summary machinery: `summaryKey`/`numFinalizedIdx`/`idxKeyAndSize`/`readPackHeader` (fixed-size header reader, used by meta's bloom header)/`readIdxHeader` (variable idx header: reads the prefix, then `numSlots`×4 count bytes), `saveSummary` (publishes a count-named concatenation of finalized-pack headers; takes a per-pack header-bytes callback so idx passes its variable-header reader and meta passes its fixed bloom reader), `SyncIdxSummary` (rebuilds `idx/h<N>.gz` from the finalized packs' headers whenever `HdrPacks` lags — covers normal finalization, pre-summary store migration, and post-`gen --bump` resets; sets `HdrPacks` only after the save succeeds), and `GCSummaries` (sweeps superseded `h<g>` names, same grace window as `GCLatest`).
- **`db_meta.go`** — Meta pack writer and folding contract. `MetaEntry` type (`f`/`w`/`t` JSONL wire — the struct tags own the format; every encode and decode goes through it) with its buffer-level decoder `parseMetaEntries` (used by `srr inspect`'s `checkMeta` and the tests). `foldSearchText` (NFD → strip Mn → lowercase → ς→σ → non-letter/number = word break, space-joined; mirrored by `search.ts fold()`). `bloomBits`/`bloomAdd`/`bloomHas` (FNV-1a-64 double-hash, 7 probes, mask `& (2^15-1)` matching `searchBloomBytes`=4096). `walkArticles` (streams `[from, to)` chron range from idx+data packs). `SyncMeta(ctx, written)` (called by `cmd_fetch` between `SyncIdxSummary` and `Commit` with the `[]ArticleData` slice `PutArticles` returned, warn-only): reconciles `meta/` with the store — reads back `meta/L<Seq-1>` (trusted iff entry count == `MetaTail`), fills the missing chron range from `written` when it covers the gap exactly (the common cycle) else via `walkArticles`, finalizes bloom-headed shards at `metaPackSize`=5000 boundaries (`saveMetaShard`), writes `meta/L<Seq>`, rebuilds the bloom summary from streaming `readPackHeader` reads when `MetaPacks` lags `numFinalized`; sets `MetaPacks`/`MetaTail` only after every save succeeds. The meta/ series feeds BOTH the home list (with a data/ fallback when `metaReady()` is false, so warn-only is safe even though the list depends on meta) and search. `finalizedMetaKey`/`metaSummaryKey`. `GCMetaSummaries` (sweeps superseded `s<g>` names, same grace window as `GCSummaries`). `golang.org/x/text` (unicode/norm) is a direct dependency.
- **`idx_read.go`** — the single Go read-side mirror of the binary idx format: `keyGetter` (fetch+gunzip a store key, local or HTTP), `idxPack`/`idxBound`, `parseIdxPack` (byte-for-byte mirror of frontend `idx.ts` `makeIdxPack().parse()`), `getPackRef` (mirrors `data.ts`), `packIdxFor`, `loadIdxPacks`. Every read-side command (`srr inspect`, `srr art ls`) goes through this one parser; the only other implementations of the format are the writer (`db_pack.go`) and the frontend reader.

Backend-specific behavior:
- `PutArticles` and `savePack` manage the idx and data series. `SyncMeta` manages the meta/ series. Order: `PutArticles` → `SyncIdxSummary` → `SyncMeta` → `ExpireArticles` → `SyncOutFeeds` → `Commit` → GC sweeps.
- `PutArticles` writes binary idx and JSONL data directly; splits idx packs at every 50,000 articles. It also charges each article's uncompressed JSONL line length to its feed's `ContentBytes` (writeArticle returns the byte count). It tracks the data-pack boundaries (where `NextPackID` advances) into a per-pack slice and emits them as the idx footer at each finalize and at the latest-pack save; on the append path it strips the loaded latest pack's old footer (keeping header+entries) and recovers its boundaries via `parseIdxFooter` before extending.
- `ArticleData` struct: `{ FeedID, FetchedAt, Published, Title, Link, Content }` — serialized as JSONL with short keys `f/a/p/t/l/c`.
- `DBCore.Feeds` is `map[int]*Feed`; serialized as a JSON object keyed by feed ID. `AddFeed` assigns the first free ID in `[0, feedIDCeiling)` (`feedIDCeiling` = 65536, since `feed_id` is a u16 in each idx entry) and returns an error if all slots are taken; it sets `AddIdx = TotalArticles` and explicitly zeroes `Expired` (a fresh incarnation has expired nothing — a reused id's legacy entries sit below the new `AddIdx`, keeping the id-reuse invariant local by construction). `NewDB` range-checks loaded ids and rejects a feed with an empty `URL` (a pre-merge `feeds[]`-only db.gz) with a delete-and-re-fetch message. `RemoveFeed` uses `delete`.
- `seq` names the latest packs (`idx/L<seq>.gz`, `data/L<seq>.gz`, `meta/L<seq>.gz`): each article-producing batch writes generation `seq+1` (idx + data series; SyncMeta adds meta), bumps `Seq` only after idx+data saves succeed, and `Commit` publishes it — names are write-once, so a generation can never be seen half-written or rewritten. `GCLatest(ctx, keep)` (called from `cmd_fetch` after `Commit`, `latestKeep` = 2, warn-only) deletes expired generations for all three series, sweeping a 4-wide trailing window so crash-leaked packs self-heal.
- `hdrs` names the idx header summary (`idx/h<hdrs>.gz`): `cmd_fetch` calls `SyncIdxSummary` between `PutArticles` and `Commit` (warn-only — a failed ~1KB summary write must not discard a durable batch; readers fall back to eager idx loading and the next run retries), then `GCSummaries` next to `GCLatest`. `srr gen --bump` resets `HdrPacks` to 0 (an in-place rebuild invalidates the copied headers).
- `mp`/`mt` name the meta shards and latest tail (`MetaPacks`/`MetaTail`): `cmd_fetch` calls `SyncMeta` between `SyncIdxSummary` and `Commit` (warn-only); `GCMetaSummaries` runs next to `GCSummaries`; `GCLatest` also sweeps `meta/L<g>` generations. `srr gen --bump` resets both `MetaPacks` and `MetaTail` to 0.
- `withDB(locked, fn)` / `withDBCtx(ctx, locked, fn)`: most cmd `Run()` methods are now a single `withDB` call wrapping the work.

### Module System (`mod/`)

Pipeline per-feed during fetch. Factory pattern: `New()` returns a fresh stateful processor; each built-in registers a `func() Processor` factory that runs once per `New()` so it can capture per-instance state (compiled policies, a shared HTTP client). A built-in processor is `mod.Processor` = `func(context.Context, Params, *RawItem) error` (the ctx is the per-fetch context, threaded through `Module.Process`); most built-ins ignore the ctx, but network-bound ones like `#readability` use it for cancellation/deadline.

**File layout** — `main.go` holds the engine (registry + `Module`/`Process` dispatch), the `RawItem` wire types, and the external/shell-module exec. Support helpers carry a `helper_` filename prefix (`helper_params.go`, `helper_assets.go`, `helper_ssrf.go` — the latter exports `SafeTransport`, the SSRF-guarded `*http.Transport` used by `#readability`) to set them apart from the built-in module files, which are named for the module itself (`sanitize.go`, `minify.go`, `readability.go`, each with a matching `_test.go`). `secrets.go` holds `SetSecrets`/`SubprocessEnv` — the process-env⊕`srr.yaml`-secrets merge (secrets win, deduped, sorted) used as the `env` for both `mod.New()` and `ingest.New()`'s external commands.

**Module parameters** — a pipeline token may carry `key=value` options after the name, e.g. `#readability timeout=30s maxbody=16MiB`. `Module.Process` splits the token on whitespace; when the first field is a registered built-in it parses the rest into `mod.Params` (see `helper_params.go`) and dispatches internally, otherwise the original string runs as a shell command (so shell commands with spaces/`=` are untouched). The param split is **quote-aware** (`splitParamFields`): a double-quoted span keeps its spaces and drops the quotes, so a value with spaces stays one param (`ua="Mozilla/5.0 (X11)"`); an unterminated quote is a hard error, and there is no escape for a literal `"`. Built-ins read options via typed accessors with a default for the absent case — `Params.Duration(key, def)`, `Params.Bytes(key, def)` (sizes accept `B`/`KB`/`MB`/`GB` decimal or `KiB`/`MiB`/`GiB` binary; bare = bytes), `Params.String(key, def)` (verbatim value; explicitly empty = hard error) — and call `Params.only(allowed...)` to reject unknown keys. A malformed value or unknown key is a hard error (loud config feedback), distinct from a module's per-article fail-open behaviour; modules that take no options reject any param. Parameters are parsed per-item but the cost is negligible. External (shell) modules don't get parsed params — they receive their args on the command line.

- `#sanitize` — bluemonday, content-only. Allows `<audio>` (`src`/`controls`/`preload`, mirroring `<video>`) so `#selfhost` can self-host audio.
- `#minify` — tdewolff/minify, content-only.
- `#readability` — for feeds that syndicate only a teaser/summary: HTTP-GETs the item's `Link` (honest keyword-free UA — `+full-text-extractor` was dropped 2026-07-03, keyword-scanning WAFs 406 on "extractor"; default 20s timeout, 8 MiB body cap) and replaces `Content` with the article's main body extracted via `go-shiori/go-readability`. Tunable per pipeline position via `timeout=` (Go duration), `maxbody=` (byte size), `ua=` (request User-Agent — quote it, real UAs have spaces; some WAFs 406 on UA keywords, e.g. blogdechollos.com blocks any UA containing "extractor") and `selector=` (CSS selector group via `cascadia`; when set, the matches' HTML — every match in document order, inner HTML per match, childless matches like `<img>` render the element itself, matches nested in an earlier match are folded in — replaces the readability heuristic entirely: the per-site escape hatch for pages whose short genuine body loses the candidate scoring to text-dense sidebar/widget chrome (deal blogs), and via comma groups also the way to stitch disjoint blocks (hero image + body) into one article; a selector miss keeps the original content and WARNs, it never falls back to the heuristic), e.g. `#readability ua="Mozilla/5.0 (compatible; SRR/1.0)" selector=div.entry-content`; the timeout is enforced via a per-call `context.WithTimeout` so the shared client can carry different budgets. **Fail-open**: empty/invalid link, non-2xx, fetch/parse error, or empty extraction all leave `Content` untouched and return nil (logged at WARN) — one bad article never fails the fetch (but a bad `timeout=`/`maxbody=` value, or an unknown param, is a hard error). Place it **before** `#sanitize`/`#minify` (e.g. recipe pipe `["#readability", "#default"]`) so the fetched HTML gets clamped to the allowed element set. Network-bound, so unlike the other built-ins it honours the fetch `context` (cancellation/deadline).
- `#filter` — content-based item dropping (`filter.go`). Params (all optional; item dropped if ANY condition fires): `drop_title=/regex/[i]`, `keep_title=/regex/[i]`, `drop_content=/regex/[i]`, `keep_content=/regex/[i]`, `min_words=N`. On a match sets `i.Drop=true; return nil`. Unknown params / bad regex / bad int are hard errors. Does NOT mutate GUID/Published/Title/Content/Link.
- `#dedupmedia` — duplicate-media removal (`dedupmedia.go`): an article never shows the same picture (or video/audio) twice — the best copy survives. Groups `<img>`/`<video>`/`<audio>` by normalized file identity (`mediaFileID`: scheme + query/fragment ignored, WordPress `-WxH` size suffix stripped, `iN.wp.com` Photon proxy unwrapped; identity is per element type — an `<img>` and a `<video>` off one URL never merge) and removes all but the richest copy per group (`dedupScore`: canonical un-suffixed/un-proxied URL > alt text > declared pixel area > attribute count; ties keep document order), pruning wrapper chains a removal leaves without text or keep-worthy elements (`<a>`/`<p>`/`<figure>` → gone; `dedupKeepTags` keep them alive). Inline-glyph images are exempt (`isMediaGlyph`) — repetition is meaningful there: emoji classes (`emoji`/`wp-smiley`/`smiley`), `height: 1em`-style, `s.w.org/images/core/emoji` URLs, ≤2-rune emoji alt, both sides ≤ 96px declared, blank/spacer/pixel/placeholder/loading file names. No params (`p.only()`). Pure CPU, never fails an item: unparseable content and no-op passes return verbatim (no re-render). **Place BEFORE `#sanitize`** (`["#dedupmedia", "#default"]`) so the glyph heuristics still see the class/style attributes the sanitizer strips. Motivating case: WordPress "featured image in feed" plugins prepending a copy of an image the body repeats (finofilipino.org — every post shipped its image twice).
- `#unlazy` — lazy-load recovery (`unlazy.go`): promotes the real URL out of `data-src`-style attributes (`lazySrcAttrs`, URL-shaped values only — a boolean `data-lazyload="true"` never becomes a src) into `src` on `<img>`/`<video>`/`<audio>`; an `<img>` whose src is still missing or a placeholder (`lazyPlaceholderRe`: data URIs, about:blank, 1x1/spacer/loading names) takes the best `srcset`/`data-srcset` candidate (largest width descriptor, else density, else first) — a genuine src is never overridden by srcset; `<noscript>` fallbacks contributing an image the document doesn't already show (by `mediaFileID` identity) are unwrapped in place, redundant ones dropped (the parser treats noscript children as raw text, so the payload is re-parsed). No params (`p.only()`). Pure CPU, fail-open, verbatim no-op. **Place BEFORE `#sanitize`** — after it the `data-*` attributes holding the real URLs are gone. Essential in pipes using `#readability` (live article pages are where lazy-load markup comes from; without this they publish with blank/1-px images).
- `#embed` — iframe-to-link conversion (`embed.go`): `#sanitize` deletes `<iframe>` wholesale, so a post built around an embed publishes with its point silently missing. Rewrites known providers (`classifyEmbed`: YouTube incl. shorts + nocookie + protocol-relative — `/embed/videoseries` playlists excluded; Vimeo `player.vimeo.com/video/<digits>`; Dailymotion classic + `geo.dailymotion.com/player.html?video=`; Spotify `open.spotify.com/embed/*`) into a `<p>` holding the derivable thumbnail (YouTube `i.ytimg.com/vi/<id>/hqdefault.jpg`, Dailymotion `/thumbnail/video/<id>`) wrapped in a link, plus always a "▶"-prefixed text link (labelled by the iframe `title` attr, else "Watch on <Provider>") — the thumbnail may 404 and the frontend collapses broken images, so the target stays reachable. Unknown iframes (ads included) are deliberately left for `#sanitize` to strip. No params. Pure CPU, fail-open, verbatim no-op. **Place BEFORE `#sanitize`**; composes with `#selfhost` (self-hosts the injected thumbnail).
- `#enclosure` — out-of-body media surfacing (`enclosure.go`): prepends media that rides in item metadata the body never mentions — RSS `<enclosure>`, Atom `<link rel="enclosure">`, Media RSS `media:content`/`media:thumbnail` (directly or inside `media:group`, YouTube's layout), `<itunes:image>` — as leading `<p>`-wrapped `<img>` / `<video controls>` / `<audio controls>` blocks. Reads the parsed entry from `RawItem.Raw` (type-asserts `RawFeedItem`; **no-ops for external-ingest items** and on url-less `content` entries, which are text bodies not Media RSS). Picks the best image (full images beat thumbnails, then largest declared area), first video, first audio; kind from MIME prefix, then `medium` attr, then URL extension (`enclosureExtKinds`) — YouTube's `x-shockwave-flash` content entry falls out while its thumbnail survives. Skips candidates already visible in the body (exact URL, HTML-escaped URL, or `mediaFileID` identity — a WordPress size variant counts as present); non-http(s) URLs ignored. No params. Pure CPU, fail-open, verbatim no-op. **Place BEFORE `#sanitize` and `#dedupmedia`** (`["#enclosure", "#dedupmedia", "#default"]`).
- `#untrack` — tracking-freight removal (`untrack.go`): three cleanups. (1) Beacon images: an `<img>` matching `trackerPixelRe` (feedburner `~4`, Mailchimp `list-manage.com/track/open`, `pixel.wp.com`, WordPress.com stats, GA `collect`/`__utm.gif`, Facebook `tr`, scorecardresearch, quantserve, doubleclick, Yandex) or declaring both sides ≤ 2px is removed + `pruneEmptyWrappers` — the size rule skips any `<img>` still carrying a lazy `data-src` URL (a not-yet-promoted placeholder can declare 1x1). (2) Tracking query params (`utm_*` prefix + the `trackingParams` set: fbclid/gclid/mc_eid/…) stripped from `a href`, media `src`, `video poster` (`untrackAttrs`) — remaining params keep order and encoding verbatim, non-http(s) URLs never touched, emptied query drops its `?` (fragment kept). (3) The trailing WordPress "The post … appeared first on …" footer (`wpTrailerRe`, matched whitespace-collapsed against the **last** element only — the phrase mid-article survives). No params. Pure CPU, fail-open, verbatim no-op. **Place BEFORE `#sanitize`, AFTER `#unlazy`** (`["#unlazy", "#untrack", "#default"]`).
- `#selfhost` — downloads remote `<img>`/`<video>`/`<audio>` media (src + `<video poster>`, not `<a href>`) into the run's shared cache dir and rewrites each to a `#`-marker, so the end-of-pipeline upload step peeks/converts (`SRR_ASSET_PROCESS`)/uploads it to `assets/`. Network-bound (SSRF-guarded `SafeTransport`, like `#readability`); reads the run cache dir from the fetch context (`mod.WithCacheDir`, set in `feed.go:Feed.Fetch`) and **no-ops when absent** (e.g. `srr preview`). Params: `timeout=` (default 120s), `maxbody=` (default 128MiB, **clamped to `--max-asset-size`** via the `mod.MaxAssetSize` package var so the self-host object cap is enforced here at download — an over-cap body never lands in the cache); bad/unknown param is a hard error. **Fail-open per asset**: a bad URL / non-2xx / oversize / SSRF-blocked dial / write error leaves the remote URL in place (WARN). Download dedup is URL→cache-file; upload dedup is `UploadCacheRef`'s content hash. **Place AFTER `#default`** (`["#default", "#selfhost"]`): it then downloads only sanitizer-approved media and its markers (incl. `<video poster>`) never round-trip through `#sanitize`. Limitations: with `webify` as `SRR_ASSET_PROCESS`, audio is self-hosted but not transcoded (webify is image/video only); `<source>` children stay stripped; cross-run URL cache may serve stale bytes (clear the cache dir).
- External modules: `/bin/sh -c`, stdin/stdout JSON (`RawItem`), stderr passthrough. Emitting `{"drop":true}` on stdout sets `i.Drop=true` (via the existing `json.Unmarshal(out, i)` path — no peek struct needed); subsequent pipeline steps are skipped and the item is not stored. Dropped items retain their GUID in the feed boundary. The author-facing protocol spec + a reference implementation live in `backend/README.md` (External mod protocol); the protocol tests are in `mod/main_test.go`.

#### Asset self-hosting (`assetFetcher.UploadCacheRef` + `RewriteAttrs`)

SRR self-hosts files an ingest strategy or the built-in `#selfhost` mod leaves in the run's shared cache dir: it stores them under a content-hash key (`assets/<2-hex>/<16-hex><ext>`) and rewrites the in-content reference to that key. This is **not** a pipeline module — it is the automatic end-of-pipeline step `fetchRun.uploadAssets`, run from `Feed.fetchURL`.

- Implementation `assetFetcher` (`assets.go`): `UploadCacheRef(ctx, cacheDir, localname)` returns `(key, uploadedBytes, err)` — uploadedBytes is the payload size of the Put THIS call led (0 on memo/store dedup hits and for singleflight followers, so summing it counts each stored object exactly once; it feeds `Feed.AssetBytes`). It resolves `localname` inside `cacheDir` (guards: containment via `EvalSymlinks`+`Rel`, reject non-regular/symlink), keys by `contentHashKey` (sha256 of the **original source file bytes**, with the source extension) and `Put`s only if the key isn't already in the store (`Backend.Stat` existence probe — zero-body; `size == 0` falls back to the body-carrying `Get` check, which stays authoritative for the cases a HEAD can't prove: a zero-byte stored asset, an HTTP store omitting Content-Length). The `SRR_MAX_ASSET_SIZE` cap is **enforced at download** (`#selfhost` / external ingest via `Request.MaxAssetSize`), so `UploadCacheRef` trusts the cache file and does **not** re-check the source size. It does no outbound HTTP. Built once per `fetch` run in `cmd_fetch.go` from `db.Backend`, shared across workers (concurrent-safe). The external-ingest upload step (`fetchRun.uploadAssets` → `rewriteItemAssets`, run from `Feed.fetchURL`; see Ingest §) calls it once per `#`-marker against the run's shared cache dir (`fetchRun.cacheDir` = `assetCacheRoot()`).
- **Per-asset pre-upload processing (`SRR_ASSET_PROCESS` / `globals.AssetProcess`):** an optional operator-configured command (e.g. `webify -m 720`) run on **every** asset just before upload to process its bytes — image/video transcoding being the motivating case (covers external-ingest and any internal-mod marker uniformly, since both funnel through this one step). It runs **only on a store miss**: the existence check keys on the **source** hash *before* processing, so an asset already uploaded is never re-processed (the common case for an image reused across articles/feeds). `runProcess` substitutes the cache file path for each `{input}` token in the command (per arg, e.g. `conv -i {input} --flags`), or appends it as the final arg when absent. With a `{output}` token the command runs in **file mode**: SRR substitutes a fresh staging path under `<cache-dir>/_processed/` (`assetFetcher.procDir`, set by cmd_fetch; OS temp dir fallback when unset — big transcodes must not land in a tmpfs `/tmp`, and a crash-leaked output inside the cache tree ages out via the post-cycle sweep; success removes it immediately), the command writes its processed bytes there, and prints a `{mimetype,extension,encoding}` JSON to stdout — the mimetype/encoding ride the upload as the stored object's `store.ObjectMeta` (Content-Type/-Encoding), the extension is for the peek-decided key (see asset-peek). Without `{output}`, processed bytes are read from stdout (no declared metadata, so the object gets the `application/octet-stream` default). stderr is **captured, not passed through** (asset commands narrate progress on stderr — webify's tty progress bar — and parallel jobs would garble srr's own output/status line): `mod.RunCommandTimeout` buffers the most recent 4 KiB (`tailBuffer`) and on failure folds the trailing lines (CRs count as line breaks) into the returned error, so the warn line carries the diagnostic; on success it is discarded. This is deliberately unlike `RunSubprocess` — external ingest/mod commands keep passthrough stderr as their documented log channel. **Fail-soft** — a non-zero exit, empty output, or (file mode) invalid metadata JSON uploads the original unchanged, so a processing hiccup or an unhandled file type never wedges a feed. The command is bounded by its **own** timeout `--asset-process-timeout` / `SRR_ASSET_PROCESS_TIMEOUT` (default **0 = unlimited**: no deadline, bounded only by run cancellation on SIGINT/SIGTERM, since media transcoding can run arbitrarily long), separate from the shared `--cmd-timeout` — which governs ingest/mod commands only and **never** affects asset work: both `runProcess` and asset-peek's `runPeek` run through `mod.RunCommandTimeout(ctx, procTimeout, …)`, where a `procTimeout <= 0` adds no deadline. The dedup key always hashes the **source** bytes (the existence check must run pre-process), but its **extension** comes from `asset-peek` when configured (else the source extension); the asset-process **output** keeps a fail-soft size guard in **both** modes (`readProcOutput` for `{output}` mode, an inline `len(out)` check for stdout mode): an over-`SRR_MAX_ASSET_SIZE` output uploads the original instead — that output is generated here and isn't download-bounded, unlike the source. Empty `SRR_ASSET_PROCESS` disables it (pure pass-through, the previous behaviour). Asset jobs run concurrently across a feed's items; the peek/transcode/upload work is bounded globally by `SRR_ASSET_WORKERS` (the `assetFetcher`'s `a.sem` pool, held by the leader job only). A marker referenced by several items (same feed or across feeds — the run's `assetFetcher` is shared) is coalesced by `assetFetcher.flight` (a `singleflight.Group` keyed on the source content hash) via `DoChan`, so one concurrent wave peeks/transcodes/uploads the asset exactly once and the rest receive the leader's key without taking a slot — a waiter whose own feed cancels bails on its caller ctx instead of blocking on the leader. The body runs under the run/shutdown ctx (`a.baseCtx`), not the leader feed's, so the leader's result is shared with the wave (a shared asset that genuinely fails to upload fails every referencing feed, since none can publish content pointing at a missing asset; each self-heals next fetch) WITHOUT one feed's cancellation poisoning a healthy follower feed. The `seen` memo still short-circuits the cheaper sequential/cross-call hit (and a wave's late arrivals) before `flight`, and the store-existence check stays the cross-run backstop. The matching out-of-repo media converter is `webify` (`~/ws/webify`, on PATH as `~/.local/bin/webify`): auto-detects image→WebP / video→WebM, `webify <input>` writes to stdout.
- **Per-asset identification (`SRR_ASSET_PEEK` / `globals.AssetPeek`, the `assetFetcher.peek` field set in `cmd_fetch.go`):** an optional probe run on **every** asset **before** the dedup check (so it runs even for already-uploaded assets — the accepted cost). `runPeek` substitutes/append the cache path like `runProcess` and parses a `{mimetype,extension,supported,encoding}` JSON from stdout. The **extension** becomes the stored object's key extension (`contentHashKey(peek.extension, sha256(source))` — so a transcoded `.gif`→WebP rides a `.webp` key while dedup still keys on the source bytes); the **mimetype/encoding** become the object's `store.ObjectMeta`; **supported=false** hosts the original bytes and **skips** `asset-process` (peek reports the real source type for an unsupported asset). Precedence: in `{output}` mode `asset-process`'s declared mimetype/encoding (the actual result) override peek's; a `process.extension` disagreeing with the peek key only **warns** (the key is already chosen). **Fail-soft**: a peek error / invalid JSON falls back to the source extension and `supported=true`. Empty disables (the object then gets the source extension + `application/octet-stream` default). **Corrupt-media guard**: a peek that runs fine but identifies NOTHING (`supported=false`, empty/`octet-stream` mimetype, empty extension) on a source whose extension claims media (`isMediaExt`) classifies the file as broken bytes — `UploadCacheRef` returns the `errCorruptAsset` sentinel, `rewriteItemAssets` declines the marker with a WARN (the article publishes without working media; the reader collapses the dead element) instead of hosting garbage (the chron-437 bug) or failing the feed (the bytes won't improve on retry — a hard error would wedge it). Identified-but-unsupported formats and non-media extensions keep the host-as-is contract. **Cycle-level degradation warning**: per-run `assetFetcher.procFailed`/`corrupt` counters (bumped on each peek/process command failure and each corrupt decline) are surfaced by `runFetch` as one WARN per cycle — a systemic cause (webify missing from the service PATH) otherwise drowns in per-asset noise while every asset silently ships unprocessed with `octet-stream`.
- `RewriteAttrs(content, fn)` (`mod/helper_assets.go`): the generic HTML attribute walk over `assetAttrs` (img/video/audio `src`, video `poster`, **plus `a href`**, the generic-file set) behind the upload step. The caller's `fn` returns `(newVal, ok, err)` and owns the `#`-marker upload policy — which values are markers, how files are stored, and which failures are fatal (a non-nil `err` aborts the walk and fails the feed) — keeping that policy out of `mod`.
- Idempotency: download dedup is the ingest strategy's (its cache check); upload dedup is SRR's — `UploadCacheRef` skips the `Put` (and the `SRR_ASSET_PROCESS` run) when the source-hash key is already in the store (`Backend.Stat` probe, `Get` fallback on size 0), so a re-run re-uploads and re-processes nothing.
- `assets/` is a reserved store prefix in the writer↔reader contract (analogous to `idx/`/`data/`); the frontend (`fmt.ts`) resolves it against the pack base. The store's `AtomicPut` carries a `store.ObjectMeta{ContentType, ContentEncoding}`: `S3` stamps it (with meta unset, Content-Type falls to `contentTypeForKey` — SRR's own gzip objects, db.gz + pack-grammar names, declare `application/gzip`; anything else gets `application/octet-stream` — an asset's type is never derived by extension or byte-sniffing, peek/process is its single source of truth); local/SFTP ignore it and rely on the static server's mime-by-extension at request time. So a peek-corrected key **extension** is what fixes local/SFTP rendering, while the explicit mimetype hardens S3.
- `GUID` and `Published` are immutable for all modules (built-in or external; change = error). Enforced in `processItem` after each pipeline step — the captured value before the step is compared to the post-step value, attributing changes to the offending module.
- `RawItem.Raw` is set by the ingest parser (`ingest/feed.go`) to the parsed entry as `mod.RawFeedItem` (`map[string][]mod.RawField`); modules can type-assert it for typed access. External (shell) modules see the same data as JSON via the short keys `@`/`$`/`+` (text/attrs/children).
- `RawItem.Drop bool` (`json:"drop,omitempty"`) — set to `true` by a pipeline step (built-in or external) to signal that the item should be discarded. `processItem` short-circuits on the first `Drop=true`: returns nil, skips remaining steps and post-loop normalization. Callers (`feed.go` `fetchURL`, `cmd_preview.go`) check `i.Drop` after `processItem` returns nil and `continue` to skip storing/rendering the item. The GUID is already in the boundary map by the time the second-pass loop runs, so a dropped item stays deduped on the next fetch without any extra handling.

### Recipes

Processing config lives in named `{ingest, pipe}` recipes (`DBCore.Recipes map[string]Recipe`), referenced by feeds via `Feed.Recipe`. The reserved `default` recipe (always present, seeded `["#sanitize","#minify"]` by `NewDB`) is the fallback.

- A feed with empty/absent `Recipe` resolves to `default` via `recipeFor`.
- Each axis falls back to `default` independently: a recipe with only `Ingest` set uses its own ingest and `default`'s pipe; only `Pipe` ⇒ its pipe and `default`'s ingest.
- **Feed-level overrides**: `Feed.Ingest` (`ingest`) and `Feed.Pipe` (`pipe`) sit on top of the recipe, each axis independently — set, they win over the recipe's value; empty, they inherit it. Resolved at the start of `Feed.Fetch`: `resolvePipe(resolvePipe(default.Pipe, recipe.Pipe), feed.Pipe)` and `ingest.Select(feed.Ingest, recipe.Ingest, default.Ingest)` (variadic first-non-empty).
- `#default` (`pipeDefault`) expands inline to the next pipe down the fallback chain: in a recipe's pipe, the `default` recipe's pipe; in a feed's pipe, the feed's *effective recipe pipe* (already `#default`-expanded). The `default` recipe itself forbids `#default` (enforced by `validatePipe(allowDefault = name != defaultRecipeName)` in `RecipeSetCmd`); a feed pipe always allows it (`normalizeFeed` runs `filterPipe` + `validatePipe(…, true)`).
- CLI: `srr recipe set <name> [-i ingest] [-p step]…` (add/replace); `srr recipe ls/show/rm`; `srr feed add/upd/import -r <name>`; feed-level overrides via `srr feed add/upd [-i ingest] [-p step]…` (on `upd`, `-i ""` clears the ingest override and a lone `-p ""` clears the pipe override). Every `-p` flag is **repeatable, one step per flag** (`sep:"none"`).
- Clean break, amended: a pre-recipes db.gz's legacy **root** `pipe`/`ingest` still drop on load (DBCore has no such json tags) and `NewDB` re-seeds `default` if absent — but the legacy **feed-level** keys are the same keys `Feed.Ingest`/`Feed.Pipe` use today, so those values revive as feed-level overrides (deliberate: the old per-feed meaning matches the new one; pinned by `TestNewDBLegacyPipeIngest`).

## Conventions

- **Imports**: stdlib → external → internal, blank-line separated, alphabetical within groups
- **Errors**: Always wrap with `fmt.Errorf("context: %w", err)`. No custom error types; the only package-level sentinels are `errNotFeed` (`ingest/feed.go`), `errNotAsset` and `errCorruptAsset` (`assets.go`), and `errExpireDone` (`db_expire.go` — the expiration walk's early-stop channel, since `walkArticles`' callback contract has no other) — don't add more without need.
